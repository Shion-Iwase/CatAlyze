ソースドキュメント品質・選定（Source Document Quality & Selection）

RAGシステムに投入するドキュメントを選定する際の基準。ドキュメントの信頼性・権威性・ドメイン適合性が応答の品質と正確性を左右する。

ドメイン推奨ソース（一次ソース）避けるべきソース
医療査読済み医学研究論文・政府の健康ガイドライン患者ブログ・SNS・教科書の要約
金融規制当局への提出書類・公式金融レポートニュース記事・ブログ・フォーラム
法律政府レポート・公式法的文書フォーラムの書き込み・ユーザーブログ
技術サポート技術サポート要件に基づくコンテンツ（関連性・品質を優先）関連性のない長いドキュメント
試験頻出ポイント
  • ドメイン固有アプリケーション → 権威ある一次ソースを使うことが最重要
  • 「長いドキュメントを優先」は不正解 → 長さは品質を保証しない
  • 「コンテンツ関連性よりソースの多様性を重視」は不正解 → 関連性が多様性に勝る
  • 「関連性に関わらず公開されているすべてのテキストを含める」は不正解 → 無関係テキストが精度を希薄化する

テキスト抽出ツール（Text Extraction Tools）

様々なファイル形式（PDF・HTML・画像・スキャン文書）からテキストを抽出するための Python ライブラリ群。入力形式によって適切なツールが異なり、間違ったツールを選ぶと抽出が失敗する。

ファイル形式正解ツール理由
スキャン画像（JPG・PNG）pytesseract / TextractOCR ベース。画像からテキストを認識・抽出
テキスト・画像混在 PDFunstructured最小限のコードで両方に対応するために特別設計
PDF の構造化テキスト・表pdfplumber表データの抽出に優れる（スキャン画像には不可）
テキストベース PDF のみPyPDF2スキャン画像・表・HTML には非対応
HTML（ネストされたテーブル含む）BeautifulSoup + LXMLHTML/XML のパースとテキスト抽出に特化
スキャンされたレシート・請求書（数値データ）pytesseract + TextractOCR で数値データを抽出
  • Image Preprocessing（画像前処理）：透かし・ロゴ等の視覚的ノイズを OCR 前に除去してテキスト抽出品質を向上させる
  • Delta Lake への保存パイプライン順序：スキーマ定義 → OCR（pytesseract 等）でテキスト抽出 → Spark で Delta Lake に書き込む
試験頻出ポイント
  • pytesseract はスキャン画像専用（PDF の中身がスキャン画像の場合に使用）
  • pdfplumber はスキャン画像には使えない → 通常 PDF（テキスト埋め込み）専用
  • BeautifulSoup は PDF には使えない → HTML/XML 専用
  • 透かし・ロゴを含むスキャン文書 → Image Preprocessing でノイズ除去してから OCR

データクリーニング・フィルタリング（Data Cleaning & Filtering）

RAGシステムや LLM のトレーニング前に、不要・無関係・有害なコンテンツをデータセットから除去するプロセス。クリーニングの質が retrieval 精度と応答品質に直結する。

  • 無関係セクションの削除：付録・目次・冗長な脚注・マーケティング説明・免責事項など。RAGタスクに関連するコンテンツのみを保持
  • ノイズ除去：広告・ハイパーリンク・埋め込みメディア・著者名・タイムスタンプ等の削除
  • 誤情報フラグ付きドキュメントの除外：前処理段階でフラグの立てられたドキュメントをパイプラインから排除（免責事項追加やモデル無視への依存は誤答）
  • 不適切言語フィルタリング：取り込み前にフィルタリングツールで検出・除去（取り込み後のモニタリングへの依存は誤答）
  • 選択的メタデータ保持（法的ドキュメント）：ケース名・引用など有用なメタデータは保持しながら不要なもの（タイムスタンプ等）のみ削除
  • タイムスタンプベースフィルタリング：古いドキュメントをタイムスタンプで除外して最新コンテンツを優先
試験頻出ポイント
  • 「完全性のためにすべてを残す」は常に誤答 → 不要コンテンツはノイズとなり retrieval を低下させる
  • 誤情報への対処 → 前処理段階での除外が正解（取り込み後のモニタリングや免責事項追加では不十分）
  • 不適切言語への対処 → 取り込み前のフィルタリングが正解（LLM に無視させることは誤答）
  • メタデータ → 「全削除」も「全保持」も誤り。「有用なものは残す、不要なものだけ削除」の選択的アプローチが正解

バイアス対処（Bias Handling）

採用推薦や医療応用など公平性が重要なシステムにおいて、トレーニングデータや RAG データセットに潜在するバイアスを特定・除去・軽減するプロセス。

  • バイアス分析と削除・再重み付け：データセットのバイアスを特定し、バイアスのあるエントリを削除するか再重み付けして公平性を確保する
  • 事前学習済みモデルのバイアス：事前学習済みモデルも元のデータセットのバイアスを反映する可能性がある（モデルを変更しないだけではバイアスは解決しない）
試験頻出ポイント
  • バイアス対処の正解：「潜在的なバイアスを分析し、バイアスのあるエントリを削除または再重み付けする」
  • 「事前学習済みモデルのみ使用してデータセット変更を避ける」は不正解 → モデルも元データのバイアスを反映する
  • 「データセットサイズを削減する」は不正解 → バイアスの除去は保証されない
  • 「バイアスを無視する（ユーザークエリで出力が決まるため）」は不正解 → バイアスを持続させるリスクがある

言語フィルタリング（Language Filtering）

特定の言語のみを対象とするシステムで、対象外言語のドキュメントを前処理段階でフィルタリングする手法。言語の一貫性を確保することで retrieval の精度を保つ。

  • 言語検出ツール：非英語ドキュメントを自動検出してフィルタリングするツール
  • 翻訳 vs フィルタリング：翻訳は追加のオーバーヘッドと潜在的な不正確さをもたらすため、フィルタリングが優先される
試験頻出ポイント
  • 英語専用 RAG システムの多言語ドキュメント処理 → 「翻訳」より「言語検出ツールで非英語ドキュメントをフィルタリング」が正解
  • 「すべてのコンテンツを英語に翻訳する」は不正解 → 追加コストと不正確さのリスクがある
  • 「潜在的な多言語機能のためにすべてのドキュメントを保持する」は不正解 → 英語のみシステムでは品質が下がる

チャンキング戦略（Chunking Strategy）

ドキュメントをRAGシステムでのベクトル検索・取得に適したサイズに分割する手法。トークン制限の遵守と意味的文脈の保持を両立させることが目的。チャンクのサイズや分割方法によって retrieval 品質と LLM の応答完全性が大きく変わる。

  • 論理的チャンキング：見出し・サブ見出し・段落・チャプターなどドキュメント構造に沿って分割。コンテキストが保持され retrieval 品質が高い。最頻出の正解パターン
  • 固定サイズチャンキング：500トークン・1000トークンなど均等サイズで分割。意味のあるセクションを分断するリスクがあり通常は誤答
  • オーバーラップ：チャンク間でテキストを重複させる手法。過度なオーバーラップ（30%・50%・200トークンなど）は冗長性を生むだけで効果が薄く、試験では誤答扱いが多い
  • チャンクサイズを小さくする：LLMのトークン制限超過による不完全応答への対処として有効
  • 1チャンク1行フラット化：チャンク配列を1チャンク1行にフラット化し一意IDを付与して Delta テーブルに保存。Vector Search インデックス用の最適形式
チャンキング戦略評価使用場面
論理的チャンキング（見出し・段落単位）✅ 最優先法的テキスト・政府レポート・学術論文など構造のあるドキュメント全般
チャンクサイズを小さくする✅ 有効トークン制限超過で応答が不完全になる場合
固定サイズチャンキング⚠️ 通常は不正解意味のあるセクションを分断するリスクあり
過度なオーバーラップ❌ 誤答扱い多い冗長性が増すだけで論理的分割より劣る
ドキュメント全体を1チャンク❌ 不正解トークン制限超過のリスク
試験頻出ポイント
  • 法的テキスト・小説・政府レポートなど構造のあるドキュメント → 論理的チャンキング（見出し・段落・チャプター単位）
  • Delta Lake への保存順序：スキーマ定義 → チャンキング → 書き込み（スキーマを後から定義するのは誤り）
  • embedding モデルの context length（例：512）とチャンクサイズを一致させるとコスト・latency が最小化される
  • 「チャンクサイズを大きくしてコンテキスト断片化を減らす」は不正解 → トークン制限を超えて不完全応答が悪化する

トークン制限と Context Length（Token Limit & Context Length）

LLM がプロセスできる最大トークン数（Context Length）を超えると、入力が切り捨てられ（Truncation）重要な情報が失われる。ドキュメントサマリーパイプラインの設計においてContext Lengthの評価が不可欠。

  • Truncation：Context Length を超えた部分が切り捨てられ重要情報が失われる現象
  • Context Length の評価理由：モデルが入力ドキュメントを truncation なく処理できるよう確保するため（要約の長さ制御やコスト削減のためではない）
  • チャンクサイズとContext Lengthの一致：embedding モデルの Context Length（例：512）とチャンクサイズを一致させることでコスト・latency を最小化できる
Context Length 超過への対処法評価
チャンクサイズを小さくする✅ 正解
取得レコード数を削減する✅ 正解
要約モデルで長さを短縮する❌ 誤答（重要な詳細が省略されるリスク）
より長い Context Length のモデルを新規トレーニングする❌ 誤答（リソースを大量消費し常に実現可能ではない）
ALiBi でモデルを再トレーニングする❌ 誤答（不要で複雑すぎる）
試験頻出ポイント
  • Context Length の評価の主目的：「truncation なしでドキュメント全体を処理できること」（コスト削減・トレーニング効率とは無関係）
  • 512 トークンのチャンクには context length 512 の最小 embedding モデルを使用 → コスト・latency 最小の最適設定

冗長データの除去（Redundancy Removal）

データセット内の冗長・重複したコンテンツを識別・除去してモデルトレーニングの効率と応答品質を向上させるプロセス。

  • 冗長な例の削除：技術マニュアルに含まれる繰り返しの例を削除しつつユニークで意味のあるコンテンツを保持する
  • プレースホルダー置換の問題：冗長コンテンツをプレースホルダーで置き換えてもモデルを混乱させる可能性がある（誤答）
  • マニュアル全体の除外の問題：冗長性を含むマニュアル全体を除外すると価値ある情報も失う（誤答）
試験頻出ポイント
  • 正解：「ユニークで意味のあるコンテンツを保持しながら冗長な例を削除する」
  • 「完全性のためにすべての例を保持する」は不正解 → 冗長な例は価値を追加せずデータセットを膨らませるだけ
  • 「冗長な例をプレースホルダーで置き換える」は不正解 → モデルを混乱させる可能性がある
  • 「マニュアル全体を除外する」は不正解 → 価値ある情報も失う

メタデータ管理（Metadata Management）

データセット内の補助情報（タイムスタンプ・著者名・ケース名・引用等）を適切に扱う技術。不要なメタデータはモデルのパフォーマンスを低下させるノイズとなり、有用なメタデータはコンテキストの理解に寄与する。

メタデータの種類処理方針例
無関係メタデータ（削除対象）削除するタイムスタンプ・著者名・透かし・ロゴ・ハイパーリンク・埋め込みメディア
有用なメタデータ（保持対象）保持する法的ドキュメントのケース名・引用など、ドメイン固有の重要情報
Vector Search用の識別子付与する各チャンクへの一意 ID（インデックスでの効率的な取得のため）
試験頻出ポイント
  • ファインチューニング用データセット → 不要なメタデータを前処理で全て削除してモデルが実際のコンテンツに集中できるようにする
  • 法的ドキュメント → 「全削除」ではなく「有用なものを保持しながら不要なものだけ削除」が正解
  • 「コンテキストのためにメタデータを保持する（ファインチューニング用）」は不正解 → 無関係メタデータがトピック識別のパフォーマンスを低下させる

ベクトルストア・Delta Lake へのデータ保存

チャンク化されたドキュメントを Vector Search インデックスや Delta Lake に格納するための最適な形式と順序。効率的なストレージが retrieval のパフォーマンスに直結する。

  • 1チャンク1行形式（Flattened DataFrame）：テキストチャンクの配列を1チャンク1行にフラット化し、各行に一意の識別子を作成して Delta テーブルに保存。Vector Search インデックスでの最適な取得が可能
  • スキーマ定義の優先：データを Delta Lake に書き込む前に必ずスキーマを定義する（後から定義するのは誤り）
  • Delta Lake vs JSON ファイル：Delta Lake の方が大規模検索でパフォーマンスが高い（各チャンクを個別 JSON ファイルとして保存するのは劣る）
  • Unity Catalog Volume：WAV・テキスト文字起こしなどのファイル保存場所（Delta テーブルとは異なる用途）

Delta Lake 保存の正しい順序：スキーマ定義 → テキストチャンキング → Delta Lake テーブルへの書き込み

試験頻出ポイント
  • Vector Search 用の保存：「1チャンク1行にフラット化 ＋ 一意 ID ＋ Delta テーブル」が正解パターン
  • 「スキーマを後で定義してもよい」は不正解 → 書き込み前にスキーマを定義しないと一貫性が欠如する
  • 「各チャンクを個別 JSON ファイルとして保存する」は不正解 → 大規模検索では Delta テーブルの方が高パフォーマンス

Retrieval 評価メトリクス（Retrieval Evaluation Metrics）

RAGシステムの検索コンポーネントのパフォーマンスを評価するための指標群。取得されたドキュメントの関連性・網羅性・ランキング品質を測定する。指標の用途を混同しないことが重要。

指標用途Retrieval 評価に適しているか
Precision（精度）取得ドキュメントのうち関連するものの割合✅ 適切（無関係な結果が頻出する問題の改善指標）
Recall（再現率）全関連ドキュメントのうち取得できたものの割合✅ 適切（関連ドキュメントを見落とす問題の改善指標）
F1 ScorePrecision と Recall の調和平均✅ 適切（両者のバランスを評価）
Cosine Similarityベクトル間の意味的アライメント評価✅ 部分的（生成された回答を完全には評価しない）
NDCG取得ドキュメントのランキング品質評価✅ 適切（法的ドキュメント RAG など）
BLEU機械翻訳評価❌ 不適切（翻訳タスク専用）
ROUGE要約品質の評価❌ 不適切（生成評価に使用）
Perplexity言語モデルの流暢さ・確信度❌ 不適切（retrieval の関連性・精度とは無関係）
Sentiment Analysisテキストのトーン評価❌ 無関係
試験頻出ポイント
  • 「無関係な結果が頻繁に表示される」→ Precision の改善が必要
  • 「関連ドキュメントを見落とす」→ Recall の改善が必要
  • BLEU は retrieval 評価には常に不適切（翻訳専用）
  • Cosine Similarity だけで最終生成回答を包括的に評価するのは不十分（retrieval 品質の一部しか評価できない）
  • RAG システムの最善の評価方法：検索コンポーネントと生成コンポーネントを分離してテストするデータセットを作成し MLflow の組み込み評価指標を使用

プロンプト-応答ペアの品質評価（Prompt-Response Pair Quality）

LLM アプリケーション（特にチャットボット）のトレーニング・評価データとして使用するプロンプトと応答のペアの品質を評価するための基準。

  • 応答の完全性（Completeness）：クエリのすべての側面に対処する応答かどうか。チャットボットの有用性とユーザー満足度を向上させる
  • プロンプトの関連性（Relevance）：顧客のクエリに対してプロンプトが関連性を持つかどうか
  • プロンプトの冗長性（Redundancy）：クエリの焦点を薄める可能性があるため評価基準としては重要でない（誤答扱い）
  • トークン使用効率：コスト削減には重要だが、整合性確保とは無関係（誤答扱い）
試験頻出ポイント
  • プロンプト-応答ペアの評価で最重要な2軸：「プロンプトの関連性」と「応答の完全性」
  • 「冗長性」「トークン効率」「ランダムサンプリング」は評価基準として重要でない（誤答）

Q1. 10,000語を超える法的契約書のチャンキングに最適な戦略は何ですか？固定サイズチャンキングが不正解な理由も答えてください。

▶ 答えを見る

A. 論理的チャンキング（条項・付属文書など論理的セクション単位で分割）が正解。固定サイズチャンキングが不正解な理由：一定トークン数で機械的に分割するため、意味のある条項の境界を無視してセクションを分断し、retrieval 品質が低下するから。

Q2. テキストと画像が混在する PDF からテキストを抽出するためのライブラリは何ですか？また、スキャンされた JPG 画像からテキストを抽出するためのツールは？

▶ 答えを見る

A. テキスト・画像混在 PDF → unstructured（最小限のコードで両方に対応するために特別設計）。スキャン JPG 画像 → pytesseract（OCR ベースで画像からテキストを認識・抽出）。

Q3. フィードバックデータを Delta Lake に保存する正しい順序を3ステップで答えてください。

▶ 答えを見る

A. ① スキーマ定義 → ② テキストチャンキング → ③ Delta Lake テーブルへの書き込み。スキーマは書き込み前に定義しなければ一貫性が欠如する。

Q4. RAGシステムで「無関係な結果が頻繁に表示される」問題と「関連ドキュメントを見落とす」問題をそれぞれ改善する評価指標は何ですか？

▶ 答えを見る

A. 「無関係な結果が頻繁に表示される」→ Precision（精度）の改善が必要（取得ドキュメントのうち関連するものの割合）。「関連ドキュメントを見落とす」→ Recall（再現率）の改善が必要（全関連ドキュメントのうち取得できたものの割合）。

Q5. 医療チャットボット向け RAG システムのソースドキュメントとして適切なものと不適切なものをそれぞれ2つ挙げてください。

▶ 答えを見る

A. 適切（一次ソース）：① 査読済み医学研究論文 ② 政府の健康ガイドライン。不適切：① 患者ブログ（主観的・未検証情報を含む）② SNSの書き込み（信頼性が低い）。ドメイン固有アプリケーションでは権威ある一次ソースを使うことが最重要。

Q6. ファインチューニング用データセットのメタデータはどう扱うべきですか？法的ドキュメント用データセットの場合はどう違いますか？

▶ 答えを見る

A. ファインチューニング用 → 不要なメタデータを前処理でできるだけ全て削除する（無関係なメタデータがトピック識別のパフォーマンスを低下させるため）。法的ドキュメント用 → 「全削除」でも「全保持」でもなく、ケース名・引用など有用なメタデータは保持しながら不要なもの（タイムスタンプ等）のみ削除する選択的アプローチが正解。

Q7. Context Length を超過したことで LLM の応答が不完全になっている場合、正しい対処法を2つ答えてください。

▶ 答えを見る

A. ① チャンクサイズを小さくする ② 取得レコード数を削減する。「要約モデルで長さを短縮する」は不正解（重要な詳細が省略されるリスク）。「より長い Context Length のモデルを新規トレーニングする」は不正解（リソースを大量消費し常に実現可能ではない）。

Q8. 採用推薦システムのトレーニングデータにバイアスが含まれている場合、正しい対処法は何ですか？

▶ 答えを見る

A. 潜在的なバイアスを分析し、バイアスのあるエントリを削除するか再重み付けして公平性を確保する。「事前学習済みモデルのみを使用してデータセット変更を避ける」は不正解（モデルも元データのバイアスを反映するため）。「データセットサイズを削減する」は不正解（バイアスの除去にはならない）。
