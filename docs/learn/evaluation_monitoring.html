Evaluation and Monitoring

ROUGE・BLEU・Precision/Recall・MLflow Evaluate・Inference Tables・本番モニタリング指標・コスト最適化を扱う。約26問・15トピック。

この分野の概要

Evaluation and Monitoring 分野は、GenAIアプリケーションの品質測定・運用監視・コスト管理を扱う。 「どの指標がどのコンポーネントの評価に使えるか」の明確な区別が問われる。 検索品質（Precision/Recall/NDCG）・生成品質（ROUGE/BLEU/Perplexity）・本番運用（Latency/Throughput/エラーレート）は それぞれ別の指標体系であり、混同しないことが合格の鍵。

出題内容詳細

RAGシステムの評価（検索と生成の分離評価）

RAGシステムは「検索（Retrieval）」と「生成（Generation）」の2つのコンポーネントで構成されており、 評価も両者を分離して行うことが推奨される。 検索コンポーネントにはPrecision/Recall/NDCGなどを用い、生成コンポーネントにはBLEU/ROUGE/Perplexityを使う。

  • 検索評価データセット: 既知の質問と正解ドキュメントペアを作成し、検索品質を定量評価する
  • 生成評価データセット: 既知の質問と参照回答ペアを作成し、生成品質を定量評価する
  • 分離評価の重要性: 検索の問題なのか生成の問題なのかを切り分けることで、改善の焦点が明確になる
  • MLflow組み込み評価指標: Databricksでの評価に標準的に使えるフレームワーク
  • 取得されたコンテキストの品質評価: RAGが無関係な情報を返す問題の診断では最初に行うステップ
試験頻出ポイント:
  • RAGの正式評価は「検索と生成を別々にテストするデータセットを作成し、MLflow組み込み指標を使う」が正解
  • RAGが無関係な情報を返す問題の診断では「取得されたコンテキストの品質評価」を最初に行う（LLMを変えたり検索アルゴリズムを変える前に）
  • MLflowでのRAGアプリ評価に最も関連する指標は「検索関連性指標（precision/recall）＋応答精度指標（BLEU/ROUGE）」の組み合わせ

テキスト生成品質指標：ROUGE・BLEU・Perplexity

テキストの品質を定量的に評価するための標準指標群。これらはすべて「生成品質（Generation Quality）」の指標であり、 「検索品質（Retrieval Quality）」の指標ではない点に注意。

  • ROUGE（Recall-Oriented Understudy for Gisting Evaluation）: 生成サマリーと参照サマリーのRecall・Precision・F1スコアのオーバーラップを測定。要約タスク向け
  • BLEU（Bilingual Evaluation Understudy）: 生成テキストと参照テキストのn-gramオーバーラップを比較。機械翻訳品質評価の標準指標
  • Perplexity: 言語モデルの流暢さ・自然さを測定。低いほど自然な応答。トレーニング評価にも使われる
指標主な用途使えないケース
ROUGE要約品質の評価翻訳・検索評価・本番監視
BLEU翻訳品質の評価・語列精度要約・検索評価・本番監視
Perplexity流暢さ・自然さの評価検索評価・本番運用監視
試験頻出ポイント:
  • 要約品質の評価 → ROUGEが正解（BLEUは翻訳向け）
  • 翻訳品質の評価 → BLEUが正解（ROUGEは要約向け）
  • 流暢さ・自然さの評価 → Perplexityが正解
  • 財務ニュース要約LLMの評価 → Perplexity + ROUGE + BLEU + Latencyの4指標が必要
  • 法的ドキュメント要約pipelineの評価 → ROUGEとRecallを優先（コンテンツの関連性・カバレッジ）

検索品質指標：Precision・Recall・NDCG・Retrieval Latency

RAGシステムの検索ステップの品質を評価するための指標群。 これらはすべてRetrieverコンポーネントの評価に使う指標であり、生成品質の評価には使わない。

  • Precision（適合率）: 取得されたドキュメント中の関連ドキュメントの割合。高いほどノイズが少ない
  • Recall（再現率）: 全関連ドキュメントのうち取得できた割合。高いほど見落としが少ない
  • NDCG（Normalized Discounted Cumulative Gain）: ランキングの精度を評価。上位に関連ドキュメントが来るほど高スコア
  • Retrieval Latency: 検索ステップの応答時間。ボトルネック特定に使う
パターン意味対処法
Precision高・Recall低関連ドキュメントを取り逃している（見落とし多い）検索範囲の拡大・Embeddingのチューニング
Recall高・Precision低無関係なドキュメントも大量に取得（ノイズ多い）EmbeddingのFine-tuning・フィルタリング改善
試験頻出ポイント:
  • RAG検索システムのrelevanceとボトルネック診断 → Precision・Recall・Retrieval Latencyが正解
  • 法的ドキュメントRAGシステムの評価 → NDCGとRecallの組み合わせ（ランキング品質＋取得率）
  • チャンキング戦略の最適化 → RecallやNDCGなどの評価指標を使用してテスト
  • 「Recall高・Precision低が示すこと」→ 無関係ドキュメントの混入 → より良いEmbeddingまたはフィルタリングで対処

MLflowによる評価・追跡

MLflowはDatabricks上でモデルの評価・追跡・管理を行うためのオープンソースフレームワーク。 RAGシステムの評価では、retrieval指標（precision/recall）の記録から、 生成品質指標（BLEU/ROUGE）の追跡まで、システム全体の評価を一元管理できる。

  • MLflow組み込み評価指標: RAGシステム評価のための標準指標セット（検索・生成それぞれの指標を含む）
  • Retrieval指標の追跡: Precision/Recall等の検索品質指標をMLflowで記録・可視化
  • 評価実験管理: 異なる設定・パラメータでの実験結果を比較・管理
試験頻出ポイント:
  • RAGシステムの正式評価 → 検索と生成を分離したデータセット + MLflow組み込み評価指標が最適
  • MLflowでのRAG評価に最も関連する指標 → 検索関連性指標（precision/recall）+ 応答精度指標（BLEU/ROUGE）
  • DatabricksのRAGアプリでretrieval性能を記録・改善するツール → MLflow
  • MLflowでトレーニング損失や検証精度を追跡してRAGを評価するのは誤り（これらはモデルトレーニング指標）

LLM-as-a-Judge による評価

LLM自身を評価者（judge）として使い、他のLLMの出力品質を評価する手法。 チャンキング戦略の最適化など、定量的指標が難しい評価に活用できる。 ただし主観的であり、専用のテストデータセットを使った評価より信頼性が低い場合がある。

  • チャンキング評価への応用: 特定のチャンクが以前の質問にどれだけ答えられているかをLLM-as-a-judgeで評価し、チャンキングパラメータを最適化する
  • 主観性の問題: LLM-as-a-judgeは主観的であり、作成済みデータセットを使った評価より信頼性が低い
  • 補助的手段としての位置づけ: 専用テストデータセット + MLflow指標が主軸
試験頻出ポイント:
  • チャンキング戦略の最適化アプローチ → RecallやNDCGなどの評価指標でのテスト + LLM-as-a-judgeメトリクスの組み合わせが有効
  • 「RAGシステムの評価はLLM-as-a-judgeだけで十分」は誤り — 主観的で信頼性が低く、データセット + MLflow指標と組み合わせが推奨される

チャンキング戦略の評価・最適化

RAGシステムのドキュメントチャンキング戦略（チャンクの分割方法・サイズ）を評価・最適化するための体系的アプローチ。 直感的なパラメータ選択から、RecallやNDCGなどの評価指標とLLM-as-a-judgeを組み合わせた科学的アプローチへの移行が重要。

  • 評価指標による戦略テスト: RecallやNDCGを用いて複数のチャンキング戦略（段落分割・チャプター分割など）を比較
  • LLM-as-a-judge: 各チャンクが過去の質問にどれだけ答えられているかをLLMで評価し、チャンキングパラメータを最適化
  • embeddingモデル変更との区別: embeddingモデルの変更はセマンティックアライメント改善だがチャンキング戦略の最適化ではない
試験頻出ポイント:
  • チャンキング戦略の体系的最適化 → RecallやNDCGでの評価指標テスト + LLM-as-a-judgeの2アプローチが正解
  • 「分類器追加でretrievalをフィルタリング」は誤り: チャンキング戦略の最適化にはならない
  • 「LLMに統計量でチャンクサイズを決めさせる」は誤り: 構造化されたアプローチが欠ける

Context Length管理とToken Limit問題

LLMにはトークン数の上限（Context Length）が存在し、入力がこれを超えるとエラーが発生する。 RAGシステムではベクトルDBから取得するドキュメントのチャンクサイズや取得レコード数が Context Lengthに大きく影響する。

  • Context Length: LLMが一度に処理できるトークンの最大数
  • BAD_REQUEST / InvalidArgument エラー: token数がcontext lengthを超えた場合に発生
  • チャンクサイズ縮小: 各ドキュメントから取得されるトークン数を減らし、context limit内に収める
  • 取得レコード数の削減: vector databaseから取得するレコード数を制限して合計token数を削減
  • ALiBi（Attention with Linear Biases）: 長いコンテキストに対応するためのモデル再トレーニング手法（token limit問題の解決には不要で複雑すぎる）
試験頻出ポイント:
  • Context Length超過エラーへの対処（モデル変更なし） → チャンクサイズを小さくする + 取得レコード数を削減（2つ選択）
  • embeddingモデルのサイズ変更はContext Lengthエラーの解決策にならない
  • output tokensの削減はinput tokenの超過問題に対処しない
  • コスト・latency優先のembeddingモデル設定 → チャンクサイズに合わせたcontext length（512） + 最小モデルサイズ + 最小embedding次元

Inference Logging と Inference Tables

Inference Loggingはデプロイされたモデルエンドポイントへの推論リクエスト・レスポンスの詳細を記録する仕組み。 DatabricksではInference Tables（推論テーブル）というネイティブ機能として提供され、 Model Servingエンドポイントのリクエスト・レスポンスのログ記録と監視を自動的に行う。

  • Inference Tables: Databricksネイティブの推論ログ記録機能。マイクロサービスによるカスタムログ実装の代替
  • 提供されるインサイト: クエリパターン・応答時間・エラー発生（tokenやBLEUではない）
  • コスト監視への活用: 高トラフィック下でのコストとパフォーマンスの傾向分析に使用
ツール用途Inference Loggingの代替か？
Inference Tables推論リクエスト/レスポンスのログ記録これがネイティブ機能
MLflowモデル評価・実験管理・retrieval指標追跡別用途（評価・追跡）
LakeviewDelta Lakeの監視用× Inference Logsを処理しない
DBSQL（Databricks SQL）データクエリ用× 推論ログ記録には不適切
試験頻出ポイント:
  • Databricksでリクエスト・レスポンスをネイティブにログ記録する機能 → Inference Tables（LakeviewやDBSQL、Vector Searchではない）
  • Inference Loggingが提供するインサイト → クエリパターン・応答時間・エラー発生（tokenやBLEUではない）
  • DatabricksのRAGアプリのシステムパフォーマンスとコスト監視 → MLflow + Inference Logging + GPU使用率監視の3点セット

本番モニタリング指標：Latency・Throughput・エラーレート

デプロイされたLLMアプリケーションの運用パフォーマンスを監視するための3つの主要指標。 言語品質指標（BLEU/ROUGE）やタスク固有の評価指標（Precision/F1）とは明確に区別される。

  • Latency（レイテンシ）: 応答時間。リアルタイム性が求められるシステムで重要
  • Throughput（スループット）: 単位時間あたりに処理されたクエリ数。高負荷シナリオでの評価に必要
  • Error Rate（エラーレート）: エラーの発生頻度。システムの安定性・信頼性を示す
試験頻出ポイント:
  • 本番デプロイLLMのパフォーマンス監視 → Latency・Throughput・エラーレートが正解
  • カスタマーサービス問い合わせアプリの本番パフォーマンス評価 → 単位時間あたりに処理された顧客問い合わせ件数（Throughput）が正解
  • BLEUとperplexityは言語品質・トレーニング指標であり運用監視には使えない
  • 精度やF1スコアはタスク固有の評価指標であり運用監視指標ではない

GPU使用率監視

ハードウェアリソースの効率的な活用を確認し不要なコストを削減するための監視指標。 コスト最適化戦略の一環として、GPU使用率の追跡は高負荷・低負荷両シナリオで有効。

  • GPU使用率（GPU Utilization）: GPU処理能力がどの程度使われているかを示す。低使用率は無駄なコストを示す
  • 動的スケーリングとの組み合わせ: GPU使用率監視と動的スケーリングを組み合わせることで、需要変動に応じたコスト最適化が可能
  • 本番監視の3点セット: MLflow（retrieval指標） + Inference Logging（token/latency） + GPU使用率監視
試験頻出ポイント:
  • DatabricksのRAGアプリのシステムパフォーマンスとコスト監視 → MLflow + Inference Logging + GPU使用率監視の3点セット
  • 高トラフィックQAシステムのコスト最適化 → 動的GPUスケーリング + token使用量監視 + inference logging

スケーラビリティ評価

本番LLMアプリケーションが増加するクエリ量に対応できるか（スケーラビリティ）を評価するための指標と戦略。 主にThroughputとLatencyを様々なワークロード条件で測定することで評価する。 モデルサイズや言語品質指標はスケーラビリティの直接指標にはならない。

  • Throughput（スループット）: 様々なワークロード下での処理能力。高負荷シナリオでの測定が特に重要
  • Latency（レイテンシ）: 応答時間。スケール時に許容範囲内に収まるかを確認
  • 動的GPUスケーリング: 需要に応じてGPUリソースを動的に増減させる
試験頻出ポイント:
  • 本番LLMのスケーラビリティ評価 → 様々なワークロード下でのThroughputとLatencyが最重要
  • LLMのコスト効率比較 → クエリあたりのコスト + Latency + 高負荷シナリオでのThroughputの3指標
  • モデルサイズ単独ではスケーラビリティを評価できない
  • BLEUやperplexityはスケーラビリティ評価には使えない

コスト最適化と監視

LLMアプリケーションの推論コストを管理・最適化するための戦略群。 Token使用量の監視、モデルサイズ選択（クエリ複雑度に応じた適切なモデル）、 クエリのバッチ処理、推論ロギングの活用などが主要な手法。

  • Token使用量監視: クエリあたりのトークン消費量を追跡し、コストの高い操作を特定する
  • モデルサイズ選択: シンプルなクエリには小型LLMを使用し、重要タスクには大型モデルを確保する
  • クエリバッチ処理: 複数のクエリを1リクエストにまとめてAPI呼び出し回数を削減
  • 推論ロギング有効化: クエリコストの監視・最適化戦略の根拠データを収集
  • GPU使用率監視: ハードウェアリソースの効率的な活用を確認し不要なコストを削減
  • 動的GPUスケーリング: 需要に基づいてGPUリソースを動的にスケールし、低需要時の無駄なコストを回避
試験頻出ポイント:
  • 財務ドキュメント処理RAGのコスト最小化 → 小型LLM使用 + クエリバッチ処理 + 推論ロギングの3戦略
  • 高トラフィックQAシステムのコスト最適化 → 動的GPUスケーリング + token使用量監視 + inference logging
  • Context Lengthを増やすとtoken使用量が増加しコストが上昇する（コスト削減にはならない）
  • 頻繁な出力のキャッシュはlatencyを削減するがLLM推論コストには大きな影響を与えない

プロビジョニングスループット vs トークン課金（従量制）

Databricksでのモデルデプロイにおける2つの課金モデル。 リクエスト量が少ない場合はトークン課金の方がコスト効率が高い。

  • プロビジョニングスループット: 一定のリクエスト処理能力を保証するが、使用量が少ない場合はコスト非効率
  • トークン課金（従量制）: 使用量に応じたコスト。低リクエスト量では効率的、高トラフィックでは高コストになりうる
  • コスト効率の判断基準: リクエスト量がプロビジョニングスループットに対して不十分な場合、従量制が最適
試験頻出ポイント:
  • リクエスト量がプロビジョニングスループットに対して不十分な場合のコスト効率化 → トークン課金（従量制）に切り替えが正解
  • 手動スロットリングはコスト効率を最適化しない
  • External ModelsはリクエストスループットのコストEfficiencyに対処しない

モデル選定基準（LLM選択時の評価軸）

ユースケースに適したLLMを選択するための評価基準。 ユースケースごとに異なる評価軸を使うことが重要。

ユースケース主要な評価軸
カスタマーサポートチャットボットクエリあたりのLatency + 推論コスト のバランス
長文ドキュメント要約RAGContext Length（長文処理能力） + 要約品質 + Inference Speed
FAQシステム（精度・コスト・スケーラビリティのバランス）モデルサイズ・アーキテクチャ + Latency指標 + 応答Perplexity
LLMのコスト効率比較クエリあたりのコスト + Latency + 高負荷でのThroughput
試験頻出ポイント:
  • カスタマーサポートLLM選定指針 → クエリあたりのLatencyと推論コスト
  • 長文要約RAGのLLM選定 → Context Length・要約品質・Inference Speedの3要素
  • FAQシステムのLLM選定 → モデルサイズ・Latency指標・応答Perplexityの3要素
  • BLEUスコアはtoken limitとの組み合わせでは要約品質や推論速度に対応しないため不適切

応答の多様性（エンゲージメント指標）

チャットボットのユーザーエンゲージメントとリテンションを高めるための指標。 応答の多様性（Response Diversity）は、同じような応答の繰り返しを避けてユーザーの興味を維持するために重要。 ランダム性（Randomness）とは異なり、文脈に合った多様な応答を生成する能力を指す。

  • 応答の多様性（Response Diversity）: ユーザーの興味を維持し、エンゲージメントとリテンションを向上させる
  • ランダム性との違い: ランダム性はユーザーを混乱させるリスクがあるが、多様性は文脈に合った異なる応答を提供する
  • 応答の繰り返しの問題: 同じ応答の繰り返しはユーザーを飽き飽きさせリテンションを低下させる
試験頻出ポイント:
  • ゲーミングchatbotのエンゲージメント・リテンション向上 → 応答の多様性が正解（ランダム性や応答の繰り返しは不正解）
  • ランダム性はユーザーを混乱させエンゲージメントを低下させる可能性がある

理解度チェックシート

Q1. ROUGE・BLEU・Perplexityの3つの指標はそれぞれ何を評価しますか？ また、これら3つの指標が使えないケースを1つ挙げてください。

▼ 答えを見る

A. ROUGE：要約品質（生成サマリーと参照サマリーのオーバーラップ）、BLEU：翻訳品質（n-gramオーバーラップ）、Perplexity：流暢さ・自然さ（低いほど自然）。 使えないケースの例：本番運用監視（これらは言語品質・トレーニング評価指標であり、Latency/Throughput/エラーレートなどの運用指標ではない）。

Q2. RAGシステムでRecallが高くPrecisionが低い状態は何を示しますか？ また、その改善策は？

▼ 答えを見る

A. Recall高・Precision低は無関係なドキュメントが大量に取得されている（ノイズが多い）ことを示す。 改善策：より良いEmbeddingモデルのfine-tuningまたはフィルタリングの改善。 バッチサイズの変更やLatencyの調整では解決しない。

Q3. RAGシステムが無関係な情報を返す問題を診断するとき、最初に行うべき評価ステップは何ですか？

▼ 答えを見る

A. 取得されたコンテキストの品質評価（RetrieverコンポーネントがどのドキュメントをLLMに渡しているかを確認）。 LLMを変更したり検索アルゴリズムを調整する前に、まず検索結果の関連性を評価する。RAGの問題は検索側にある場合が多いため、検索と生成を分離して診断することが重要。

Q4. DatabricksでModel Servingエンドポイントのリクエスト・レスポンスをネイティブにログ記録する機能は何ですか？ LakeviewやDBSQLが代替にならない理由は？

▼ 答えを見る

A. Inference Tables（推論テーブル）がDatabricksネイティブの機能。 LakeviewはDelta Lakeの監視用でInference Logsを処理しない。 DBSQL（Databricks SQL）はデータクエリ用であり推論ログ記録には不適切。 これらはそれぞれ別の目的に設計されており、Inference Tablesの代替にはならない。

Q5. Context Length超過エラー（BAD_REQUEST）が発生した場合、モデルを変更せずに解決する方法を2つ挙げてください。

▼ 答えを見る

A. ①チャンクサイズを小さくする（各ドキュメントから取得されるトークン数を削減）、②取得レコード数を削減する（ベクトルDBから取得するレコード数を制限して合計token数を削減）。 embeddingモデルのサイズ変更やoutput tokensの削減は無効。ALiBiでのモデル再トレーニングは不要で複雑すぎる。

Q6. リクエスト量がプロビジョニングスループットに対して不十分な場合、コスト効率を改善するための最適な選択肢は何ですか？

▼ 答えを見る

A. トークン課金（従量制）に切り替える。プロビジョニングスループットは一定の処理能力を確保する固定コストモデルであり、リクエスト量が少ない場合はコスト非効率。従量制はトークン使用量に応じたコストなので、低リクエスト量では効率的。 手動スロットリングはリクエスト管理には役立つがコスト効率の最適化にはならない。

Q7. 長文ドキュメント要約RAGのためにLLMを選定する際の3つの主要な評価軸を挙げてください。

▼ 答えを見る

A. ①Context Length（長い文書を一度に処理できるか）、②要約品質（生成された要約の精度・関連性）、③Inference Speed（推論速度）（タイムリーな回答提供に直結）。 BLEUスコアは翻訳評価指標であり、要約品質や推論速度の評価軸として不適切。

Q8. ゲーミングチャットボットのユーザーエンゲージメントとリテンションを高めるために重要な指標は何ですか？ ランダム性との違いも説明してください。

▼ 答えを見る

A. 応答の多様性（Response Diversity）。同じ応答の繰り返しを避けてユーザーの興味を維持し、エンゲージメントとリテンションを向上させる。 ランダム性との違い：ランダム性は文脈に関係なく予測不能な出力を生成するためユーザーを混乱させるリスクがある。 多様性は文脈に合った異なる応答を提供するものであり、品質を保ちながら変化をもたらす。
