Governance

Guardrail・PII保護・ライセンスコンプライアンス・Unity Catalog / RBAC・セキュリティ設計・バイアス対策を扱う。約24問・10トピック。

この分野の概要

Governance 分野は、GenAIアプリケーションの安全・適法・公正な運用のための仕組みを扱う。 Guardrailの種類の使い分け（Safety/Compliance/Security/Contextual）、PII保護のマスキングと暗号化の違い、 ライセンス確認の原則（公開データでも事前確認が必須）、Unity Catalog/RBACによるアクセス制御の設計パターンが頻出。 「後処理よりも前処理」「トレーニングよりも明示的な制御」という設計原則を理解することが合格の鍵。

出題内容詳細

Guardrail（ガードレール）の種類と選択

LLMアプリケーションの出力や動作を制約・監視する仕組み。 不適切・有害・ポリシー違反の出力をブロック・修正するために使われる。 用途に応じて4つの種類があり、試験ではその違いの区別が繰り返し問われる。

種類目的典型的な使用例
Safety Guardrail不適切・有害な出力をブロック・置換する政治的発言のフィルタリング、会社ポリシー違反出力を定型メッセージで置き換える
Compliance Guardrail法的・規制上の基準への準拠を強制するGDPR・HIPAAなどの規制対応、法令遵守の確認
Security Guardrail機密データ・インフラの保護に焦点機密情報の出力防止、セキュリティ上の防御
Contextual Guardrailユーザーのコンテキストに基づいてモデル出力を調整するユーザーの役割・状況に応じた応答の調整
  • Metaprompt（メタプロンプト）: 特定の動作（例：法的アドバイスを避ける）をモデルに指示するシステムプロンプト。Guardrailの実装手段の一つ
  • 後処理フィルター: 生成後の出力から機密情報を検出・削除する。前処理と組み合わせて多層防御を構成
  • Retrieval-based augmentation（検証ステップ）: RAGベースで取得した正確な情報で生成を根拠づけるガードレール。不正確なトラブルシューティング手順の生成防止に有効
試験頻出ポイント:
  • 政治的発言やポリシー違反のブロック → Safety Guardrailが正解（Compliance/Security/Contextualは不正解）
  • 法的アドバイス制限chatbotには Metaprompt（動作制限の指示）+ Disclaimer（免責事項）の2つが有効。トレーニングやtemperature調整では対応不可
  • テクニカルサポートQAの不正確手順防止には 検証ステップ + retrievalベース拡張のガードレールが最善
  • インジェクション攻撃対策は 入力サニタイズ（特殊文字削除・フォーマット検証）が正解。ログ監視や要約モデルは防御にならない

ハルシネーション対策

LLMが事実に基づかない内容（架空の条項・データなど）を生成する「ハルシネーション」を防ぐための技術的・設計的アプローチ。 単一の手法では防げず、複合的な対策が必要。

  • Metaprompt（メタプロンプト）: 「提供されたテキストのみに基づいて生成せよ」という明示的な制約指示。最も直接的なハルシネーション防止手段
  • Strong System Prompt（強力なシステムプロンプト）: モデルの動作を調整し、望ましい出力フォーマット・範囲をガイド
  • Guardrail（出力フィルター）: 生成後の出力を検証・フィルタリング。ハルシネーションされた内容を後処理でブロック
  • Access Control（アクセス制御）: ユーザーのアクセスレベルに基づき利用可能なデータを制限することで、機密データ漏洩リスクを低減
  • Fine-tuning（ファインチューニング）: 自社データで再学習するが、単独ではハルシネーション防止の信頼性がない。補助手段にはなるが単独解決策としては不適切
試験頻出ポイント:
  • ハルシネーション防止にファインチューニング単独では不十分。ガードレール・アクセス制御・system promptの組み合わせが有効
  • 法的契約書の要約でハルシネーション条項を生成する場合、Metapromptで「提供テキストのみ」を指示するのが正解
  • context length増加・事前トレーニング・分類モデルでの検証は根本解決にならない

PII・機密データ保護（マスキング・フィルタリング）

個人を特定できる情報（PII：氏名・電話番号・クレジットカード番号・医療情報など）をアプリケーション処理から保護する技術。 主に前処理段階のマスキング・後処理フィルタリングの2層で実装。

  • データマスキング（Data Masking）: 機密情報をプレースホルダーに置き換え、コンテキストを失わずにプライバシーを確保。分析的価値を維持しながらプライバシーリスクを低減
  • Regexベースのマスキング: 電話番号・クレジットカード情報などの構造化パターンを正規表現で検出してマスク。効率的で正確
  • 入力フィルタリング + 応答バリデーション: 処理前の入力フィルタリングと生成後の応答バリデーションを組み合わせてプライベートデータをブロック。医療チャットボット等に有効
  • 後処理フィルター: 生成済み出力から機密情報を検出して削除。共有前に確実に機密データを除去
  • Unity Catalog + データマスキング: Databricksで開発環境から本番データへの安全なアクセスを提供するためのマスクビュー作成機能
データマスキング暗号化
目的匿名化（コンテキスト維持）データ秘匿（データを読めなくする）
分析価値維持される復号しないと使えない
主な用途処理中のPII保護、開発環境での本番データ代替保存中・転送中のデータ保護
試験頻出ポイント:
  • 顧客サポートチャットボットの機密情報マスク → regexベースのパターン実装が最善
  • 医療チャットボットのプライバシー保護 → 入力フィルタリング + 応答バリデーションの組み合わせが正解
  • 開発環境での安全な本番データアクセス → Unity Catalogのアクセス制御 + データマスキングによるビュー作成が正解
  • Maskingと暗号化の違い：Maskingは匿名化（データの分析価値維持）、暗号化はデータ秘匿（ストレージ中のデータ保護）

データセット・ライセンスコンプライアンス

LLMのトレーニングやRAGアプリケーションに使用するデータセットの法的利用可能性を確認するプロセス。 商用利用の可否・プライバシー規制準拠（HIPAA等）・著作権の確認が主要要素。

  • ライセンス条項の確認: 商用利用・変更・再配布の許可・制限を規定する条項。オープンライセンスであっても条項を必ず確認する
  • HIPAA（医療情報の保護）: 医療記録・患者データを扱う場合に必須のコンプライアンス確認。医療分野での生成AIには必須知識
  • データキュレーターへの事前連絡: 第三者データを使用する場合、使用開始前に許可を確立するかライセンス条項を確認する義務。使用開始後の事後連絡では法的責任が発生しうる
  • 誤情報ドキュメントの除外（前処理段階）: 有害な誤情報を含むとフラグが立てられたドキュメントは、取り込み前のpreprocessingで除外する。モニタリングだけでは不十分
試験頻出ポイント:
  • 「公開データなら自由に使える」は誤り。公開 ≠ 商用利用許可。必ずライセンス確認が必要
  • 「非商用利用はライセンス確認不要」も誤り。法的コンプライアンスは使用目的に関わらず必要
  • 医療記録データセットの法的コンプライアンス確保には ライセンス条件の確認（商用利用可否）+ HIPAA等プライバシー規制への準拠の2点が正解
  • データ使用後のキュレーターへの連絡は「不適切な行動」として出題される。事前連絡・事前確認が原則

セキュリティ（入力バリデーション・サニタイズ・暗号化・認証）

LLMアプリケーションをセキュリティ上の脅威（悪意ある入力・不正アクセス・データ漏洩）から保護するための技術的手段。 入力段階・データ保護・アクセス認証の多層防御が基本。

  • 入力サニタイズ（Input Sanitization）: 特殊文字の削除・データ形式の検証によりインジェクション攻撃を防止。LLMアプリのプロンプトインジェクション対策としても有効
  • 入力バリデーション（Input Validation）: 許容できないパターン・フォーマットを検出するルール実装。敵対的ユーザー入力によるシステムクラッシュを防止
  • 暗号化（保存中・転送中）: 保存中（at rest）および転送中（in transit）のデータを暗号化し、機密情報を保護。プライバシー規制を満たすための基本要件
  • APIキー認証: アクセス権を持つユーザーのみがシステムにアクセスできることを保証。カスタマーサポートアプリ等のコンプライアンスとセキュリティに不可欠
  • LLM Guardrail（機密データ漏洩防止）: 会話エージェントが機密情報を出力しないように制御。暗号化と組み合わせて銀行アプリ等で使用
  • 環境変数（Environment Variables）: MLflowエンドポイントにシークレット・認証情報を安全に渡すための手段。プレーンテキストでの受け渡しは重大なセキュリティリスク
試験頻出ポイント:
  • 悪意ある入力対策はセーフティフィルター（検出・拒否）が最善。インタラクション時間の短縮・コンピュート増加はセキュリティに無関係
  • プレーンテキストでのシークレット受け渡しは絶対NG。環境変数が正解（spark.conf.set()はSpark設定用であり不適切）
  • 患者向け医療チャットボットのコンプライアンス要件：保存・転送中の暗号化 + APIキー認証が必須
  • 銀行アプリの機密情報安全処理：暗号化 + LLM guardrailの組み合わせが正解
  • 「プライベートネットワークアクセスだけでセキュリティは十分」は誤り。暗号化が別途必要

アクセス制御（Unity Catalog / RBAC）

データ・モデル・アプリケーション機能へのアクセスを適切に制限するための仕組み。 Databricks環境ではUnity Catalogがデータガバナンスの中心的ツール。 ロールベースのアクセス制御（RBAC）がLLMアプリでの標準手法。

  • Unity Catalog: Databricksのデータガバナンスツール。アクセス制御・データマスキング・メタデータ管理・モデル登録などを一元管理
  • ロールベース権限（RBAC）: ユーザーまたはチームのロールに応じてアプリケーション機能へのアクセスを制御。マルチユーザーLLMアプリケーションのアクセス管理に最適
  • 開発専用マスクビュー: Unity Catalogを使い、開発環境が本番データに安全かつセキュアにアクセスできる機密フィールドマスク済みビューを作成
  • Delta Sharing: 外部パートナーとのデータ共有ツール。内部開発ワークフロー（開発環境と本番環境間）への利用は不適切
  • MLflow + Unity Catalog でのモデル登録: ガバナンスと再現性向上のため、MLflowにログ → model signatureを追加 → Unity Catalogに登録の順序が正しい
試験頻出ポイント:
  • 開発環境で本番データを安全に使うには Unity Catalog + データマスキングビューが正解。本番テーブルへの直接ポイント・合成データ・完全コピーは不可
  • マルチユーザーアプリの適切なアクセス制御 = ロールベース権限（RBAC）。temperature設定・暗号化はアクセス権管理ではない
  • Unity CatalogへのLLM登録手順：MLflowにログ → model signatureを追加 → Unity Catalogに登録の順序が正しい
  • Unity Catalog登録時に設定すべきメタデータ：モデル名・バージョン・スキーマ・タグ（スキーマ・タグなしの最小限登録は不十分）
  • Delta Sharingは外部パートナー向け。内部開発ワークフローへの適用は誤り

医療・法律チャットボットの安全設計

医療や法律など高リスク分野でのLLMアプリケーション設計では、 通常のチャットボット設計に加えて、緊急時対応・不正確アドバイス防止・コンプライアンスの特別な考慮が必要。

  • 緊急時エスカレーション（Medical Emergency Escalation）: 激しい頭痛・めまい等の潜在的な医療緊急事態の症状が示された場合、情報収集・記事提案より緊急サービスへの誘導を最優先する
  • 法的アドバイス制限（Legal Advice Restriction）: Metapromptで「一般情報のみに回答を制限する」指示と、すべての出力への免責事項（Disclaimer）の追加が有効な組み合わせ
  • RAGによる根拠付け: 検索ベースの拡張により、トラブルシューティング手順の正確性を担保するガードレール
試験頻出ポイント:
  • 医療chatbotで激しい頭痛・めまい等の症状が2日以上続く場合、緊急サービスへの誘導（「地元の緊急サービスに電話してください」）が最適な応答。詳細情報の収集・記事提案は緊急時には不適切
  • 法的アドバイス制限chatbotには Metaprompt（一般情報のみ）+ Disclaimer（免責事項）の2つが有効。法的データセットでのトレーニング・temperature低下・法律用語クエリのブロックは誤り
  • 法律クエリをすべてブロックするのは誤り。chatbotの機能を制限しすぎて使いやすさが低下する

データキュレーション（取り込み前フィルタリング）

RAGアプリケーションやLLMトレーニングに使用するデータをパイプラインに取り込む前に、 有害・不適切・誤ったコンテンツを除外・浄化するプロセス。 後処理より前処理での対応が優先される。

  • 不適切言語の取り込み前フィルタリング: 問題のあるテキスト（不適切な言語・炎上コンテンツ）を取り込み前に検出・除去するフィルタリングツール。プレースホルダーへの置換ではなく完全な除去が推奨
  • 誤情報ドキュメントの前処理除外: 有害な誤情報を含むとフラグが立てられたサードパーティドキュメントをpreprocessingで除外。出力モニタリングや免責事項追加では不十分
  • アップストリームデータキュレーション: RAGシステムの炎上的出力に対処するため、取り込み前の手動レビューによるアップストリームデータのキュレーションが最も効果的
  • バイアスのあるエントリの削除・再重み付け: 採用推薦等の公平性が必要なユースケースでは、データセットのバイアスを分析して問題のあるエントリを削除または再重み付け
試験頻出ポイント:
  • 不適切コンテンツ対策は「後処理監視」ではなく「前処理フィルタリング」が正解。取り込んだ後にモニタリングしても、すでにシステムに影響している
  • RAGの炎上的出力対策は 取り込み前の手動レビューによるキュレーションが最善。アクセス制限・ユーザー通知・更新頻度増加は無効
  • バイアス対策はデータセットサイズ削減ではなく、バイアスエントリの分析・削除・再重み付け

ヘルプデスク・サポートシステムのデータソース設計

ヘルプデスクやカスタマーサポートチャットボットのために、 チケットの根本原因と解決策を効率的に特定できるデータソースを選択・設計するプロセス。

  • call_detail テーブル: root_causeとresolutionフィールドを持つ全通話詳細スナップショット。チケットの根本原因・解決策の主要データソース
  • transcript Volume: 録音のwavファイルとテキスト文字起こしを含むUnity Catalog Volume。詳細な通話コンテキストを提供
  • call_rep_history: 担当者のパフォーマンス指標を追跡。チケット解決コンテキストは提供しない
  • call_cust_history: 顧客の使用状況を追跡。特定チケットの根本原因・解決策特定には不向き
  • maintenance_schedule: 障害・計画メンテナンスを追跡。特定チケットへの直接対応には役立たない
試験頻出ポイント:
  • チケット根本原因・解決策特定に最適なデータソースは call_detail（root_cause/resolutionフィールド）+ transcript Volume（通話コンテキスト）の2つ
  • 担当者パフォーマンス指標（call_rep_history）は解決コンテキストを提供しないため不適切

バイアス対策と公平性確保

採用推薦・融資審査など、人の生活に影響する意思決定システムでのLLM出力において、 不公平・差別的な結果を生まないためのデータ管理アプローチ。

  • バイアス分析と除去・再重み付け: データセットの潜在的バイアスを分析し、バイアスのあるエントリを削除または再重み付けして公平性を確保
  • 事前学習済みモデルのバイアス継承: 事前学習済みモデルもその学習データのバイアスを反映するリスクがある。Fine-tuningで悪化する可能性も
試験頻出ポイント:
  • バイアス対策はデータセットの分析と問題エントリの削除・再重み付けが正解。モデルへの依存・データサイズ削減・バイアス無視は誤り
  • 採用推薦システムのバイアスを「モデルがクエリで自動補正する」と考えるのは誤り。元データのバイアスがそのまま出力に反映される
  • 「事前学習済みモデルのみを使えばデータのバイアスを回避できる」は誤り。事前学習済みモデルも元の学習データのバイアスを引き継ぐ

A. Safety Guardrail。不適切・有害な出力をブロック・置換することに特化している。 Compliance Guardrailとの違い：Safety = 不適切出力のブロック（ポリシー違反、有害コンテンツ）、Compliance = 法的・規制上の基準への準拠を強制（GDPR・HIPAA等の法令遵守）。目的が異なる。

Q2. データマスキングと暗号化の違いを説明し、それぞれが適切なユースケースを1つずつ挙げてください。

▼ 答えを見る

A. データマスキング：機密情報を匿名化（プレースホルダーに置換）し、データの分析価値を維持しながらプライバシーリスクを低減。例：開発環境での本番データの代替ビュー作成。 暗号化：データを復号キーなしには読めない形式に変換し、保存中・転送中のデータを秘匿する。例：医療チャットボットのデータ保存・転送時のHIPAA準拠。

Q3. 公開データセットをビジネスアプリケーションに使う場合、法的リスクを避けるために最初に確認すべき事項は何ですか？

▼ 答えを見る

A. ライセンス条項の確認（特に商用利用の可否）。「公開データ = 自由に使える」は誤りで、公開されていても商用利用を禁じているライセンスがある。 また、医療記録データセットの場合はHIPAA等のプライバシー規制への準拠も確認が必要。データキュレーターへの連絡は使用開始前に行うことが原則。使用後の連絡は不適切な行動として出題される。

Q4. 医療チャットボットで激しい頭痛とめまいが2日間続くと訴えるユーザーに対して、最適な応答設計はどのようなものですか？

▼ 答えを見る

A. 緊急サービスへの誘導（「地元の緊急サービスに電話してください」）を最優先する。 激しい頭痛・めまいは潜在的な医療緊急事態（脳卒中等）の可能性があるため、情報収集（年齢・活動・症状）や記事提案より緊急サービスへの誘導が優先される。 情報収集は緊急時には致命的な遅延を招く可能性がある。

Q5. 開発環境のチームが本番データにアクセスする必要がある場合、Databricksでの最適なアクセス制御設計を説明してください。

▼ 答えを見る

A. Unity Catalog + データマスキングビューの作成。本番データの機密フィールドをマスクしたビューを作成し、開発環境はそのビューにアクセスする。 本番テーブルへの直接アクセス・合成データ・完全コピーは不可。 Delta Sharingは外部パートナー向けのデータ共有ツールであり、内部開発ワークフローへの適用は不適切。

Q6. RAGシステムが炎上的・不適切な出力を生成している場合、最も効果的な対処法は何ですか？ 後処理監視よりも前処理が優先される理由も説明してください。

▼ 答えを見る

A. 取り込み前の手動レビューによるアップストリームデータのキュレーション（不適切なコンテンツを取り込み前に除外）が最善。 後処理監視より前処理が優先される理由：後処理監視では問題のあるコンテンツがすでにシステムに取り込まれており、RAGパイプライン全体に影響が及んでいる。 前処理で除外することで、問題コンテンツがシステムに入り込む前に根本から対処できる。

Q7. MLflowとUnity Catalogを使ってLLMをDatabricksに登録する正しい手順を説明してください。

▼ 答えを見る

A. ① MLflowにモデルをログ（mlflow.log_model等でモデルをMLflow実験に記録）→ ② model signatureを追加（入出力スキーマを明示）→ ③ Unity Catalogに登録（モデル名・バージョン・スキーマ・タグを設定）。 Unity Catalogで先にフォーマット定義をしてからMLflow登録する順序は誤り。スキーマ・タグなしの最小限登録も不十分。

Q8. ハルシネーション防止に「ファインチューニング単独」が不十分な理由は何ですか？ 効果的な組み合わせを説明してください。

▼ 答えを見る

A. ファインチューニング単独が不十分な理由：ファインチューニングはドメイン知識の向上には役立つが、架空コンテンツの生成（ハルシネーション）を確実に防ぐ保証を提供しない。 効果的な組み合わせ：① Metapromptで「提供されたテキストのみに基づいて生成せよ」と明示的に制約する、② Guardrail（出力フィルター）で生成後の出力を検証・フィルタリングする、③ アクセス制御でユーザーのアクセスレベルに基づいてデータを制限する、という複数の手法の組み合わせが有効。
