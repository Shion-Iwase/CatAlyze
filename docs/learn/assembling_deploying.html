MLflow × Unity Catalog × Model Serving — モデルデプロイの基本フロー

Databricks における標準的なモデルデプロイフロー。MLflow でモデルをログし、Unity Catalog に登録した後、Model Serving でエンドポイントを起動する。各ステップが依存関係を持つため、順序が試験で繰り返し問われる。

  • MLflow log：mlflow.log_model() でモデル・パラメータ・メトリクスを記録。シグネチャ（入出力スキーマ）も同時に登録するのが推奨
  • Unity Catalog への登録：mlflow.register_model() でモデルレジストリに登録。メタデータとして モデル名・バージョン・スキーマ・タグ・説明 を付与
  • Model Serving エンドポイント：Unity Catalog から登録済みモデルを選択してサービングエンドポイントを作成
  • モデルシグネチャの目的：latency 削減でも logging 有効化でもなく、「正しいフォーマットで入力を受け取り出力を返すことの確保」

正しいデプロイ順序：

  1. mlflow.log_model()（シグネチャ付き）
  2. mlflow.register_model()（Unity Catalog へ名前・バージョン・スキーマ・タグ付き）
  3. Model Serving エンドポイント起動
試験頻出ポイント
  • 「MLflow log → Unity Catalog 登録 → Serving」の3ステップの順序が最重要
  • RAG デプロイの最初のステップは ベクトル embedding の作成（エンドポイント設定や Unity Catalog 登録より先）
  • 「LLM をデプロイしてから embedding を設定する」順序は不正解
  • Unity Catalog の3大メリット：ガバナンス強化・バージョン管理・アクセス制御

MLflow PyFunc — カスタム前処理・後処理

Python ベースのカスタムモデルクラスを MLflow に統合する仕組み。predict() 関数の中に前処理・推論・後処理を一体化してカプセル化できる。デプロイ後も一貫したデータ変換が保証される。

  • 前処理（preprocessing）：入力データのクリーニング・正規化・標準化を推論前に実行
  • 後処理（postprocessing）：モデルの生スコアをユーザーフレンドリーなフォーマット（例：「承認」「拒否」）に変換
  • predict() 関数：前処理 → モデル呼び出し → 後処理の3ステップを1つの関数で実装

ワークフロー：入力受信 → 前処理（クリーニング・正規化） → モデル推論 → 後処理（フォーマット変換） → 出力返却

試験頻出ポイント
  • PyFunc の有用性：「API コスト最小化」「再トレーニング削減」ではなく、「推論の前後にデータを変換して互換性と最適化された出力を確保する」
  • 前処理だけ・後処理だけでは不完全。両方向の変換が必要
  • 「生スコアをそのまま返す」は不正解 → ユーザーが混乱しアクション可能なインサイトが欠如

Vector Search / Vector Store — 構築と運用

テキストや画像を高次元ベクトルに変換し、意味的類似度に基づく高速検索を実現するシステム。Databricks では VectorSearchClient の create_delta_sync_index() が Databricks 管理の embedding 同期に使われる。

  • 基本フロー：ドキュメント前処理 → embedding 生成 → ベクトルストアへ保存 → retriever によるクエリ → 最近傍探索
  • create_delta_sync_index()：Delta テーブルと embedding を同期する Databricks の標準 API（create_vector_store() や store_embeddings() は存在しない）
  • FAISS：外部ベクトルストア。Databricks 管理外の環境で使用
  • 距離メトリクス最適化：コサイン類似度・ユークリッド距離などを最適化することでクエリパフォーマンスが向上
  • 高スループット構成：GPU アクセラレーション ＋ 分散クラウドインフラ ＋ 高性能ベクトルストア DB
試験頻出ポイント
  • Databricks 管理の embedding 同期 → create_delta_sync_index() が正解（他の関数名は誤答選択肢）
  • 検索精度改善 → セマンティック類似度最適化 retriever ＋ ドメイン固有 embedding モデル（チャンクサイズ増加・ランダムサンプリングは不正解）
  • 「retriever を最初に構築する」は不正解 → embedding とインデックスの準備が先
  • CPU のみのローカルサーバーは高スループット要件を処理できない → GPU ＋ 分散クラウドが必要

LangChain — チェーン設計とコンポーネント構成

Retriever・LLM・PromptTemplate などのコンポーネントをチェーンとして結合し、複数ステップの処理パイプラインを構築するフレームワーク。RAG・会話チャットボット・QA システムの実装に広く使用される。

  • 必須コンポーネント（基本チェーン）：PromptTemplate ＋ Retriever ＋ LLM の3点セット
  • LLMChain：llm= と prompt= の両方が必須パラメータ
  • 会話履歴対応：Retriever ＋ 会話履歴を含む PromptTemplate ＋ LLM でターン間のコンテキスト保持
  • 長いドキュメント対応：チャンキングでドキュメントを分割し、Retriever で関連チャンクのみ取得してから LLM へ渡す
試験頻出ポイント
  • 最小構成：Retriever ＋ LLM ＋ PromptTemplate の3点（要約モデルや分類モデルは代替にならない）
  • 医療 QA：Retriever（ガイドライン取得）＋ LLM（説明生成）が正解。ルールベース・分類モデルは不正解
  • 「ハードコードされたテンプレートで固定応答を返す」は不正解 → 多様なクエリへの柔軟性がない
  • 「Retriever なしの embedding だけ使う」は不正解 → 動的な取得・処理ができない

RAG パイプライン — デプロイ手順と構成要素

外部知識ソースからドキュメントを検索し、LLM の生成に組み込む手法。デプロイには決まった順序があり、embedding 生成・インデックス作成・Retriever 設定・エンドポイントデプロイの順序が試験で繰り返し問われる。

  • 完全デプロイフロー：依存ライブラリのインストール → ドキュメント前処理・チャンキング → embedding 生成 → Vector Search インデックスへ保存 → Retriever 設定 → Retriever ＋ LLM 統合 → エンドポイントデプロイ
  • ドメイン固有 embedding：汎用 embedding より精度が高く、法律・医療などの専門領域では必須
  • 大規模 RAG の必要インフラ：コンピューティングパワー ＋ ストレージ ＋ 監視 ＋ 高速ネットワークの4要素
試験頻出ポイント
  • RAG デプロイの最初のステップは必ず ベクトル embedding の作成
  • 順序は「embedding 生成 → Vector Search インデックス保存 → Retriever＋LLM 統合 → エンドポイントデプロイ」
  • 「LLM を先にデプロイし後で Retriever と embedding を設定」は不正解
  • 「Retriever を省略して embedding モデルのみ使用」は不正解 → 精密なドキュメントルックアップに Retriever が不可欠

セキュリティ・アクセス制御 — エンドポイント保護

LLM アプリケーションのデプロイでは、API キー認証・RBAC・保存中と転送中の暗号化の3層が標準的なセキュリティ構成。認証情報は環境変数で渡すのが原則で、コードへの平文埋め込みは厳禁。

  • RBAC（Role-Based Access Control）：ユーザーロールに基づいて権限を割り当て、特定機能へのアクセスを制御。マルチユーザー・マルチチームのアプリケーションで必須
  • 暗号化：保存中（at rest）と転送中（in transit）の 両方 でデータを暗号化。どちらか一方だけでは不完全
  • 環境変数でのシークレット管理：API キーや接続文字列はコードに平文で書かず環境変数として渡す
  • エンタープライズコンプライアンス：Unity Catalog 登録 ＋ 監査ログ（Audit Log）の2点セット
セキュリティ要件適切な対策
マルチユーザーアクセス制御RBAC（個別のロールベース権限管理）
機密データの保護暗号化（保存中 ＋ 転送中の両方）
API キーの安全管理環境変数（コードへの平文埋め込みは厳禁）
エンタープライズ監査Unity Catalog 登録 ＋ 監査ログ
試験頻出ポイント
  • マルチユーザーアクセス制御の問題では必ず RBAC が正解
  • 「共有アクセスキーで全ユーザー認証」は不正解 → 個別の追跡・制御ができない
  • 「temperature 設定でアクセス制御」は不正解 → temperature はランダム性に影響するがアクセス制御には無関係
  • パブリックベクトルストアは機密データを含む RAG では不可

スケーラビリティ・インフラ — 高トラフィック対応

LLM アプリケーションのトラフィックは変動的なため、固定リソースではなく Autoscaling による水平スケーリングが推奨される。リアルタイム用途ではリソース確保・レート制限管理・latency 監視の3点を常に考慮する。

  • Autoscaling（水平スケーリング）：変動するワークロードに応じてコンピューティングリソースを動的に調整。API 使用量が急増する場合の最適解
  • 固定リソースの問題点：ピーク需要に合わせた固定リソースは低需要時に過少利用、高需要時に過負荷を招く
  • GPU アクセラレーション：LLM 推論と embedding 処理には GPU が必要。CPU のみでは高スループット要件を満たせない
  • リアルタイムサービングの3要件：十分なコンピューティングリソース確保 ＋ API レート制限管理 ＋ latency 監視
  • レート制限（Rate Limiting）：流量制御であり、スケーリングとは別物。レート制限のみではスケーラビリティが向上しない
試験頻出ポイント
  • 変動ワークロードへの対応 → Autoscaling ポリシーによる水平スケーリング（固定リソース・レート制限のみは不正解）
  • 高スループットインフラ → GPU ＋ ストレージ最適化を備えた分散クラウドインフラ（CPU ローカル・単一インスタンスは不正解）
  • 「context length を短縮してコスト節約」は不正解 → 応答品質が低下しスケーラビリティにも対処しない

モデル選択 — ユースケース別の最適解

タスクやコスト・品質要件に応じて最適なモデルを選択する判断軸が問われる。コード生成・高品質オンプレミス・ドメイン特化・多言語対応など、各ユースケースに対応した推奨モデルまたはアプローチが存在する。

多言語チャットボット多言語パフォーマンスベンチマークスコアを優先

  • コード生成 → CodeLlama-34B が正解（汎用 LLM は不正解）
  • コスト効率 → ドメイン特化小型 LLM ＋ キャッシュ（最大モデルは不正解）
  • 多言語 → 多言語ベンチマークスコアが最重要（モデルサイズや latency は二次的）
  • 「latency 優先なのに 14GB・4096 次元 embedding を選ぶ」は不正解 → コスト・latency の優先事項に矛盾

Unity Catalog — モデル管理・ガバナンス

Databricks の中央集権型データ・AI ガバナンスプラットフォーム。モデルのバージョン管理・アクセス制御・監査ログ・メタデータ管理を一元化する。MLflow のモデルレジストリと統合されエンタープライズグレードのモデル管理が可能。

  • 3大メリット：ガバナンス強化・バージョン管理・アクセス制御
  • 登録時のメタデータ：モデル名・バージョン・スキーマ（入出力定義）・タグ・説明
  • 監査ログ（Audit Log）：エンドポイントのクエリと応答を追跡し、コンプライアンスとトラブルシューティングを支援
  • エンタープライズ RAG のコンプライアンス：Unity Catalog 登録 ＋ 監査ログの2点セット
試験頻出ポイント
  • 「Unity Catalog 登録はリアルタイムロギングを有効化する」は不正解 → リアルタイムロギングは Inference Tables の機能
  • 「メタデータ設定をスキップしてデプロイ」は不正解 → ガバナンス追跡に不可欠なメタデータを省略すべきでない
  • 「コスト削減のためにパブリックベクトルストアを使用」は不正解 → データセキュリティとコンプライアンスを損なう

Inference Logging / 推論監視

デプロイ済み LLM アプリケーションの品質・コスト・パフォーマンスを継続的に監視するためのログ記録機能。Databricks では Inference Tables がネイティブのリクエスト・レスポンスログ記録機能として提供されている。

  • Inference Tables：Databricks のネイティブ機能。マイクロサービスによるカスタムログの代替として使用可能
  • 監視対象：クエリパターン・応答時間（latency）・エラー発生状況・トークン使用量
  • コスト最適化の3点セット：動的 GPU スケーリング ＋ トークン使用量監視 ＋ Inference Logging
ツール用途Inference Log 記録に適しているか
Inference Tables推論リクエスト・レスポンスのログ記録・監視✅ 適切
LakeviewDelta Lake の監視❌ 不適（用途が異なる）
DBSQLデータクエリ❌ 不適（ログ記録機能ではない）
Vector Searchセマンティック検索❌ 不適（用途が異なる）
試験頻出ポイント
  • Databricks ネイティブのログ記録機能 → Inference Tables（マイクロサービスの代替）
  • 「LLM ファインチューニングでスループット・コストを改善する」は不正解 → ファインチューニングはスケーラビリティコストに直接対処しない

Feature Serving — リアルタイムデータアクセス

LLM アプリケーションにリアルタイムデータへのアクセスを提供するコンポーネント。事前計算された静的な embedding や知識ベースだけでは対応できない、刻々と変わるライブデータ（スポーツスコア・在庫状況・株価など）を LLM に供給するために使用する。

  • 主要用途：リアルタイム更新が必要なデータ（スポーツスコア・スポーツ統計・在庫・株価）を LLM アプリに供給
  • 静的知識との違い：通常の RAG は過去のドキュメントを索引化した静的な知識を使う。Feature Serving はリアルタイムの動的データにアクセスできる
コンポーネント用途
Feature Servingリアルタイム数値データ・特徴量の供給（スポーツスコア・株価など）
Vector Searchドキュメントの意味的検索（静的な知識ベース）
ファインチューニングモデルのパラメータ更新（リアルタイムデータ供給とは別物）
試験頻出ポイント
  • 「ライブスポーツスコアを表示する LLM アプリ」→ Feature Serving が正解
  • 「Vector Search でリアルタイムスポーツスコアを提供」は不正解 → Vector Search はドキュメント意味検索用
  • 「モデルをリアルタイムデータでファインチューニングする」は不正解 → 都度リアルタイム更新には非現実的

トークンベース課金 / コスト最適化

LLM アプリケーションのコストはトークン使用量・コンピューティングリソース・リクエスト数に依存する。リクエスト量が少ない場合はトークンベース（pay-per-use）が有利で、リクエスト量が多い場合はプロビジョニングスループットが経済的になる。

  • トークンベース（pay-per-use）：リクエスト数が少ない場合に cost-efficient。固定コストなし
  • プロビジョニングスループット：高トラフィック・高スループット環境向け。一定のキャパシティを確保
  • バッチ処理：複数クエリをまとめて送信して API コールを削減
  • キャッシュ：頻繁なクエリをキャッシュして重複計算を避け、latency とコストを削減
  • 小型ファインチューニング済みモデル：大型汎用モデルの代わりにドメイン特化の小型モデルを使用
状況最適な課金・構成
リクエスト数が少ないトークンベース（pay-per-use）
高スループット・高トラフィックプロビジョニングスループット
コスト優先の embedding 設定Context Length 512 ＋ モデルサイズ 0.13GB ＋ 次元 384
試験頻出ポイント
  • 「リクエストが少ないのにプロビジョニングスループットを選ぶ」は不正解 → 過剰投資
  • コスト最適化の3点セット：動的 GPU スケーリング ＋ トークン使用量監視 ＋ Inference Logging
  • 「最大の事前学習済みモデルを使う（コスト効率を優先する場面で）」は不正解 → ドメイン特化用途では過剰

Q2. MLflow PyFunc の predict() 関数に含める3つの処理を答えてください。また、生スコアをそのまま返してはいけない理由は？

▶ 答えを見る

A. ① 前処理（入力データのクリーニング・正規化）→ ② モデル推論 → ③ 後処理（ユーザーフレンドリーなフォーマット変換）の3つ。生スコアをそのまま返してはいけない理由：ユーザーが混乱し、アクション可能なインサイトが欠如するため（例：0.87 というスコアより「承認」と返すべき）。

Q3. Databricks Vector Search で Delta テーブルと embedding を同期するための正しい API は何ですか？

▶ 答えを見る

A. create_delta_sync_index()。試験では create_vector_store() や store_embeddings() などの存在しない関数名が誤答選択肢として登場するので注意。

Q4. マルチユーザー環境でエンドポイントへのアクセス制御を実装する場合、何が正しい方法ですか？「共有アクセスキーで全ユーザーを認証する」が不正解な理由も答えてください。

▶ 答えを見る

A. 正しい方法は RBAC（ロールベースアクセス制御）。共有アクセスキーが不正解な理由：全ユーザーが同一キーを使うため、個別のアクセス追跡・権限制御ができない。

Q5. 変動するトラフィックに対応するためのスケーラビリティ対策として何が正しいですか？「レート制限の実装だけで対応する」が不正解な理由も答えてください。

▶ 答えを見る

A. 正しい対策は Autoscaling ポリシーによる水平スケーリング。レート制限だけが不正解な理由：レート制限はリクエストの流量を制御するだけであり、コンピューティングリソース自体のスケーリングは行わないため、高トラフィック時にシステムが過負荷になる。

Q6. Databricks でデプロイ済みモデルのリクエスト・レスポンスをログ記録するためのネイティブ機能は何ですか？Lakeview や DBSQL が不正解な理由も答えてください。

▶ 答えを見る

A. Inference Tables が正解。Lakeview が不正解な理由：Delta Lake の監視用ツールであり、推論ログ記録機能ではない。DBSQL が不正解な理由：データクエリ用ツールであり、ログ記録機能ではない。

Q7. リアルタイムのスポーツスコアを表示する LLM アプリを構築する場合、Vector Search と Feature Serving のどちらを使うべきですか？その理由は？

▶ 答えを見る

A. Feature Serving が正解。理由：Vector Search は過去のドキュメントを索引化した静的な知識ベースへのセマンティック検索用であり、刻々と変わるリアルタイムデータへのアクセスには対応できない。Feature Serving はリアルタイムの動的データ（スポーツスコア・株価・在庫など）を LLM アプリに供給するために設計されている。

Q8. Unity Catalog にモデルを登録する際に付与するべきメタデータを4つ挙げてください。また Unity Catalog の3大メリットは何ですか？

▶ 答えを見る

A. メタデータ：モデル名・バージョン・スキーマ（入出力定義）・タグ（説明も含む）。Unity Catalog の3大メリット：ガバナンス強化・バージョン管理・アクセス制御。
