## 1. プロンプトエンジニアリング

### 定義・概要
プロンプトエンジニアリングとは、LLMから望ましい出力を引き出すために入力テキスト（プロンプト）を設計・最適化する技術。明示的な指示・出力フォーマット指定・few-shot examples（例の提示）・コンテキストの追加が主な手法。

### サブ概念・手法
- **明示的指示**: 「簡潔で事実に基づいた回答を提供してください」など、具体的な動作を指定する
- **出力フォーマット指定**: 箇条書き・フィールド指定・特定構造の要求（例：「参加者・主要決定事項・次のステップを含むサマリーを提供してください」）
- **few-shot examples**: 期待する出力形式の具体例をプロンプト内に含める（例：「参加者：アリス；決定事項：予算承認」）
- **コンテキスト拡張 (Context Augmentation)**: ユーザー入力に関連する追加情報（注文日・ID・配送時間など）をプロンプトに含める
- **metaprompt（システムプロンプト）**: モデルの振る舞い全体を定義するシステムレベルの指示（後述「ガードレール」参照）
- **汎用プロンプトの回避**: 「クエリに答えてください」のような一般的な指示は一貫性を欠く

### 試験頻出ポイント
- **一貫した出力が欲しい場合**: 明示的な指示 + 具体的な例を含む構造化プロンプト
- **特定フォーマットで出力させたい場合**: フォーマット（箇条書きなど）とコンテンツ要件の両方をプロンプトで明示
- **事実の正確さを優先する場合**: 創造性を促す指示ではなく、「簡潔で事実に基づいた」を明示
- **マルチエージェントの機能公開**: エージェントの機能と使用例を明示したプロンプトテンプレートを構築する
- **データベース操作エージェント**: 汎用指示ではなく、利用可能な機能（create_record・update_record・delete_record）の一覧と使用方法を明示

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「一般的な指示で十分」と判断する | 一般的なプロンプトは不完全・不一貫な出力につながる |
| 「創造性を指示する」 | 事実精度が必要な場合に創造性を促すと正確さが下がる |
| 「デフォルト動作に頼る」 | モデルのデフォルトはアプリ要件に沿わない可能性がある |
| temperature調整でトーンを制御しようとする | temperatureはランダム性に影響するがトーンを強制しない |

## 2. RAGアーキテクチャとパイプライン設計

### 定義・概要
RAG（Retrieval-Augmented Generation）は、LLMの生成とベクトルストアからの文書検索を組み合わせたアーキテクチャ。モデルの知識限界を補い、最新情報や専門知識に基づいた回答生成を可能にする。ハルシネーション（幻覚）の防止に特に有効。

### サブ概念・手法
- **パイプライン構成要素**: Embedding モデル → ベクトルストア → Retriever → LLM（生成）
- **正しいデプロイ順序**:
  1. ドキュメントの前処理・Embeddingモデル準備
  2. Embeddingの生成
  3. ベクトルストアへの保存・インデックス作成
  4. Retrieverの構築
  5. エンドポイントのデプロイ
- **セマンティック検索**: ベクトルデータベースによる意味ベースの検索（キーワード一致ではなく）
- **引用の追加**: 取得したコンテンツに関連する引用をパイプラインに含めることで透明性向上
- **フォールバックメカニズム**: 未解決のクエリを人間のエージェントにエスカレーション
- **ハルシネーション防止**: RAGによるグラウンディングが最も効果的な手法
- **チェーンコンポーネント（医療QAシステムの例）**: Retriever（ガイドラインセクション取得）+ LLM（説明付き応答生成）

### 試験頻出ポイント
- **パイプライン順序問題**: 必ず「前処理・Embedding生成 → ベクトルストア → Retriever → エンドポイントデプロイ」の順
- **「先にエンドポイントをデプロイして後でEmbeddingを設定」は不正解**
- **キーワード検索 vs セマンティック検索**: 法的・医療ドキュメントなど微妙なクエリにはセマンティック検索（ベクトルDB）が必要
- **ルールベースシステムは不適切**: 動的なクエリや微妙なニュアンスへの対応が不足
- **回答が曖昧・引用が不足している場合**: temperatureやファインチューニングではなく、引用を含めるステップをパイプラインに追加

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「先にエンドポイントデプロイ、後でEmbedding設定」 | EmbeddingとベクトルストアはデプロイFORを完了してから |
| 「LLMのファインチューニングで検索精度を上げる」 | ファインチューニングは生成を改善するがretrievalには対処しない |
| 「キーワード検索で十分」 | 言い換えや同義語を含む関連ドキュメントを見落とす |
| 「ルールベースシステムで検索」 | 柔軟性がなく微妙な法的・医療クエリに対応できない |
| 「引用が不足しているのでファインチューニング」 | ファインチューニングは引用問題を直接解決しない |

## 3. Embeddingモデル選択と最適化

### 定義・概要
Embeddingモデルはテキストを高次元ベクトルに変換し、意味的類似度に基づいた検索・推薦を可能にする。用途によって選択すべきモデルの特性が異なる。

### サブ概念・手法
- **ドメイン固有のEmbeddingモデル**: ドメイン固有データで学習されたモデルは汎用モデルより高精度（医療・法的・財務ドメイン）
- **デュアルエンコーダー**: クエリとドキュメントを別々にエンコードするアーキテクチャ（バイエンコーダー）
- **Sentence Transformers**: 文全体のセマンティックな関係をキャプチャするモデル。チャットボットクエリの意図識別に最適
- **Word2Vec・GloVe**: 単語レベルのEmbeddingで文レベルのセマンティックタスクには不適
- **BERT-baseの分類用モデル**: セマンティック検索のEmbedding生成には最適化されていない
- **ANN（近似最近傍）検索**: 高速ルックアップと高精度のバランスをとる（FAQシステムに有効）
- **Context Length**: 長い研究論文には長いContext LengthをサポートするEmbeddingモデルが必要
- **Embedding次元とコスト**: 次元数が大きいほど計算コストが高い（512tokenチャンクなら次元384・モデル0.13GBが最小コスト構成）

### モデル選択基準の早見表
| ユースケース | 優先すべき基準 |
|---|---|
| 文レベルのセマンティック理解（チャットボット意図） | Sentence Transformers |
| 長い研究論文の検索エンジン | Context Length + セマンティック類似度精度 |
| FAQの高速・高精度検索 | セマンティック類似度ベンチマーク + ANN検索 |
| 推薦システム（製品説明・レビュー） | Context Length + セマンティックな理解力 |
| 法的ドキュメントのエンティティ認識 | ベンチマーク精度 + 法的データセットでのファインチューニング機能 |
| コスト・Latency優先（512tokenチャンク） | Context Length 512 + モデルサイズ 0.13GB + 次元384 |

### 試験頻出ポイント
- **Word2Vec・GloVeは文レベルタスクに不適**: 個々の単語のEmbeddingで文全体の意味をキャプチャしない
- **「コスト節約のため小さいモデルを使う」は不正解**: 精度を損なう
- **「速度よりtokenの長さを優先」は不正解**: 短いFAQには無関係
- **長い文書には長いContext Lengthが必要**: 推論コストよりContext Lengthとセマンティック精度が優先

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| チャットボット意図識別に「GloVe」や「Word2Vec」を選ぶ | 単語レベルで文全体のセマンティクスをキャプチャできない |
| 「BERT-base分類用モデルを使う」 | 分類用は検索Embedding生成に最適化されていない |
| 長い文書に「推論コスト」を最優先する | Context LengthとセマンティCk精度が実際の性能を決める |

## 4. マルチターン対話・ステートトラッキング

### 定義・概要
マルチターン対話とは、複数のやり取りを通じてコンテキストを維持しながら会話を続けるシステム。各ターンで過去の会話内容を記憶・参照することが品質の鍵。

### サブ概念・手法
- **ステートトラッキング**: 以前の対話ターンのコンテキストを保持するメカニズム
- **メモリ**: ターンをまたいでユーザーコンテキストを保存する仕組み
- **コンテキストリセットの問題**: 各ターン後にコンテキストをリセットすると断片的な会話になる
- **マルチターンでの一貫性**: 前のターンの重要情報を保持することで対話品質が向上

### 試験頻出ポイント
- **チャットボットが以前のターンを忘れる問題**: ステートトラッキング（メモリ）の実装が解決策
- **「tokenの上限を増やす」は不正解**: 入力サイズに影響するがマルチターンのコンテキスト問題は解決しない
- **「temperatureを下げる」は不正解**: ランダム性に影響するがコンテキスト保持には影響しない
- **ステートトラッキングが重要な理由**: コンテキストを保持し、一貫性のある関連性の高い応答を実現するため

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「tokenの上限を増やす」でマルチターン問題を解決 | token上限は入力サイズに影響するがコンテキスト保持の問題は別 |
| 「temperatureを下げる」 | ランダム性の調整であり、コンテキスト記憶とは無関係 |
| 「各ターン後にコンテキストをリセット」 | 断片的で一貫性のない応答になる |

## 5. LangChainとチェーン設計

### 定義・概要
LangChainはLLMを使ったマルチステップのワークフロー（チェーン）を構築するためのフレームワーク。複数のコンポーネントを連結し、入力処理・検索・生成を統合した複雑なパイプラインを実現する。

### サブ概念・手法
- **LLMChain**: プロンプトテンプレート + LLMを連結した基本的なチェーン
- **Retriever**: ベクトルストアからドキュメントを検索するコンポーネント
- **PromptTemplate**: ユーザー入力を構造化するためのテンプレート
- **チェーンの標準構成**: プロンプトテンプレート + Retriever + LLM
- **分類モデルとの組み合わせ**: ユーザーの意図を識別してRetrieval精度を向上させる
- **相互運用性**: マルチステップ推論ではステージ間でデータをシームレスに共有できることが重要
- **LangChainチェーンのコード構造例**:
  - Retriever（データベースクエリ用）+ LLM（応答処理用）+ PromptTemplate（クエリ生成用）

### 試験頻出ポイント
- **チェーンに含めるべきコンポーネント**: プロンプトテンプレート + Retriever + LLM（+ 必要に応じて分類モデル）
- **「固定応答のためのハードコードされたテンプレート」は不正解**: 多様なクエリへの柔軟性がない
- **「Retrieverなしのembedding」は不正解**: embeddingだけでは動的な検索・処理ができない
- **マルチステップ推論の設計原則**: ツールの相互運用性（意図識別→retrieval→応答生成のシームレスな連携）
- **LangChainでメールIDに基づくユーザー詳細抽出**: retriever + LLM + PromptTemplateの3つが必須

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「要約モデルを追加で使う」 | 要約は重要な詳細を削除し応答品質を低下させる |
| 「ルールベースシステムでRetrieval」 | 硬直的で動的なクエリに対応できない |
| 「ツール数を最小化する」 | 複雑なタスク対応には適切なツール数が必要 |

## 6. MLflow PyFuncモデル

### 定義・概要
MLflow PyFuncはPythonで書いた任意のモデルをMLflowのシリアライズ形式でラップし、前処理・後処理を含むパイプライン全体をカプセル化するインターフェース。デプロイ時に入出力の互換性と一貫性を確保する。

### サブ概念・手法
- **前処理（Preprocessing）**: モデル入力前にデータをクリーニング・標準化・変換
- **後処理（Postprocessing）**: モデル出力を人間が読めるフォーマットに変換（例：スコア→「承認」「拒否」）
- **predict関数**: PyFuncモデルのコアとなるメソッド。前処理・推論・後処理を内包
- **MLflowでの実験追跡**: パラメータ・メトリクス・モデルアーティファクトをトラッキング
- **Unity Catalogへの登録**: モデルガバナンス・バージョン管理・アクセス制御のため

### 試験頻出ポイント
- **PyFuncが有用な理由**: 推論の前後にデータを変換でき、互換性と最適化された出力を確保する（APIコスト最小化・再トレーニング削減ではない）
- **「後処理なしに前処理のみ」は不正解**: 後処理はモデル出力を解釈可能にするために不可欠
- **「生の入力を直接モデルに渡す」は不正解**: 前処理でデータ品質を確保し予測信頼性を向上させる
- **正しいワークフロー**: 入力前処理 → MLflow PyFunc予測 → 出力後処理（ユーザーフレンドリーなフォーマット）

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「PyFuncはAPIコスト最小化のためにバッチ処理する」 | APIコスト最適化は前処理・後処理ロジックとは無関係 |
| 「PyFuncは再トレーニングの必要性を減らす」 | 前後処理パイプラインはモデル改善の再トレーニングに代わらない |
| 「後処理をスキップして生のスコアを返す」 | 生スコアはユーザーを混乱させアクション可能なインサイトが不足 |

## 7. ガードレール・セーフティフィルター

### 定義・概要
ガードレールとはLLMアプリケーションの出力が安全で正確であり、アプリケーション要件に沿っていることを確保するための制御機構。ハルシネーション・有害応答・プライバシー侵害・不正確なトラブルシューティング手順などのリスクを軽減する。

### サブ概念・手法
- **metapromptによるガードレール**: 機密情報を出力に含めないよう指示するルールをシステムプロンプトで定義
  - 機密漏洩防止: 「特定の種類の機密情報を含めないように指示」
  - 有害コンテンツ防止: 「有害または危険なフレーズをブロックするためのコンテキストフィルタリング」
  - トーン・ブランディング制御: 「希望するトーンとスタイルを明示的に定義」
- **事前定義フィルター（Content Filtering）**: 安全でない・有害な応答を除外するフィルター
- **入力サニタイズ**: 特殊文字を削除しデータ形式を検証することでインジェクション攻撃を防止
- **retrievalベースの拡張**: LLMに不正確な情報を生成させないために、検索した情報で応答を根拠づける
- **応答フィルタリング**: prompt ベースのフィルター + 後処理フィルター
- **データプライバシー**: モデルがユーザー入力を保存しないことを確保する

### 試験頻出ポイント
- **ガードレールの主な目的**: latency削減・多様性向上ではなく、安全・正確・アプリ要件整合の確保
- **インジェクション攻撃防止**: 入力サニタイズ（特殊文字削除 + データ形式検証）
- **医療チャットボットの機密保護**: metapromptで機密情報を出力しないよう指示 + ユーザー入力を保存しないモデル設計
- **不正確なトラブルシューティング手順防止**: temperatureやトレーニングではなく、検証ステップ + retrievalベース拡張
- **ソーシャルメディアモデレーション**: metapromptでコンテキストフィルタリングを強調
- **企業ブランディングに沿った応答**: metapromptによるトーン定義 + 会社ドキュメントでのファインチューニング

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「temperatureを下げればランダム性が減り安全になる」 | temperatureはランダム性に影響するが、本質的な精度や安全性を向上させない |
| 「応答の長さを短くすれば有害コンテンツが減る」 | 長さと有害性は独立した問題 |
| 「監視・ログだけで十分」 | 監視は悪意ある入力が実行されるのを防がない（サニタイズが必要） |
| 「ガードレールはlatencyを削減する」 | ガードレールはパフォーマンス最適化ではなく安全性に焦点 |

## 8. コンテキスト長とチャンキング戦略

### 定義・概要
LLMやEmbeddingモデルにはトークン制限（Context Length）があるため、長いドキュメントは適切なサイズに分割（チャンキング）してから処理する必要がある。チャンキング戦略の選択は検索精度・要約品質に直接影響する。

### サブ概念・手法
- **セクションベースチャンキング**: ドキュメントの論理的なセクション（「イントロダクション」「調査結果」「条項」「付属文書」）に基づいて分割
  - 長い政府レポート・学術論文・法的契約書に有効
  - 任意の均等分割よりも文書構造を保持
- **均等サイズチャンキング**: 一定のトークン数（例：1000token）で機械的に分割
  - 論理的なセクションを壊すリスクがある
- **オーバーラップチャンキング**: チャンク間に重複を持たせてコンテキストを補完
  - 論理的なセクション境界とは整合しない
- **チャンクとEmbeddingモデルの一致**: チャンクサイズ（例：512token）とEmbeddingモデルのContext Length（512）を合わせる
- **長い入力への対応**: 新しいモデルのトレーニング（コスト大）や要約（情報欠損）より、チャンク分割が現実的な解決策

### チャンキング戦略の適用例
| ドキュメント種別 | 推奨戦略 |
|---|---|
| 10,000語超の政府レポート | セクション（イントロ・調査結果等）に基づくチャンキング |
| 法的契約書 | 論理的セクション（条項・付属文書）に基づくチャンキング |
| 512token制限のEmbeddingモデル | 512token以内のチャンクに分割して各チャンクを独立処理 |

### 試験頻出ポイント
- **「ドキュメント全体を1チャンクとして処理」は不正解**: トークン上限を超える
- **「均等サイズに分割」は一般に不正解**: 意味のあるセクションを壊すリスク
- **「より長いContext Lengthのモデルを新しくトレーニング」は不正解**: コスト大で実現性が低い
- **「要約モデルで長さを短縮」は不正解**: 重要な詳細が欠損する
- **「オーバーラップは常に最善」ではない**: 論理的なセクション境界との整合が優先される場合がある

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「全ドキュメントを1チャンク」 | トークン上限を超え処理できない |
| 「均等サイズで分割」 | 論理的なセクションを壊し要約品質が低下 |
| 「長いEmbeddingモデルを新規トレーニング」 | コストが大きく実現性が低い |
| 「要約して長さを短縮」 | 重要な詳細が欠損する |

## 9. エージェントシステムとツール設計

### 定義・概要
エージェントシステムは、複数のツール・データソース・操作を組み合わせて複雑なタスクを自律的に処理するLLMベースのシステム。注文状況・配送・返金などの複数操作が必要なカスタマーサポートに適している。

### サブ概念・手法
- **ツール定義**: 利用可能な機能（create_record, update_record, delete_record など）を明示的に定義し、使用例も提供
- **プロンプトテンプレートによる機能公開**: エージェントの機能と使用例を指定したテンプレートでモデルが正しくツールを呼び出せるよう誘導
- **マルチターン対応**: 複数回のやり取りをまたいでコンテキストを維持
- **複数操作の対応**: 単一の会話で複数の異なる操作（注文状況確認・返金処理など）を処理
- **フォールバックメカニズム**: エージェントが処理できないクエリは人間にエスカレーション

### 試験頻出ポイント
- **エージェント機能公開の最善策**: embeddingやルールベースシステムではなく、機能と使用例を明示したプロンプトテンプレート
- **「ルールベースシステムをトレーニング」は不正解**: 動的なマルチエージェントタスクへの柔軟性が不足
- **「要約モデルでクエリを凝縮」は不正解**: エージェント機能の公開には対応しない
- **「embeddingを適用」は不正解**: 検索は支援するが機能の公開は処理しない

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「embeddingでセマンティック表現」 | Embeddingは検索補助でありエージェント機能の公開ではない |
| 「ルールベースシステム」 | 硬直的でマルチエージェントタスクの柔軟性が不足 |
| 「要約モデルでクエリを凝縮」 | 機能公開とは無関係 |

## 10. LLMモデル選択基準

### 定義・概要
ユースケースに応じて適切なLLMを選択することが重要。コンテキストウィンドウサイズ・ドメイン特化度・多言語対応・コスト・latencyが主要な選択基準となる。

### モデル特性と適切なユースケース
| モデル/特性 | 適したユースケース |
|---|---|
| MPT-30B（大コンテキストウィンドウ） | 長いドキュメントの処理・大量のコンテキストが必要なタスク |
| Llama2-70B（高品質・機密処理可） | 機密データを扱うタスク・高品質な出力が必要なケース |
| CodeLlama-34B | コード生成・コードのデバッグ |
| ファインチューニング済みモデル | ドメイン固有タスク（医療・法的・財務） |
| 小型ファインチューニング済みモデル + キャッシュ | コスト効率と低latencyのバランス（商品推薦など） |

### モデル選択の優先基準（ユースケース別）
| ユースケース | 優先基準 |
|---|---|
| リアルタイム言語翻訳 | 低latency + 高翻訳精度 |
| eコマース多言語チャットボット | 多言語パフォーマンスベンチマーク |
| 財務アドバイス生成 | トレーニングドメイン・ファインチューニングのメタデータ + 財務データセット精度 |
| 学術論文要約 | 学術テキストでファインチューニングされた要約LLM |
| FAQシステム | モデルサイズ・アーキテクチャ + latency指標 + perplexity |

### 試験頻出ポイント
- **「モデルサイズが大きいほど良い」は不正解**: latencyが増加しコスト効率も低下
- **「多言語トレーニングデータのみ」では不十分**: リアルタイムパフォーマンスや精度は保証されない
- **「高いトークン制限」は翻訳の主要基準ではない**: 短い翻訳タスクには無関係
- **財務アドバイス**: 流暢さ（低perplexity）やlatencyよりもドメイン精度とコンプライアンスが優先
- **コスト効率**: 最大モデルではなく小型ファインチューニング済みモデル + クエリキャッシュ

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「大きなモデルサイズを使う」 | latencyが増加しコスト効率が低下 |
| 「流暢さのために低perplexityを優先」（財務アドバイス） | ドメイン精度とコンプライアンスが優先 |
| 「パラメータ数が多いモデル」 | パラメータ数だけではドメイン関連性・精度が保証されない |

## 11. 評価指標（ROUGE・BLEU・Precision/Recall/NDCG）

### 定義・概要
LLMアプリケーションの品質評価には用途に応じた指標の選択が重要。大きく「要約品質」「翻訳品質」「検索品質」「運用パフォーマンス」の4カテゴリに分類できる。

### 指標の分類と用途
| 指標 | 用途 | 特徴 |
|---|---|---|
| ROUGE | テキスト要約評価 | 生成された要約と参照要約のコンテンツ重複を測定 |
| BLEU | 機械翻訳評価 | 出力テキストと参照テキストの類似度を測定（要約には不適） |
| Precision | 検索精度 | 取得されたドキュメントのうち関連するものの割合 |
| Recall | 検索網羅性 | 全関連ドキュメントのうち取得できたものの割合 |
| NDCG | 検索ランキング品質 | 取得ドキュメントの関連性とランキング順序を評価 |
| Perplexity | 言語モデルの流暢さ | 確信度を測定（要約品質は測定しない） |
| F1スコア | 分類タスク | 要約には直接適用できない |
| Latency/Throughput/エラーレート | 運用パフォーマンス監視 | 本番環境での応答性・処理能力・信頼性 |

### Precision/RecallのRAGシステムへの適用
- **高Recall・低Precision**: 多くの関連ドキュメントを取得するが無関係なものも多い → フィルタリングやEmbedding改善が必要
- **低Recall・高Precision**: 取得したものは正確だが関連ドキュメントを見落とす → 検索範囲を広げる必要

### 試験頻出ポイント
- **要約品質の評価**: ROUGE（+ 人間による定性評価）。BLEUは翻訳向けで要約には不適
- **法的ドキュメント要約の評価**: ROUGE + Recall（コンテンツ関連性とカバレッジ）
- **RAGシステムの評価**: Precision/Recall + NDCG
- **MLflowでのRAG評価**: Precision/Recall（検索関連性）+ BLEU/ROUGE（応答精度）
- **本番LLM監視**: Latency・Throughput・エラーレート（BLEUやF1スコアは運用監視指標ではない）
- **「Perplexityで要約品質を評価」は不正解**: Perplexityは流暢さを評価するが情報量は測定しない

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「要約評価にBLEUを使う」 | BLEUは翻訳向け。要約にはROUGEが適切 |
| 「要約評価にF1を使う」 | F1は分類タスク用 |
| 「検索評価にBLEUを使う」 | BLEUは生成テキスト向けでRetrieval評価には不適 |
| 「本番監視にBLEU/perplexityを使う」 | これらは言語品質指標であり、運用監視にはlatency/throughput/エラーレート |
| 「Perplexityで要約の情報量を測定」 | Perplexityは確信度・流暢さを測定するが情報量は測定しない |

## 12. コスト・Latency最適化

### 定義・概要
LLMアプリケーションを本番環境で運用する際、コスト効率と低latencyを両立させることが重要。モデルサイズ・インフラ・キャッシュ・スケーリング戦略が主な最適化手段。

### サブ概念・手法
- **小型ファインチューニング済みモデル**: 品質を維持しながらコストとlatencyを削減
- **クエリキャッシュ**: 頻繁なクエリをキャッシュすることで繰り返しクエリのパフォーマンス向上
- **水平スケーリング + Autoscaling**: 変動するワークロードに対応し、過少/過剰リソースを防ぐ
- **GPU + 分散クラウドインフラ**: 高throughputのリアルタイム要約・Embeddingには GPU アクセラレーション + ストレージ最適化の分散クラウドインフラ
- **Context Length の最適化**: チャンクサイズに合わせたContext Length設定（512tokenチャンクには512 Context Length）
- **APIレート制限管理**: 十分なリソース + レート制限管理 + latency監視の三点セット
- **Inference Logging（Databricks Inference Tables）**: 本番エンドポイントのリクエスト・レスポンスをネイティブにログ記録

### 試験頻出ポイント
- **コスト・Latency優先（512tokenチャンク）**: Context Length 512・モデルサイズ 0.13GB・次元384 の最小構成
- **スケーラビリティ最適化**: Autoscalingポリシーによる水平スケーリング（固定リソースやレート制限だけでは不十分）
- **「ローカルストレージ」は不正解**: 本番システムのスケーラビリティとlatencyを損なう
- **「シングルノードサーバー」は不正解**: 高需要アプリには不十分
- **「CPUのみのローカルサーバー」は不正解**: 高throughput要件を効果的に処理できない
- **Inference Tables**: マイクロサービスの代替としてDatabricksネイティブでリクエスト・レスポンスをログ記録

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「最大モデルを使う」 | コストが高くlatencyが増大する |
| 「固定リソースでピーク対応」 | 変動ワークロード時に過少利用または過負荷 |
| 「レート制限だけで対応」 | 使用量を制御するだけでスケーラビリティは向上しない |
| 「context lengthを短縮してリソース節約」 | 応答品質が低下する可能性がある |
| 「複数モデルを並行デプロイ」 | コストと複雑さが増加するだけ |

## 13. ファインチューニングの適用場面

### 定義・概要
ファインチューニングは事前学習済みモデルを特定ドメインのデータでさらに学習させる手法。汎用モデルではカバーしきれないドメイン固有の知識・スタイル・トーンをモデルに習得させるために使用する。

### ファインチューニングが適切な場面
- ドメイン固有の用語・知識が必要（法的・医療・財務テキスト）
- 特定のトーン・スタイルを維持したい（丁寧なカスタマーサービス）
- 学術論文の要約など特定タスクへの最適化
- 企業のブランディングや通信ガイドラインへの適合

### ファインチューニングが適切でない（他の手法が優れている）場面
| 問題 | 適切な解決策 | なぜファインチューニングでは不十分か |
|---|---|---|
| Retrievalの精度が低い | ドメイン固有Embeddingモデル・ベクトルDB改善 | ファインチューニングは生成を改善するがretrieval自体には対処しない |
| 回答が曖昧で引用不足 | パイプラインに引用ステップを追加 | ファインチューニングは引用問題を直接解決しない |
| スケーラビリティの確保 | Autoscalingによる水平スケーリング | ファインチューニングはthroughput向上には直接対処しない |
| 不正確なトラブルシューティング手順 | Retrievalベース拡張 + 検証ステップ | ファインチューニングは不正確さのリスクに対処しない |

### 試験頻出ポイント
- **丁寧なトーン + ドメイン知識**: 丁寧なトーンのデータセット + 返品ポリシーでのファインチューニング
- **学術要約**: 学術テキストでファインチューニングされた要約LLM（汎用モデルはドメイン固有インサイトが不足）
- **「temperatureを上げて多様な要約」は不正解**: ランダム性が生まれ精度が低下
- **法的テキスト処理**: 法的データセットでのファインチューニング（ただしretrieval精度はVector DBで対応）

### 誤答パターン
| 誤答の傾向 | 正解の理由 |
|---|---|
| 「retrieval精度低下にファインチューニング」 | ファインチューニングは生成改善でretrieval問題は別 |
| 「スケーラビリティにファインチューニング」 | ファインチューニングはthroughput向上には対処しない |
| 「汎用LLMで学術要約」 | 汎用モデルはドメイン固有インサイトが欠けた要約を生成する可能性がある |
| 「temperatureを上げて多様な要約」 | ランダム性が増加し要約精度が低下する |

## 横断的な頻出判断軸のまとめ

以下は複数のトピックにまたがる重要な判断軸：

### 「ファインチューニング vs プロンプトエンジニアリング vs RAG」の使い分け
- **プロンプトエンジニアリング**: 出力フォーマット・トーン・指示が明確な場合。迅速で低コスト
- **RAG**: 最新情報・ドメイン固有知識・ハルシネーション防止が必要な場合
- **ファインチューニング**: ドメインスタイル・用語・トーンの深い適合が必要な場合

### 「retrieverが必要か・不要か」の判断
- **必要**: 動的情報取得・最新ドキュメント参照・ハルシネーション防止
- **不要**: 静的な応答・単純なテキスト変換・分類タスク

### 「評価指標の選択」早見表
- 要約品質 → **ROUGE**
- 翻訳品質 → **BLEU**
- 検索網羅性 → **Recall**
- 検索精度 → **Precision**
- 検索ランキング → **NDCG**
- 本番監視 → **Latency / Throughput / エラーレート**
- 定性評価 → **人間による評価（関連性・情報量）**

### よく出る「不正解の選択肢」パターン
1. **「Latencyだけを測定する」** → Latencyは品質を直接示さない
2. **「Tokenカウントで要約の長さを測定する」** → 精度や関連性は測定しない
3. **「ルールベースシステムを使う」** → 動的クエリへの柔軟性が不足
4. **「temperatureを調整する」** → ランダム性の変更であり、コンテキスト保持・トーン・品質には影響しない（例外：創造性が目的の場合）
5. **「最大モデル/最大パラメータを選ぶ」** → コスト高・latency増大・ドメイン精度を保証しない
