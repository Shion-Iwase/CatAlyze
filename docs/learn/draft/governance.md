### Guardrail（ガードレール）の種類と選択

- **定義・概要：**
  LLMアプリケーションの出力や動作を制約・監視する仕組み。不適切・有害・ポリシー違反の出力をブロック・修正するために使われる。用途に応じて複数の種類がある。

- **サブ概念：**
  - **Safety Guardrail（安全ガードレール）：** 不適切または望ましくない出力をブロック・置換する。政治的発言のフィルタリング、有害コンテンツの検出・拒否など。「会社のポリシーに違反する出力を定型メッセージで置き換える」のに使う。
  - **Compliance Guardrail（コンプライアンスガードレール）：** 法的・規制上の基準への準拠を強制する。Safety Guardrailとは目的が異なり、特定トピックのブロックではなく規制対応。
  - **Security Guardrail（セキュリティガードレール）：** 機密データやインフラの保護に焦点。出力フィルタリングではなくセキュリティ上の防御を担う。
  - **Contextual Guardrail（コンテキストガードレール）：** ユーザーのコンテキストに基づいてモデル出力を調整する。特定トピックのブロックではない。
  - **Metaprompt（メタプロンプト）：** 特定の動作（例：法的アドバイスを避ける）をモデルに指示するシステムプロンプト。Guardrailの実装手段の一つ。
  - **後処理フィルター（Post-processing Filter）：** 生成後の出力から機密情報を検出・削除する。前処理と組み合わせて多層防御を構成。
  - **入力バリデーション / 入力サニタイズ：** 悪意ある入力（インジェクション攻撃等）をブロックするために、特殊文字の除去・フォーマット検証を行う。
  - **Retrieval-based augmentation（検証ステップ）：** RAGベースで取得した正確な情報で生成を根拠づけるガードレール。不正確なトラブルシューティング手順の生成防止に有効。

- **試験頻出ポイント：**
  - **政治的発言やポリシー違反のブロック → Safety Guardrail** が正解。Compliance / Security / Contextualとの違いを明確に区別できること。
  - 「Safety GuardrailはCompliance Guardrailと同じ」と思いがち。Safety＝不適切出力のブロック、Compliance＝法令準拠の強制、Security＝インフラ保護、と用途が異なる。
  - 法的アドバイス制限チャットボットには **Metaprompt（動作制限の指示）＋Disclaimer（免責事項）** の2つが有効。トレーニングやtemperature調整では対応不可。
  - テクニカルサポートQAの不正確手順防止には **検証ステップ＋retrievalベース拡張のガードレール** が最善。temperatureやトレーニングは根本解決にならない。
  - インジェクション攻撃対策は **入力サニタイズ（特殊文字削除・フォーマット検証）** が正解。ログ監視や要約モデルは防御にならない。

- **誤答パターン：**
  - 「政治的発言をブロックするのはCompliance Guardrail」と思いがちだが、実際はSafety Guardrailが正しい。理由：Complianceは法的・規制準拠、Safetyは不適切出力のブロックに特化。
  - 「法的アドバイス制限にはtemperatureを下げる」と思いがちだが、temperatureはランダム性を減らすだけで、不正確なアドバイス生成の根本原因（知識の不確実性）には対処しない。
  - 「インジェクション攻撃への対策として入力をログ・監視する」と思いがちだが、監視は後手の対応であり、リアルタイムに攻撃を防げない。入力サニタイズで事前に無効化する必要がある。
  - 「敵対的入力対策はモデルをトレーニングする」と思いがちだが、すべての形式の悪意ある入力を学習で網羅することはできない。入力バリデーションルールの方が確実。

### ハルシネーション対策

- **定義・概要：**
  LLMが事実に基づかない内容（架空の条項・データなど）を生成する「ハルシネーション」を防ぐための技術的・設計的アプローチ。単一の手法では防げず、複合的な対策が必要。

- **サブ概念：**
  - **Metaprompt（メタプロンプト）：** 「提供されたテキストのみに基づいて生成せよ」という明示的な制約指示。最も直接的なハルシネーション防止手段。
  - **Strong System Prompt（強力なシステムプロンプト）：** モデルの動作を調整し、望ましい出力フォーマット・範囲をガイド。
  - **Guardrail（出力フィルター）：** 生成後の出力を検証・フィルタリング。ハルシネーションされた内容を後処理でブロック。
  - **Access Control（アクセス制御）：** ユーザーのアクセスレベルに基づき利用可能なデータを制限することで、機密データ漏洩リスクを低減。
  - **Fine-tuning（ファインチューニング）：** 自社データで再学習するが、単独ではハルシネーション防止の信頼性がない。補助手段にはなるが単独解決策としては不適切。

- **試験頻出ポイント：**
  - **ハルシネーション防止にファインチューニング単独では不十分。** ガードレール・アクセス制御・system promptの組み合わせが有効。
  - 法的契約書の要約でハルシネーション条項を生成する場合、**Metapromptで「提供テキストのみ」を指示する**のが正解。context length増加、事前トレーニング、分類モデルでの検証は根本解決にならない。
  - 「ファインチューニングで『何が適切か』を学習させれば機密データ漏洩を防げる」は誤り。

- **誤答パターン：**
  - 「context lengthを増やせばハルシネーションが減る」と思いがちだが、実際はハルシネーションを防げず計算コストが増えるだけ。理由：ハルシネーションはコンテキスト量の問題ではなく、生成制約がないことが原因。
  - 「法的データセットでトレーニングすれば正確になる」と思いがちだが、ファインチューニングはドメイン知識を向上させるだけで、架空内容の生成防止は保証されない。
  - 「分類モデルで要約を検証すればよい」と思いがちだが、分類モデルは要約のハルシネーション検出には特化しておらず、根本原因への直接対処にならない。

### PII・機密データ保護（マスキング・フィルタリング）

- **定義・概要：**
  個人を特定できる情報（PII：氏名・電話番号・クレジットカード番号・医療情報など）をアプリケーション処理から保護する技術。主に前処理段階のマスキング・後処理フィルタリングの2層で実装。

- **サブ概念：**
  - **データマスキング（Data Masking）：** 機密情報をプレースホルダーに置き換え、コンテキストを失わずにプライバシーを確保。分析的価値を維持しながらプライバシーリスクを低減。暗号化とは異なる（暗号化はストレージ保護、マスキングは匿名化）。
  - **Regexベースのマスキング：** 電話番号・クレジットカード情報などの構造化パターンを正規表現で検出してマスク。効率的で正確。
  - **入力フィルタリング + 応答バリデーション：** 処理前の入力フィルタリングと生成後の応答バリデーションを組み合わせてプライベートデータをブロック。医療チャットボット等に有効。
  - **後処理フィルター（Post-processing Filter）：** 生成済み出力から機密情報を検出して削除。共有前に確実に機密データを除去。
  - **Unity Catalog + データマスキング：** Databricksで開発環境から本番データへの安全なアクセスを提供するためのマスクビュー作成機能。

- **試験頻出ポイント：**
  - **機密データ漏洩防止の実装手段として、前処理（マスキング）と後処理（フィルター）の両方が登場する。** 状況に応じた使い分けが問われる。
  - 顧客サポートチャットボットの機密情報マスク → **regexベースのパターン実装**が最善。手動フィルタリング・要約モデル・トレーニング依存は不可。
  - 医療チャットボットのプライバシー保護 → **入力フィルタリング＋応答バリデーション**の組み合わせが正解。応答長制限・暗号化・匿名化データでのトレーニングは不十分。
  - 開発環境での安全な本番データアクセス → **Unity Catalogのアクセス制御＋データマスキングによるビュー作成**が正解。
  - **Maskingと暗号化の違い：** Maskingは匿名化（データの分析価値維持）、暗号化はストレージ中のデータ保護（データを読めなくする）。目的が異なる。

- **誤答パターン：**
  - 「Maskingはデータを暗号化してストレージ中に保護する」と思いがちだが、実際はMaskingは匿名化（コンテキスト維持）、暗号化はデータ秘匿（コンテキスト消去）。目的・効果が異なる。
  - 「チャットボットのトレーニングに頼れば機密データの漏洩を避けられる」と思いがちだが、LLMは明示的な前処理なしに機密データの除外を信頼性高く保証できない。
  - 「応答を暗号化すればプライバシーが保護される」と思いがちだが、暗号化は転送中保護であり、出力への機密データの包含自体を防がない。
  - 「すべての要約を手動レビューすればよい」と思いがちだが、大規模アプリケーションでは実用的でない。自動化された後処理フィルターが必要。

### データセット・ライセンスコンプライアンス

- **定義・概要：**
  LLMのトレーニングやRAGアプリケーションに使用するデータセットの法的利用可能性を確認するプロセス。商用利用の可否・プライバシー規制準拠（HIPAA等）・著作権の確認が主要要素。

- **サブ概念：**
  - **ライセンス条項の確認：** 商用利用・変更・再配布の許可・制限を規定する条項。オープンライセンス（CC、ApacheなどのOSSライセンス等）であっても条項を必ず確認する。
  - **HIPAA（医療情報の保護）：** 医療記録・患者データを扱う場合に必須のコンプライアンス確認。医療分野での生成AIには必須知識。
  - **データキュレーターへの事前連絡：** 第三者データを使用する場合、使用**開始前**に許可を確立するかライセンス条項を確認する義務。使用開始後の事後連絡では法的責任が発生しうる。
  - **誤情報ドキュメントの除外（前処理段階）：** 有害な誤情報を含むとフラグが立てられたドキュメントは、取り込み前のpreprocessingで除外する。モニタリングだけでは不十分。

- **試験頻出ポイント：**
  - **「公開データなら自由に使える」は誤り。** 公開≠商用利用許可。必ずライセンス確認が必要。
  - **「非商用利用はライセンス確認不要」も誤り。** 法的コンプライアンスは使用目的に関わらず必要。
  - 医療記録データセットの法的コンプライアンス確保には **ライセンス条件の確認（商用利用可否）＋HIPAA等プライバシー規制への準拠** の2点が正解。
  - 誤情報ドキュメントの対策は **前処理段階での除外** が正解。免責事項追加・モデルのトレーニング・推論中の監視は不十分。
  - **データ使用後のキュレーターへの連絡は「不適切な行動」として出題される。** 事前連絡・事前確認が原則。

- **誤答パターン：**
  - 「データセット内の機密情報の存在が主要な法的考慮事項」と思いがちだが、機密情報は重要だが**ライセンス条項が最優先の法的考慮事項**。理由：ライセンスが商用利用を禁じていれば、機密情報がなくても法的リスクがある。
  - 「フラグが立ったドキュメントも取り込み、出力に免責事項を追加すればよい」と思いがちだが、免責事項は誤情報がシステムに影響するリスクを軽減しない。取り込み前の除外が唯一の根本的解決策。
  - 「アプリケーション出力でソースを引用すれば法的リスクを避けられる」と思いがちだが、引用はライセンス確認の代替にならない。

### セキュリティ（入力バリデーション・サニタイズ・暗号化・認証）

- **定義・概要：**
  LLMアプリケーションをセキュリティ上の脅威（悪意ある入力・不正アクセス・データ漏洩）から保護するための技術的手段。入力段階・データ保護・アクセス認証の多層防御が基本。

- **サブ概念：**
  - **入力サニタイズ（Input Sanitization）：** 特殊文字の削除・データ形式の検証によりインジェクション攻撃（SQLインジェクション等）を防止。LLMアプリのプロンプトインジェクション対策としても有効。
  - **入力バリデーション（Input Validation）：** 許容できないパターン・フォーマットを検出するルール実装。敵対的ユーザー入力によるシステムクラッシュを防止。
  - **暗号化（保存中・転送中）：** 保存中（at rest）および転送中（in transit）のデータを暗号化し、機密情報を保護。プライバシー規制を満たすための基本要件。
  - **APIキー認証：** アクセス権を持つユーザーのみがシステムにアクセスできることを保証。カスタマーサポートアプリ等のコンプライアンスとセキュリティに不可欠。
  - **LLM Guardrail（機密データ漏洩防止）：** 会話エージェントが機密情報を出力しないように制御。暗号化と組み合わせて銀行アプリ等で使用。
  - **環境変数（Environment Variables）：** MLflowエンドポイントにシークレット・認証情報を安全に渡すための手段。プレーンテキストでの受け渡しは重大なセキュリティリスク。

- **試験頻出ポイント：**
  - **悪意ある入力対策はセーフティフィルター（検出・拒否）が最善。** インタラクション時間の短縮・コンピュート増加はセキュリティに無関係。
  - **プレーンテキストでのシークレット受け渡しは絶対NG。** 環境変数が正解。
  - 患者向け医療チャットボットのコンプライアンス要件：**保存・転送中の暗号化 ＋ APIキー認証**が必須。プライベートネットワーク依存・暗号化スキップは不可。
  - 銀行アプリの機密情報安全処理：**暗号化 ＋ LLM guardrail**の組み合わせが正解。temperatureや要約モデルはセキュリティと無関係。
  - 「プライベートネットワークアクセスだけでセキュリティは十分」は誤り。暗号化が別途必要。

- **誤答パターン：**
  - 「暗号化をスキップしてプライベートネットワークアクセスに依存すれば十分」と思いがちだが、プライベートネットワークだけではデータセキュリティもコンプライアンスも保証されない。暗号化は別途必要。
  - 「入力をログ・監視すれば敵対的入力に対応できる」と思いがちだが、監視は問題の検出に役立つが攻撃を**防ぎはしない**。入力バリデーションで事前ブロックが必要。
  - 「悪意ある入力を処理するようにモデルをトレーニングすれば解決できる」と思いがちだが、トレーニングだけではすべての形式の悪意ある入力攻撃を防ぐことはできない。
  - 「spark.conf.set()でシークレットを渡せる」と思いがちだが、これはSpark設定用であり、セキュアなシークレット受け渡しには不適切。

### アクセス制御（Unity Catalog / RBAC）

- **定義・概要：**
  データ・モデル・アプリケーション機能へのアクセスを適切に制限するための仕組み。Databricks環境ではUnity Catalogがデータガバナンスの中心的ツール。ロールベースのアクセス制御（RBAC）がLLMアプリでの標準手法。

- **サブ概念：**
  - **Unity Catalog：** Databricksのデータガバナンスツール。アクセス制御、データマスキング、メタデータ管理、モデル登録などを一元管理。セキュアビューの作成・マスキングルールのサポート。
  - **ロールベース権限（RBAC）：** ユーザーまたはチームのロールに応じてアプリケーション機能へのアクセスを制御。マルチユーザーLLMアプリケーションのアクセス管理に最適。
  - **開発専用マスクビュー：** Unity Catalogを使い、開発環境が本番データに安全かつセキュアにアクセスできる機密フィールドマスク済みビューを作成。
  - **Delta Sharing：** 外部パートナーとのデータ共有ツール。内部開発ワークフロー（開発環境と本番環境間）への利用は不適切。
  - **MLflow + Unity Catalog でのモデル登録：** ガバナンスと再現性向上のため、学習済みモデルをMLflowにログ→model signatureを追加→Unity Catalogに登録する手順が正しい。

- **試験頻出ポイント：**
  - **開発環境で本番データを安全に使うには Unity Catalog + データマスキングビュー**が正解。本番テーブルへの直接ポイント・合成データ・完全コピーは不可。
  - **マルチユーザーアプリの適切なアクセス制御 = ロールベース権限（RBAC）**。temperature設定・暗号化はアクセス権管理ではない。
  - Unity CatalogへのLLM登録手順：**MLflowにログ→model signatureを追加→Unity Catalogに登録**の順序が正しい。Unity Catalogで先にフォーマット定義→MLflow登録の順序は誤り。
  - Unity Catalog登録時に設定すべきメタデータ：**モデル名、バージョン、スキーマ、タグ**（スキーマ・タグなしの最小限登録は不十分）。
  - **Delta Sharingは外部パートナー向け**。内部開発ワークフローへの適用は誤り。

- **誤答パターン：**
  - 「Delta Sharingを使って開発環境から本番テーブルへの限定クエリアクセスを提供する」と思いがちだが、Delta Sharingは外部パートナー向けであり内部開発ワークフローには不適切。
  - 「暗号化を使用して不正なアクションを制限できる」と思いがちだが、暗号化は保存中・転送中のデータ保護であり、アクセス権限の管理はしない。RBACが必要。
  - 「Unity CatalogでモデルをFine-tuningする」と思いがちだが、Unity CatalogはモデルのトレーニングやFine-tuningをサポートしていない。

### 医療・法律チャットボットの安全設計

- **定義・概要：**
  医療や法律など高リスク分野でのLLMアプリケーション設計では、通常のチャットボット設計に加えて、緊急時対応・不正確アドバイス防止・コンプライアンスの特別な考慮が必要。

- **サブ概念：**
  - **緊急時エスカレーション（Medical Emergency Escalation）：** 激しい頭痛・めまい等の潜在的な医療緊急事態の症状が示された場合、情報収集・記事提案より**緊急サービスへの誘導を最優先**する。
  - **法的アドバイス制限（Legal Advice Restriction）：** チャットボットが法的アドバイスを提供しないよう、Metapromptで「一般情報のみに回答を制限する」指示と、すべての出力への**免責事項（Disclaimer）**の追加が有効な組み合わせ。
  - **RAGによる根拠付け（Retrieval-Augmented Validation）：** 検索ベースの拡張により、トラブルシューティング手順の正確性を担保するガードレール。

- **試験頻出ポイント：**
  - 医療chatbotで激しい頭痛・めまい等の症状が2日以上続く場合、**緊急サービスへの誘導（「地元の緊急サービスに電話してください」）が最適な応答**。詳細情報の収集・記事提案は緊急時には不適切。
  - 法的アドバイス制限chatbotには **Metaprompt（一般情報のみ）＋ Disclaimer（免責事項）** の2つが有効。法的データセットでのトレーニング・temperature低下・法律用語クエリのブロックは誤り。
  - **法律クエリをすべてブロックする**のは誤り。chatbotの機能を制限しすぎて使いやすさが低下する。

- **誤答パターン：**
  - 「2日続く激しい頭痛には詳細情報（年齢・活動・症状）を収集すべき」と思いがちだが、このような症状は緊急事態の可能性があるため、情報収集より緊急サービスへの誘導が優先。
  - 「法的アドバイス制限にはtemperatureを下げる」と思いがちだが、temperatureの調整はランダム性を減らすだけで不正確なアドバイスの核心的問題（LLMが架空の法的情報を生成する）には対応しない。
  - 「法律用語を含むクエリをすべてブロックする」と思いがちだが、これはchatbotの有用性を著しく低下させ、一般情報の提供まで妨げてしまう。

### データキュレーション（取り込み前フィルタリング）

- **定義・概要：**
  RAGアプリケーションやLLMトレーニングに使用するデータをパイプラインに取り込む前に、有害・不適切・誤ったコンテンツを除外・浄化するプロセス。後処理より前処理での対応が優先される。

- **サブ概念：**
  - **不適切言語の取り込み前フィルタリング：** 問題のあるテキスト（不適切な言語・炎上コンテンツ）を取り込み前に検出・除去するフィルタリングツール。プレースホルダーへの置換ではなく完全な除去が推奨。
  - **誤情報ドキュメントの前処理除外：** 有害な誤情報を含むとフラグが立てられたサードパーティドキュメントをpreprocessingで除外。出力モニタリングや免責事項追加では不十分。
  - **アップストリームデータキュレーション：** RAGシステムの炎上的出力に対処するため、取り込み前の手動レビューによるアップストリームデータのキュレーションが最も効果的。
  - **バイアスのあるエントリの削除・再重み付け：** 採用推薦等の公平性が必要なユースケースでは、データセットのバイアスを分析して問題のあるエントリを削除または再重み付け。

- **試験頻出ポイント：**
  - **不適切コンテンツ対策は「後処理監視」ではなく「前処理フィルタリング」が正解。** 取り込んだ後にモニタリングしても、すでにシステムに影響している。
  - RAGの炎上的出力対策は **取り込み前の手動レビューによるキュレーション** が最善。アクセス制限・ユーザー通知・更新頻度増加は無効。
  - **バイアス対策はデータセットサイズ削減ではなく、バイアスエントリの分析・削除・再重み付け。**
  - 「プレースホルダーへの置換」ではなく「完全除去」が推奨される。プレースホルダーでは問題のあるコンテキストが残る可能性がある。

- **誤答パターン：**
  - 「すべてのドキュメントを取り込み、出力モニタリングに頼る」と思いがちだが、取り込み後のモニタリングでは問題のあるコンテンツがシステムに影響するのを防げない。
  - 「フラグが立ったドキュメントも使用して、LLMがそれらを無視するようトレーニングする」と思いがちだが、LLMは前処理なしに有害コンテンツを確実に除外できない。
  - 「バイアス対策にデータセットサイズを削減する」と思いがちだが、サイズ削減はバイアスの除去を保証しない。バイアスのある具体的なエントリを特定して対処する必要がある。

### ヘルプデスク・サポートシステムのデータソース設計

- **定義・概要：**
  ヘルプデスクやカスタマーサポートチャットボットのために、チケットの根本原因と解決策を効率的に特定できるデータソースを選択・設計するプロセス。構造化データと非構造化テキストの両方を活用する。

- **サブ概念：**
  - **call_detail テーブル：** root_causeとresolutionフィールドを持つ全通話詳細スナップショット。チケットの根本原因・解決策の主要データソース。
  - **transcript Volume：** 録音のwavファイルとテキスト文字起こしを含むUnity Catalog Volume。詳細な通話コンテキストを提供。
  - **call_rep_history：** 担当者のパフォーマンス指標を追跡。チケット解決コンテキストは提供しない。
  - **call_cust_history：** 顧客の使用状況を追跡。特定チケットの根本原因・解決策特定には不向き。
  - **maintenance_schedule：** 障害・計画メンテナンスを追跡。特定チケットへの直接対応には役立たない。

- **試験頻出ポイント：**
  - チケット根本原因・解決策特定に最適なデータソースは **call_detail（root_cause/resolutionフィールド）＋ transcript Volume（通話コンテキスト）** の2つ。
  - 担当者パフォーマンス指標（call_rep_history）は解決コンテキストを提供しないため不適切。

- **誤答パターン：**
  - 「call_cust_historyが根本原因特定に役立つ」と思いがちだが、顧客使用状況の追跡はサービス使用の把握であり、特定チケットの原因解決には直接役立たない。

### バイアス対策と公平性確保

- **定義・概要：**
  採用推薦・融資審査など、人の生活に影響する意思決定システムでのLLM出力において、不公平・差別的な結果を生まないためのデータ管理アプローチ。

- **サブ概念：**
  - **バイアス分析と除去・再重み付け：** データセットの潜在的バイアスを分析し、バイアスのあるエントリを削除または再重み付けして公平性を確保。
  - **事前学習済みモデルのバイアス継承：** 事前学習済みモデルもその学習データのバイアスを反映するリスクがある。Fine-tuningで悪化する可能性も。

- **試験頻出ポイント：**
  - バイアス対策は**データセットの分析と問題エントリの削除・再重み付け**が正解。モデルへの依存・データサイズ削減・バイアス無視は誤り。
  - 採用推薦システムのバイアスを「モデルがクエリで自動補正する」と考えるのは誤り。元データのバイアスがそのまま出力に反映される。

- **誤答パターン：**
  - 「事前学習済みモデルのみを使えばデータのバイアスを回避できる」と思いがちだが、事前学習済みモデルも元の学習データのバイアスを引き継ぐ。
  - 「データセットサイズを削減すればバイアスが減る」と思いがちだが、サイズ削減はバイアスの除去を保証しない。問題となる特定エントリを識別・対処することが必要。
