## 1. MLflow × Unity Catalog × Model Serving — モデルデプロイの基本フロー

### 定義・概要

MLflow でモデルをログし、Unity Catalog に登録した後、Databricks Model Serving でエンドポイントを起動するのが Databricks における標準的なモデルデプロイフローである。各ステップが依存関係を持つため、順序が試験で頻繁に問われる。

### サブ概念・手法

- **MLflow log**: `mlflow.log_model()` でモデル成果物・パラメータ・メトリクスを記録。シグネチャ（入出力スキーマ）も同時に登録するのが推奨。
- **Unity Catalog への登録**: `mlflow.register_model()` で Unity Catalog のモデルレジストリに登録。メタデータとして **モデル名・バージョン・スキーマ・タグ・説明** を付与する。
- **Model Serving エンドポイント**: Unity Catalog から登録済みモデルを選択し、サービングエンドポイントを作成。
- **モデルシグネチャ**: エンドポイントデプロイ時にシグネチャを登録しておくことで、入力フォーマットの整合性を自動検証できる。
- **デプロイの正しい順序**:
  1. MLflow でモデルをログ（シグネチャ付き）
  2. Unity Catalog に登録（名前・バージョン・説明・タグ）
  3. Model Serving エンドポイントを起動

### 試験頻出ポイント

- **順序問題**: 「MLflow log → Unity Catalog → Serving」の3ステップが最重要。LLM をデプロイしてから embedding を設定する順序は誤り。
- **シグネチャの目的**: latency 削減でも logging 有効化でもなく「**正しいフォーマットで入力を受け取り出力を返すことの確保**」。
- **Unity Catalog の利点**: ガバナンス強化・バージョン管理・アクセス制御の3点セットで覚える。
- **RAG デプロイの最初のステップ**: ドキュメントの**ベクトル embedding 作成**が最初（エンドポイント設定や Unity Catalog 登録より先）。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「LLM をデプロイしてから embedding を設定する」 | embedding とretriever の準備が LLM デプロイより先に必要 |
| 「シグネチャ登録はlatency 削減のため」 | シグネチャは入出力フォーマット整合性の確保が目的 |
| 「Unity Catalog 登録はリアルタイムロギングを有効化する」 | ロギングはシグネチャ・Inference Tables の役割 |
| 「RAG のデプロイ最初のステップはエンドポイント設定」 | embedding 作成が最初。検索の基盤となるため |

## 2. MLflow PyFunc — カスタム前処理・後処理

### 定義・概要

MLflow PyFunc は Python ベースのカスタムモデルクラスを MLflow に統合する仕組みで、`predict()` 関数の中に前処理・推論・後処理を一体化してカプセル化できる。これにより、デプロイ後も一貫したデータ変換が保証される。

### サブ概念・手法

- **前処理（preprocessing）**: 入力データのクリーニング・正規化・標準化を推論前に実行する。
- **後処理（postprocessing）**: モデルの生スコア・出力をユーザーフレンドリーなフォーマット（例: 「承認」「拒否」、ラベル文字列）に変換する。
- **`predict()` 関数**: 前処理 → モデル呼び出し → 後処理の3ステップを1つの関数で実装する。
- **用途例**:
  - センチメント分析: テキストクリーニング → モデル呼び出し → ユーザーフレンドリーなラベル出力
  - ローン審査: データクリーニング → 予測 → 「承認/拒否」フォーマット出力
  - 汎用: 前処理で互換性確保、後処理でJSON 形式など最適化された出力

### 試験頻出ポイント

- **PyFunc の有用性**: 「APIコスト最小化」でも「再トレーニング削減」でもなく、**「モデル推論の前後にデータを変換して互換性と最適化された出力を確保する」**。
- **両方向の変換が必要**: 前処理だけ、後処理だけでは不完全。入力変換と出力フォーマット化の両方が必要。
- **後処理の重要性**: 生スコアをそのまま返すことはNG。ユーザーを混乱させ、アクション可能なインサイトが欠如するため。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「後処理なしに前処理のみ使用する」 | 後処理はモデル出力を解釈可能にするために不可欠 |
| 「生の入力をモデルに直接渡す（高速化のため）」 | 前処理はデータ品質と予測信頼性を確保するために必須 |
| 「後処理をスキップして生スコアを返す」 | 生スコアはユーザー理解困難でアクション不可能 |
| 「PyFunc はハイパーパラメータチューニングを自動化する」 | チューニングは別プロセス。PyFunc は変換パイプラインの役割 |

## 3. Vector Search / Vector Store — 構築と運用

### 定義・概要

Vector Search はテキストや画像を高次元ベクトルに変換し、意味的類似度に基づく高速検索を実現するシステムである。Databricks では `VectorSearchClient` の `create_delta_sync_index()` が Databricks 管理の embedding 同期に使われる。外部利用には FAISS などが選択肢となる。

### サブ概念・手法

- **基本フロー**: ドキュメント前処理 → embedding 生成 → ベクトルストアへの保存 → retriever によるクエリ → 最近傍探索（nearest-neighbor search）
- **Databricks Vector Search**:
  - `create_endpoint()` でエンドポイント作成
  - `create_delta_sync_index()` で Delta テーブルと embedding を同期
  - Databricks が embedding の自動管理・更新を担う
- **FAISS**: 外部ベクトルストア。Databricks 管理外の環境で使用。
- **距離メトリクス**: nearest-neighbor search に距離メトリクス（コサイン類似度・ユークリッド距離など）を最適化することでクエリパフォーマンスが向上。
- **インフラ要件**: GPU アクセラレーション + 分散クラウドインフラ + 高性能ベクトルストア DB が低遅延・高スループットのために必要。
- **Databricks RAG でのデプロイ順序**:
  1. embedding モデルを準備
  2. ベクトルストア（Vector Search インデックス）に embedding を保存
  3. retriever を設定
  4. Model Serving でエンドポイントをデプロイ

### 試験頻出ポイント

- **`create_delta_sync_index()`**: Databricks 管理の embedding 同期の標準 API。`create_vector_store()` や `store_embeddings()` は存在しない（誤答選択肢として登場）。
- **検索精度改善**: 関連性の低いドキュメントが多い場合は「セマンティック類似度最適化 retriever + ドメイン固有 embedding モデル」が正解。チャンクサイズ増加やランダムサンプリングは誤り。
- **FAISS + GPU**: 外部環境での低遅延 Vector Search には FAISS + GPU + 分散 DB の組み合わせが最適。
- **RAG の正しい順序**: embedding 生成 → Vector Search インデックス保存 → retriever+LLM 統合 → エンドポイントデプロイ。retriever 構築を最初に行うのは誤り。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「`create_vector_store()` でインデックスを作成する」 | 正しくは `create_delta_sync_index()` |
| 「retriever を最初に構築する」 | embedding とインデックスの準備が retriever より先 |
| 「チャンクサイズを増やして検索精度を改善する」 | 大きなチャンクは無関係なコンテンツを含め関連性を希薄化する |
| 「ランダムサンプリングで多様な検索結果を生成する」 | ランダムサンプリングは無関係ドキュメント取得の可能性を高める |
| 「CPUのみのローカルサーバーで高スループット対応」 | CPU のみでは高スループット要件を処理できない。GPU + 分散クラウドが必要 |

## 4. LangChain — チェーン設計とコンポーネント構成

### 定義・概要

LangChain は retriever・LLM・プロンプトテンプレートなどのコンポーネントをチェーン（Chain）として結合し、複数ステップの処理パイプラインを構築するフレームワークである。RAG・会話チャットボット・QA システムの実装に広く使用される。

### サブ概念・手法

- **必須コンポーネント（基本チェーン）**:
  - `prompt=` プロンプトテンプレート（ユーザー入力の構造化）
  - `retriever=` データ取得コンポーネント（関連ドキュメントの検索）
  - `llm=` LLM（応答生成）
- **`LLMChain`**: `llm=` と `prompt=` の両方が必須パラメータ。
- **RAG チェーン**: retriever でドキュメントを取得し、プロンプトテンプレートにコンテキストとして注入してから LLM に渡す。
- **会話履歴対応チェーン**: retriever + 会話履歴を含むプロンプトテンプレート + LLM。ターン間のコンテキスト保持が鍵。
- **分類モデルの組み込み**: ユーザーの意図を識別する分類モデルを追加することで retrieval 精度が向上する。
- **トークン制限への対応**: チャンキングでドキュメントを分割し、retriever で関連チャンクのみ取得してから LLM に渡す。

### 試験頻出ポイント

- **最小構成**: retriever + LLM + プロンプトテンプレートの3点セット。要約モデルや分類モデルは代替にならない。
- **長いドキュメントへの対応**: 全文処理は NG。チャンキング → retriever → LLM の流れで処理する。
- **会話型 QA**: retriever のみでは NG。会話履歴を含むプロンプトテンプレートも必要。
- **医療 QA**: retriever（ガイドライン取得）+ LLM（説明生成）が正解。要約モデル・ルールベース・分類モデルは不正解。
- **財務レポート**: embedding モデル + retriever の組み合わせが財務報告書の RAG に最適。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「要約モデルだけで構成する」 | 要約は特定の質問に答えられない |
| 「ルールベースシステムでクエリをセクションにマッピング」 | ルールベースは動的クエリに対し硬直的すぎる |
| 「分類モデルだけで応答を生成する」 | 分類は詳細な応答や説明を提供しない |
| 「retriever なしの事前学習済み embedding だけ使う」 | embedding だけでは動的取得・処理ができない |
| 「ハードコードされたテンプレートで固定応答を返す」 | 多様なユーザークエリへの柔軟性がない |

## 5. RAG パイプライン — デプロイ手順と構成要素

### 定義・概要

RAG（Retrieval-Augmented Generation）は、外部知識ソースからドキュメントを検索し、その内容を LLM の生成に組み込む手法である。デプロイには決まった順序があり、embedding 生成・インデックス作成・retriever 設定・エンドポイントデプロイの順序が試験で繰り返し問われる。

### サブ概念・手法

- **完全デプロイフロー**:
  1. 依存ライブラリのインストール
  2. ドキュメントの前処理とチャンキング
  3. embedding モデルで embedding 生成
  4. Vector Search インデックスに保存
  5. retriever を設定（セマンティック類似度最適化）
  6. retriever と LLM を統合
  7. Model Serving でエンドポイントをデプロイ
- **文書取り込みと索引化が先**: embedding 生成とインデックス作成が完了していないとretriever は機能しない。
- **コンテキスト補完**: context-aware summarizer（コンテキスト対応要約器）を追加することで応答の具体性が向上。
- **ドメイン固有 embedding**: 汎用 embedding より精度が高く、法律・医療などの専門領域では必須。
- **前処理の重要性**: embedding に渡す前にドキュメントを適切にチャンキングし、無関係セクションをフィルタリングする。

### 試験頻出ポイント

- **デプロイ順序の確定パターン**: 「embedding 生成 → Vector Search インデックス保存 → retriever+LLM 統合 → エンドポイントデプロイ」。この順序の変則が誤答として提示される。
- **最初のステップ**: RAG デプロイの最初のステップは必ず **ベクトル embedding の作成**。エンドポイント設定や Unity Catalog 登録は後のステップ。
- **retriever のスキップは NG**: embedding モデルだけで retriever を省略するアーキテクチャは精密なドキュメントルックアップができない。
- **大規模 RAG に必要なインフラ**: コンピューティングパワー + ストレージ + 監視 + 高速ネットワークの4要素。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「LLM を先にデプロイし後で retriever と embedding を設定」 | embedding と retriever の準備が LLM デプロイより先に必要 |
| 「retriever を省略して embedding モデルのみ使用」 | retriever は精密なドキュメントルックアップに不可欠 |
| 「retriever のみ設定して LLM に直接接続」 | retriever だけではベクトル検索・embedding を処理できない |
| 「要約モデルをデプロイした後に embedding を保存」 | embedding とインデックス化が最初に必要 |
| 「分類モデルをトレーニングしてから retriever をデプロイ」 | 分類モデルは検索と生成に特化した RAG に不要 |

## 6. セキュリティ・アクセス制御 — エンドポイント保護

### 定義・概要

LLM アプリケーションのデプロイでは、APIキー認証・RBAC（ロールベースアクセス制御）・保存中と転送中の暗号化の3層が標準的なセキュリティ構成である。認証情報は環境変数で渡すのが原則で、コードへの平文埋め込みは厳禁。

### サブ概念・手法

- **RBAC（Role-Based Access Control）**: ユーザーロールに基づいて権限を割り当て、特定機能へのアクセスを制御。マルチユーザー・マルチチームのアプリケーションで必須。
- **暗号化**: 保存中（at rest）と転送中（in transit）の両方でデータを暗号化することで機密データへの不正アクセスを防止。
- **API キー認証**: エンドポイントへのアクセス認証に使用。すべてのユーザーへの共有アクセスキーは NG。
- **環境変数でのシークレット管理**: API キーや接続文字列はコードに平文で書かず、環境変数として渡す。
- **コンプライアンス要件への対応**: 暗号化（at rest + in transit）+ API キー認証の組み合わせが規制準拠の標準パターン。
- **Unity Catalog への登録 + 監査ログ**: エンタープライズ RAG のコンプライアンス確保には、モデルの Unity Catalog 登録と監査ログによるエンドポイント追跡が重要。

### 試験頻出ポイント

- **RBAC の適用場面**: マルチユーザーアクセス制御の問題では必ず RBAC が正解。「すべてのユーザーにアクセスを許可」「共有アクセスキーを使用」「ネットワークレベル制限のみ」はすべて誤り。
- **暗号化の必須要件**: 「保存中と転送中の両方」が正解。どちらか一方だけでは不完全。
- **temperature 設定は NG**: アクセス制御に temperature 調整は無関係。
- **パブリックベクトルストアは NG**: 機密データを含む RAG でパブリックベクトルストアを使うとデータセキュリティが損なわれる。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「共有アクセスキーで全ユーザー認証する」 | 個別のアクセス追跡・制御ができない |
| 「ネットワークレベル制限だけに頼る（認証なし）」 | 認証なしのネットワーク制限は堅牢なアクセス制御を提供しない |
| 「アクセス制御なしでデプロイ（開発簡素化のため）」 | マルチユーザーシステムではセキュリティとコンプライアンスが必須 |
| 「temperature 設定でユーザー固有クエリを制御」 | temperature はランダム性に影響するがアクセス制御には無関係 |
| 「暗号化だけで不正アクションを制限する」 | 暗号化はデータ保護だが、アクセス権限管理（RBAC）とは別物 |

## 7. スケーラビリティ・インフラ — 高トラフィック対応

### 定義・概要

LLM アプリケーションのトラフィックは変動的であるため、固定リソースではなく Autoscaling（自動スケーリング）による水平スケーリングが推奨される。リアルタイム用途ではリソース確保・レート制限管理・latency 監視の3点を常に考慮する必要がある。

### サブ概念・手法

- **Autoscaling（水平スケーリング）**: 変動するワークロードに応じてコンピューティングリソースを動的に調整。API 使用量が急増する場合の最適解。
- **固定リソースの問題点**: ピーク需要に合わせた固定リソースは低需要時に過少利用、高需要時にシステム過負荷を招く。
- **GPU アクセラレーション**: LLM 推論と embedding 処理には GPU が必要。CPU のみでは高スループット要件を満たせない。
- **分散クラウドインフラ**: 高スループット・低レイテンシの要件には分散クラウドインフラが必要。ローカルサーバーは NG。
- **リアルタイムサービングの3要件**: 十分なコンピューティングリソース確保 + API レート制限管理 + latency 監視。
- **レート制限（Rate Limiting）**: スケーリングなしのレート制限はスケーラビリティを向上させない。あくまで流量制御であり、スケールアップとは別物。

### 試験頻出ポイント

- **スケーラビリティの正解**: 変動ワークロードへの対応は「autoscaling ポリシーによる水平スケーリング」。レート制限のみ・固定リソース・context length 短縮はすべて誤り。
- **高スループットインフラの正解**: 「GPU + ストレージ最適化を備えた分散クラウドインフラ」。CPU ローカルや単一インスタンスは NG。
- **リアルタイム API サービングの正解**: コンピューティングリソース + レート制限管理 + latency 監視の3点セット。
- **context length 短縮は NG**: コスト節約のための context length 短縮は応答品質を低下させ、スケーラビリティに対処しない。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「インフラをスケーリングせずにレート制限を実装する」 | レート制限は流量制御。スケーラビリティは向上しない |
| 「コスト節約のために context length を短縮する」 | 応答品質が低下し、スケーラビリティにも対処しない |
| 「固定数のコンピューティングリソースを使用する」 | 変動ワークロード時に過少利用または過負荷 |
| 「CPU のみのローカルサーバーを使用する」 | 高スループット要件を処理できない |
| 「最小限のスケーリング機能を持つ単一インスタンス」 | リアルタイムの高スループット需要に不十分 |

## 8. モデル選択 — ユースケース別の最適解

### 定義・概要

タスクやコスト・品質要件に応じて最適なモデルを選択する判断軸が試験で問われる。コード生成・高品質オンプレミス・ドメイン特化・多言語対応など、各ユースケースに対応した推奨モデルまたはアプローチが存在する。

### サブ概念・手法

- **CodeLlama-34B**: コード生成タスクに最適化されたモデル。
- **Llama2-70B**: 高品質な応答が必要でかつサードパーティを使用できない（オンプレミス）シナリオに最適。
- **ドメイン特化小型 LLM**: コスト効率が優先される場合、大型汎用モデルより特定ドメインにファインチューニングされた小型モデルが有利。
- **多言語パフォーマンスベンチマーク**: グローバルなユーザーを対象とする多言語チャットボットでは、モデルサイズや latency よりも多言語ベンチマークスコアを優先。
- **Sentence Transformers**: センテンスレベルの意味関係を捉えるのに最適な embedding モデル。
- **ドメイン固有 embedding モデル**: 検索精度（retrieval precision）を最大化するためにドメイン固有データでトレーニングされた embedding モデルを使用。
- **コスト vs. 品質のトレードオフ**:
  - コスト・latency 優先: 小さいモデル（0.13GB など）、低次元 embedding（384次元など）、512 トークン context length
  - 品質優先: 大型モデル、高次元 embedding、長い context length

### 試験頻出ポイント

- **コスト効率の正解**: ドメイン特化小型 LLM が「大型汎用モデルより安価で特定ドメインに適している」が正解。
- **コード生成**: CodeLlama-34B が専用モデルとして最適。
- **サードパーティ API 不使用**: Llama2-70B のような大型オープンソースモデルをオンプレミス運用。
- **多言語**: 多言語パフォーマンスベンチマークが最重要。モデルサイズや latency は二次的。
- **512 チャンクの最適設定**: context length 512 + 最小モデルサイズ 0.13GB + embedding 次元 384 がコスト・latency 最小化の組み合わせ。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「コード生成に汎用 LLM を使う」 | CodeLlama-34B のようなコード特化モデルが最適 |
| 「多言語チャットボットでモデルサイズを優先」 | 大きなモデルが多言語で常に優れるとは限らない |
| 「latency 優先なのに 14GB・4096次元 embedding を選ぶ」 | 過剰なスペックはコスト・latency の優先事項に矛盾 |
| 「ドメイン特化用途に汎用大型モデルを使う」 | 小型ドメイン特化モデルの方がコスト効率が高い |

## 9. Unity Catalog — モデル管理・ガバナンス

### 定義・概要

Unity Catalog は Databricks の中央集権型データ・AI ガバナンスプラットフォームであり、モデルのバージョン管理・アクセス制御・監査ログ・メタデータ管理を一元化する。MLflow のモデルレジストリと統合されており、エンタープライズグレードのモデル管理が可能。

### サブ概念・手法

- **モデル登録時のメタデータ**: モデル名・バージョン・スキーマ（入出力定義）・タグ・説明が必須メタデータ。
- **ガバナンス機能**: モデルのライフサイクル管理、権限管理、変更履歴追跡。
- **監査ログ（Audit Log）**: エンドポイントのクエリと応答を追跡し、コンプライアンスとトラブルシューティングを支援。
- **エンタープライズ RAG での役割**: 全モデルを Unity Catalog に登録 + 監査ログでエンドポイントを追跡することがコンプライアンス確保の方法。
- **モデルのバージョン管理**: バージョンを管理することで、ロールバックや A/B テストが可能になる。

### 試験頻出ポイント

- **Unity Catalog の3大メリット**: ガバナンス強化・バージョン管理・アクセス制御。
- **登録に必要な4要素**: モデル名・バージョン・スキーマ・タグ（説明も含む）。
- **エンタープライズコンプライアンス**: Unity Catalog 登録 + 監査ログの2点セット。
- **メタデータスキップは NG**: メタデータ設定をスキップするとガバナンスとコンプライアンス基準の追跡ができなくなる。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「メタデータ設定をスキップしてデプロイ」 | ガバナンス追跡に不可欠なメタデータを省略すべきでない |
| 「コスト削減のためにパブリックベクトルストアを使用」 | データセキュリティとコンプライアンスを損なう |
| 「Unity Catalog 登録はリアルタイムロギングを有効化する」 | リアルタイムロギングは Inference Tables の機能 |

## 10. Inference Logging / 推論監視

### 定義・概要

デプロイ済み LLM アプリケーションの品質・コスト・パフォーマンスを継続的に監視するために、inference logging（推論ログ）が使われる。Databricks では **Inference Tables** がネイティブのリクエスト・レスポンスログ記録機能として提供されている。

### サブ概念・手法

- **Inference Tables**: Databricks のネイティブ機能。デプロイ済みモデルエンドポイントのリクエスト・レスポンスをログ記録・監視するために特別に設計されている。マイクロサービスによるカスタムログの代替として使用可能。
- **監視対象**:
  - クエリパターン（どんなクエリが多いか）
  - 応答時間（latency）
  - エラー発生状況
  - トークン使用量（コスト最適化に活用）
- **コスト最適化への応用**: inference logging でトークン使用量のパターンを分析し、コストの高い操作を特定する。
- **動的スケーリングとの組み合わせ**: inference logging + 動的 GPU スケーリング + トークン使用量監視の3点がコスト最適化の標準セット。
- **Lakeview・DBSQL との違い**: Lakeview は Delta Lake 監視、DBSQL はデータクエリ用。どちらも inference log の記録には適していない。

### 試験頻出ポイント

- **Inference Tables**: Databricks ネイティブのログ記録機能。マイクロサービスの代替として問題に登場する。
- **コスト最適化の3点セット**: GPU 動的スケーリング + トークン使用量監視 + inference logging。
- **ファインチューニングはスケーラビリティコストに直接対処しない**: ファインチューニングはモデルパフォーマンス向上だが、コスト・スケーラビリティ最適化とは別物。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「Lakeview でリクエスト・レスポンスをログ記録する」 | Lakeview は Delta Lake 監視用。inference log 記録には不適 |
| 「DBSQL で inference log を管理する」 | DBSQL はデータクエリ用。ログ記録機能ではない |
| 「Vector Search でログを管理する」 | Vector Search はセマンティック検索用 |
| 「LLM ファインチューニングでスループット・コストを改善する」 | ファインチューニングはスケーラビリティコストに直接対処しない |

## 11. Feature Serving — リアルタイムデータアクセス

### 定義・概要

Feature Serving は LLM アプリケーションにリアルタイムデータへのアクセスを提供するコンポーネントである。事前計算された静的な embedding や知識ベースだけでは対応できない、刻々と変わるライブデータ（例: スポーツスコア、在庫状況、株価）を LLM に供給するために使用する。

### サブ概念・手法

- **主要用途**: リアルタイム更新が必要なデータ（スポーツスコア、スポーツ統計、在庫、株価など）を LLM アプリケーションに供給。
- **静的知識との違い**: 通常の RAG は過去のドキュメントを索引化した静的な知識を使う。Feature Serving はリアルタイムの動的データにアクセスできる。
- **Databricks の Feature Store**: Feature Serving のバックエンドとして利用可能。

### 試験頻出ポイント

- **Feature Serving の適用場面**: 「ライブスポーツスコアを表示する LLM アプリ」のような**リアルタイム更新が必要なユースケース**。
- **Vector Search との区別**: Vector Search はドキュメント検索用。Feature Serving はリアルタイム数値データ・特徴量供給用。
- **Fine-tuning との区別**: ファインチューニングはモデルのパラメータ更新。Feature Serving はリアルタイムデータ供給。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「Vector Search でリアルタイムスポーツスコアを提供」 | Vector Search はドキュメント意味検索用。リアルタイムデータ供給には Feature Serving |
| 「モデルをリアルタイムデータでファインチューニングする」 | ファインチューニングは都度リアルタイム更新には非現実的 |
| 「静的な知識ベースで対応する」 | 刻々と変わるデータには静的知識ベースでは追いつかない |

## 12. トークンベース課金 / コスト最適化

### 定義・概要

LLM アプリケーションのコストはトークン使用量・コンピューティングリソース・リクエスト数に依存する。リクエスト量が少ない場合はトークンベース（pay-per-use）課金が有利で、リクエスト量が多い場合はプロビジョニングスループットが経済的になる。

### サブ概念・手法

- **トークンベース（pay-per-use）課金**: リクエスト数が少ない場合に cost-efficient。固定コストなし。
- **プロビジョニングスループット**: 高トラフィック・高スループット環境向け。一定のキャパシティを確保。
- **バッチ処理**: 複数クエリをまとめて送信することで API コールを削減し、コストを下げる。
- **キャッシュ**: 頻繁なクエリをキャッシュすることで同一クエリの重複計算を避け、latency とコストを削減。
- **小型ファインチューニング済みモデル**: 大型汎用モデルの代わりにドメイン特化の小型モデルを使用することでコストと latency を削減。
- **コスト最適化の3点セット**: 動的 GPU スケーリング + トークン使用量監視 + inference logging。

### 試験頻出ポイント

- **低リクエスト量**: トークンベース（pay-per-use）が最適。プロビジョニングスループットは過剰投資。
- **高スループット**: プロビジョニングスループットが経済的。
- **コスト削減手法**: バッチ処理 + キャッシュ + 小型モデルの組み合わせ。
- **コスト優先の embedding 設定**: context length をチャンクサイズに合わせ（512）、最小モデル（0.13GB）、低次元 embedding（384次元）を選択。

### 誤答パターン

| 誤答 | 正解の理由 |
|------|-----------|
| 「リクエストが少ないのにプロビジョニングスループットを選ぶ」 | 低リクエストにはトークンベース課金が cost-efficient |
| 「精度向上のために検索バッチサイズを増やす（コスト最適化として）」 | 大バッチはコストを直接最適化しない |
| 「複数モデルを並行デプロイして同じクエリを処理」 | コストと複雑さが増すだけで効率は上がらない |
| 「最大の事前学習済みモデルを使う（コスト効率を優先する場面で）」 | 大型モデルはコストが高く、ドメイン特化用途では過剰 |

## 付録: 試験頻出の「正しい順序」まとめ

### RAG デプロイの標準フロー

```
ドキュメント取り込み
  → チャンキング（論理的セクション or トークン上限に合わせて）
  → embedding 生成（ドメイン固有モデル推奨）
  → Vector Search インデックスに保存
  → retriever 設定（セマンティック類似度最適化）
  → retriever と LLM を統合
  → Model Serving でエンドポイントデプロイ
```

### MLflow × Unity Catalog フロー

```
mlflow.log_model()（シグネチャ付き）
  → mlflow.register_model()（Unity Catalog へ名前・バージョン・スキーマ・タグ付き）
  → Model Serving エンドポイント起動
```

### MLflow PyFunc のパターン

```
入力受信
  → 前処理（クリーニング・正規化）
  → モデル推論（predict()）
  → 後処理（ユーザーフレンドリーなフォーマット変換）
  → 出力返却
```

### LangChain チェーンの最小構成

```
プロンプトテンプレート（ユーザー入力構造化）
  + retriever（関連ドキュメント取得）
  + LLM（応答生成）
```

## 付録: よく出る誤答選択肢パターン一覧

| 誤答パターン | 正解への判断軸 |
|-------------|--------------|
| 「要約モデルで〇〇を実現する」 | 要約は特定 QA・動的応答生成に不十分 |
| 「ルールベースシステムでマッピング」 | 動的クエリには硬直的。LLM+retriever に置き換える |
| 「分類モデルで応答生成」 | 分類は詳細な回答・説明を提供しない |
| 「retriever をスキップして embedding だけ使う」 | retriever は精密なドキュメントルックアップに不可欠 |
| 「固定リソースでピーク対応」 | 変動ワークロードには autoscaling が必要 |
| 「CPU のみでコスト節約」 | 高スループット推論には GPU が必要 |
| 「チャンクサイズを増やして精度改善」 | 大きなチャンクは無関係コンテンツを含め関連性を希薄化 |
| 「共有アクセスキーで全ユーザー認証」 | 個別追跡・制御のために RBAC が必要 |
| 「LLM ファインチューニングでコスト・スケール改善」 | スケーラビリティコストは autoscaling + inference logging で対処 |
| 「latency 削減のためにシグネチャを登録する」 | シグネチャの目的は入出力フォーマット整合性の確保 |
