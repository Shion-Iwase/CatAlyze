### プロンプト設計（出力フォーマット指定）

- 定義・概要：
  LLMに対して「何を」「どんな形式で」出力してほしいかを明示的に指示するプロンプト技術。JSON、箇条書き、特定フィールドの列挙など、具体的なフォーマットを指定することで出力の一貫性と精度が大幅に向上する。

- サブ概念：
  - JSON出力指定: `{"date": "...", "sender_email": "...", "order_id": "..."}` のようなスキーマを例示することで、LLMが正確な構造の出力を生成する。例示がない場合は曖昧・不一致なJSON出力になりやすい。
  - 箇条書き指定: 「箇条書きで要約する」と明示することで、段落形式やアブストラクト形式との混同を防ぐ。
  - フィールド列挙: 「アクションアイテム、参加者、決定事項を含む」など必須フィールドを列挙することで、省略を防ぐ。
  - 具体的な制約指定: 「短く」「100語以内」「ステップバイステップで」などの制約も出力制御に有効。
  - 出力例の提示（Few-Shotとの組み合わせ）: 例があるとなお精度が高い。

- 試験頻出ポイント：
  - 「最高精度を確保するプロンプトはどれか」という問いでは、**必ず出力形式の例（JSON サンプルなど）を含むもの**が正解になる。
  - 「JSON形式で返してください」だけでは不十分。**具体的な例を添付**することが精度向上に不可欠。
  - 「温度設定だけで応答の長さは制御できない」→ 長さや形式の制御はプロンプト内の明示的な指示が必要。
  - 出力フォーマットを指示することと、内容（何を含めるか）の指示は**両方必要**。

- 誤答パターン：
  - 「`JSON形式で返してください`と書けば十分」と思いがちだが、実際は「例（サンプルスキーマ）を添付しないと曖昧または不一致な出力になる」。理由：LLMは例がある方が期待する構造を正確に学習できる。
  - 「`温度設定を下げれば短い応答になる`」と思いがちだが、実際は「temperatureはランダム性を制御するものであり、応答の長さは制御しない」。理由：長さの制御には明示的な長さ指示が必要。
  - 「「要約してください」のような汎用プロンプトで十分」と思いがちだが、実際は「必要フィールドを明示しないと省略が生じる」。理由：LLMは省略しやすい部分（参加者名、決定事項など）を勝手に省く。

### プロンプト設計（Few-Shot例の活用）

- 定義・概要：
  プロンプトに成功例（入出力ペア）を複数添付し、LLMが望ましい出力パターンを学習するよう誘導する手法。Zero-Shot（例なし）より安定した出力が得られる。特定のトーン・構造・ドメインへの誘導に有効。

- サブ概念：
  - Few-Shot プロンプト: 2〜5個程度の成功例をプロンプト内に提示する。
  - Zero-Shot プロンプト: 例なしで指示だけで出力させる。汎用タスクには使えるが、ドメイン特有・フォーマット固定のタスクには不十分。
  - One-Shot プロンプト: 例を1つだけ提示する中間形式。
  - 出力フォーマットの誘導: 例を通じて「説明を含めずに要約だけ出力する」などのフォーマットを伝える。
  - ドメイン別例: 広告見出し、製品説明、貯蓄プランなど、ユースケース固有の例を提供することで品質が向上する。

- 試験頻出ポイント：
  - 「LLMが要約に生成方法の説明を付けてしまう問題への対処」→ Few-Shot 例で望ましいフォーマット（説明なしの要約）を示すことが正解。
  - 「広告見出しや製品説明の生成精度向上」→ 成功例の提示 + ターゲット・トーン・目標の明示が正解。
  - 「一般的なプロンプト（"製品説明を書いてください"）は不十分」→ 具体的な制約と例の組み合わせが必要。
  - Few-Shot 例は「分類モデルのトレーニング」や「ファインチューニング」とは別物であり、モデル変更不要。

- 誤答パターン：
  - 「LLMの出力を後処理で改行分割して不要部分をカットすれば良い」と思いがちだが、実際は「根本問題（フォーマット指示不足）に対処せず、要約の重要部分が欠落するリスクがある」。理由：後処理は応急処置に過ぎず、Few-Shot で事前に形式を教える方が正確。
  - 「トーンの指定を省いた方が創造性が高まる」と思いがちだが、実際は「トーン未指定ではターゲットオーディエンスとの不一致が生じる」。理由：広告や製品説明はブランドトーンと一致させることが品質の定義に含まれる。

### プロンプト設計（コンテキスト・入出力定義）

- 定義・概要：
  パイプライン設計の**最初のステップ**として、LLMへの入力フィールドと期待する出力フィールドを明確に定義する。これによりシステム全体のアーキテクチャ設計がガイドされる。

- サブ概念：
  - 入力フィールド定義: `budget`, `preferred_weather`, `activities` など、ユーザーから受け取るデータ項目を特定する。
  - 出力フィールド定義: `destination`, `activity_recommendations` など、システムが返すべきデータ項目を特定する。
  - 入出力スキーマ例: 例として `{"budget": "$3000", "preferred_weather": "tropical"} → {"destination": "Thailand"}` のようなペアを定義する。
  - コンテキスト注入: 「ユーザーは遅延配送について質問しています」などの具体的な状況をプロンプトに含めることで関連性の高い応答が得られる。
  - 追加コンテキスト拡張: ポーションサイズ・食事の好み・予算など、ユーザーが明示しなかった情報をプロンプトに補完することで出力品質が向上する。

- 試験頻出ポイント：
  - 旅行チャットボット、食事プラン推薦、カスタマーサービスなど、**パイプライン設計の「最初のステップ」は必ず入出力定義**。
  - 「モデル選択」「フィードバック収集」「データベース構築」は入出力定義の**後**に行う。
  - カスタマーサービス向け会話型AIのプロンプトでは、トーンだけでなく「明確なコンテキスト＋望ましい出力」の両方を含めることが正解。
  - 「創造性を促すために例を避ける」「トーンのみを指定する」は不正解。AIのガイダンスには**具体的な出力目標**が不可欠。

- 誤答パターン：
  - 「まずモデルを選んでからパイプラインを設計すべき」と思いがちだが、実際は「入出力関係を先に定義することでモデル選択の基準が決まる」。理由：ビジネス要件（出力の形・内容）がモデル選択を規定する。
  - 「トーンのみ指定すれば会話型AIの品質が高まる」と思いがちだが、実際は「トーンは表現形式に影響するが、必要な出力内容をAIに伝えていないため関連性のある応答が得られない」。理由：モデルは出力の内容（何を推薦すべきか）をトーン指示だけから推測できない。

### RAGアーキテクチャ設計（Retriever + LLM の組み合わせ選択）

- 定義・概要：
  Retrieval-Augmented Generation（RAG）は、ベクトルデータベースから関連ドキュメントを検索し（Retrieval）、その内容をLLMに与えて回答を生成する（Generation）アーキテクチャ。Retriever と LLM の適切な組み合わせがシステム品質を決定する。

- サブ概念：
  - Retriever: 関連ドキュメントやセクションを取得するコンポーネント。ベクトル類似度検索を使うものが主流。
  - Embedding モデル: テキストをベクトル空間に変換し、セマンティックな類似度計算を可能にする。
  - Vector Store / Vector Database: Embedding ベクトルを保存・検索するデータベース（FAISS、Chromaなど）。
  - LLM（生成コンポーネント）: Retrieverが取得したコンテキストをもとに自然言語の回答を生成する。
  - ドメイン固有のfine-tuned Embedding: 特定ドメイン（法律・金融など）のテキストでfine-tuningすることで検索精度が向上する。
  - RAGパイプラインの正しい順序: Embedding生成 → Vector Searchインデックス保存 → Retriever構築 → LLM/要約モデルデプロイ。

- 試験頻出ポイント：
  - 「法的契約書レビューシステム」「ポリシーQ&A」「医療ガイドラインQA」「在庫QA」「映画推薦」など多様なユースケースで、**Retriever + LLMの組み合わせ**が正解パターンとして繰り返し登場する。
  - 要約モデル単体・分類モデル単体・ルールベースシステム単体はいずれも「動的なクエリへの対応」や「引用付き回答生成」ができないため不正解。
  - 検索精度（Precision）が低い（無関係な結果が多い）場合は、**ドメイン固有データでembeddingモデルをfine-tuning＋セマンティック類似度最適化のRetriever**が改善策。
  - **「RAGパイプラインの正しい構築順序」**: Embedding作成 → インデックス保存 → Retriever構築 → 生成モデルデプロイ。Retrieverを先にデプロイしてからインデックス化するのは誤り。

- 誤答パターン：
  - 「要約モデルを使えばドキュメントから回答が得られる」と思いがちだが、実際は「要約は既存テキストの圧縮であり、クエリへの動的応答ではない」。理由：ユーザーの具体的な質問に答えるにはRetrievalとGenerationの両方が必要。
  - 「分類モデルで条項を分類すれば法的QAになる」と思いがちだが、実際は「分類はラベル付けを行うだけで、回答文の生成はできない」。理由：法的アドバイスや説明の生成には LLM が必要。
  - 「キーワードベースの検索でドキュメントを取得すれば精度は十分」と思いがちだが、実際は「キーワードが一致しなくても意味的に関連するドキュメントを取り逃がす」。理由：法律・医療文書では同義語・言い換えが多く、セマンティック検索が必須。
  - 「embeddinngモデルを変更するより別のLLMに切り替えた方が応答品質が上がる」と思いがちだが、実際は「無関係な応答の原因はRetrieval（検索精度）にあり、LLMを変えても解決しない」。理由：RAGの品質はRetrieverとGeneratorを分けて診断する必要がある。

### LLMとコンポーネント選択（タスク別モデル選択）

- 定義・概要：
  タスクの性質に応じて最適なモデルタイプ（LLM、Embeddingモデル、分類モデル、要約モデルなど）を選択することは、RAGおよびGenerative AIアプリ設計の根幹。各コンポーネントの役割と限界を明確に理解することが重要。

- サブ概念：
  - LLM（大規模言語モデル）: 動的なテキスト生成・構造化出力・推論・要約・テンプレート生成に最適。「動的に生成する」タスクにはLLMを使う。
  - Embeddingモデル: テキストのセマンティックな意味をベクトルに変換する。セマンティック検索・マッチング・類似度計算に使用。
    - Sentence Transformers: 文レベルのセマンティック関係のキャプチャに最適（意図識別など）。GloVe/Word2Vecは単語レベルで文全体の意味をキャプチャしない。
    - デュアルエンコーダー（Dual Encoder）: 短いクエリと長いドキュメントに別々のEncoderを使う戦略。非対称検索（短いクエリ→長い文書）に有効。
    - ドメイン固有Fine-tuning: 金融テキストや法律テキストでfine-tuningされたEmbeddingは、汎用EmbeddingよりRAG精度が高い。
  - 分類モデル: カテゴリへの割り当て（ラベル付け）のみが目的。生成や検索はできない。
  - 要約モデル: テキストを短くまとめる（情報圧縮）。クエリへの回答生成や構造化出力生成はできない。
  - テキスト要約タスク: 会議議事録・ユーザーレビュー・法的ドキュメントを主要ポイントに凝縮する場合に最適。NER・テキスト分類・感情分析とは明確に用途が異なる。
  - RAGタスク: 検索と生成を組み合わせ、知識ベースから情報を取得しつつ動的に応答を生成するタスク。旅行チャットボット・Q&Aシステムに適合。

- 試験頻出ポイント：
  - 「会議議事録の主要ポイント・アクションアイテム・決定事項を特定するタスク」→ **テキスト要約**（NER・感情分析・テキスト分類は不正解）。
  - 「ユーザーレビューから共通テーマとセンチメントを強調する要約」→ **テキスト要約**（感情分析は「感情的トーンを判断」するだけで要約はしない）。
  - 「旅行チャットボット（目的地回答・活動提案・宿泊推薦）」→ **RAG**（テキスト分類・NER・要約は不正解）。
  - 「求人票→評価チェックリスト変換」→ **Embedding + LLM**（要約・分類・ルールベースは不正解）。
  - 「非構造化フィードバック→構造化タスク（優先事項・期限）」→ **LLM**（要約・分類・ルールベースは不正解）。
  - 「在庫QAシステム」→ **Retriever + LLM**（Embedding単体・要約・分類は不正解）。

- 誤答パターン：
  - 「感情分析で顧客レビューのテーマが把握できる」と思いがちだが、実際は「感情分析は感情的トーン（ポジティブ/ネガティブ）を判断するのみで、共通テーマの要約はしない」。理由：テーマ抽出には要約が必要。
  - 「分類モデルでユーザークエリを分類すれば詳細な回答が生成できる」と思いがちだが、実際は「分類はラベル付けのみで、回答の生成はLLMが行う」。理由：分類はカテゴリを返すだけで自然言語の説明は生成しない。
  - 「NERで会議議事録のアクションアイテムが特定できる」と思いがちだが、実際は「NERは固有表現（人名・地名など）を抽出するのであり、アクションアイテムのような意味的な要素は抽出できない」。理由：アクションアイテムは言語的意味の理解が必要で、要約タスクに属する。

### Temperature パラメータ

- 定義・概要：
  LLMの出力のランダム性・多様性を制御するハイパーパラメータ。値が高いほどランダムかつ創造的な出力になり、低いほど決定論的で一貫性のある出力になる。

- サブ概念：
  - Temperature = 0（またはそれに近い値）: 決定論的出力。最も確率の高いトークンのみを選択。事実確認・分類・JSON生成など精度重視タスクに適する。
  - Temperature = 0.5: ランダム性と決定性のバランスを取る中間値。最大の創造性は得られない。
  - Temperature >= 1.0（高い値）: ランダム性が増し、多様で創造的な出力を生成。ブレインストーミング・ストーリーテリング・広告コピーなどに適する。
  - Temperatureと応答の長さの独立性: Temperatureは「ランダム性」を制御するのみで「応答の長さ」は制御しない。長さはプロンプト内の明示的な指示が必要。
  - Temperatureと機密データ漏洩の独立性: Temperatureを下げても機密情報の漏洩は防げない。セーフティフィルターや後処理フィルターが必要。

- 試験頻出ポイント：
  - 「創造的で多様な回答を生成する設定」→ **Temperatureを1.0以上の高い値に上げる**（0に近づけるは逆効果）。
  - Temperatureは「ランダム性・創造性」に関与する。「応答の長さ」「機密データ漏洩防止」「アクセス制御」には無関係。
  - 法的アドバイスチャットボットのguardrail実装で「Temperatureを下げれば不正確な回答を防げる」は誤り → 正解はMetaprompt + 免責事項。

- 誤答パターン：
  - 「Temperatureを下げると出力が短くなる」と思いがちだが、実際は「Temperatureは長さではなく確率分布のシャープさに影響する」。理由：出力の長さはモデルの停止基準やプロンプト指示に依存する。
  - 「Temperatureを0にすれば機密情報が漏れない」と思いがちだが、実際は「Temperatureの変更は出力内容の機密性に直接影響しない」。理由：機密情報の漏洩はモデルが持つ知識の問題であり、後処理フィルターで除去する必要がある。

### マルチターン対話とメモリモジュール

- 定義・概要：
  チャットボットがマルチターン（複数回のやり取り）の会話を通じてコンテキストを維持するために、過去のインタラクション履歴を保存・参照するモジュール。LLMワークフローに統合されることで、フォローアップクエリへの連続的な対応が可能になる。

- サブ概念：
  - メモリモジュール（Memory Module）: 過去の会話履歴を動的に保存・取得するコンポーネント。LangChainなどに組み込み機能がある。
  - 会話履歴の要約: 履歴を要約して渡す手法もあるが、重要詳細が削除されコンテキスト精度が低下するリスクがある。
  - Retrieverとの違い: Retrieverは外部ドキュメントからコンテンツを検索するが、動的に会話コンテキストを蓄積・保存する機能はない。
  - 分類モデルとの違い: ユーザー意図の識別は支援するが、会話コンテキストの保持はしない。

- 試験頻出ポイント：
  - 「マルチターン対話でコンテキストを維持する不可欠なツール」→ **LLMワークフローに統合されたメモリモジュール**。
  - Retriever・要約モデル・分類モデルはいずれもコンテキスト保持の代替にはならない。
  - メモリモジュールは「動的な保存と取得」が特徴。

- 誤答パターン：
  - 「Retrieverがあれば過去の会話を取得できるからコンテキスト維持に使える」と思いがちだが、実際は「Retrieverは外部知識ベースを検索するコンポーネントであり、リアルタイムの会話履歴を動的に蓄積する機能は本質的に持たない」。理由：会話履歴の管理はメモリモジュールが専門。

### マルチステップ推論パイプラインの設計原則

- 定義・概要：
  意図識別・情報取得・応答生成などの複数ステップを持つ推論パイプラインを構築する際の設計原則。ツール間の相互運用性とデータのシームレスな受け渡しが品質とスケーラビリティの鍵。

- サブ概念：
  - 相互運用性（Interoperability）: 各ツール・コンポーネントが異なるステージ間でデータをシームレスに共有できること。ボトルネックの防止に不可欠。
  - ステージ間の接続: 意図識別 → ドキュメントRetrieval → 応答生成という流れが基本。各ステージの出力が次ステージの入力になる。
  - ルールベースシステムの限界: 事前定義したルールに縛られ、多様なクエリへの適応が困難。動的な推論パイプラインには不適。
  - ツール数の最小化の誤解: ツールを減らすことで複雑さは減るが、マルチステップ推論に必要な機能が失われるリスクがある。

- 試験頻出ポイント：
  - 「マルチステップ推論パイプラインの重要な設計原則」→ **ツール間の相互運用性とデータのシームレスな共有**。
  - 「ツール数を最小化する」「ルールベースでドキュメントRetrievalを行う」「Retrievalなしで応答生成に集中する」はいずれも誤り。

- 誤答パターン：
  - 「複雑さを減らすためにツールの数を最小化すべき」と思いがちだが、実際は「マルチステップ推論では複数のツールが必要であり、ツールを減らすと対応できるタスクが制限される」。理由：意図識別・検索・生成はそれぞれ専門のコンポーネントが最適。

### ベクトルデータベース（Vector Store / Vector Search）

- 定義・概要：
  テキストや画像などをEmbeddingモデルでベクトル化し、高速な類似度検索を可能にするデータベース。RAGシステムにおけるRetrievalコンポーネントの基盤技術。キーワードマッチングではなく意味（セマンティクス）に基づく検索が特徴。

- サブ概念：
  - FAISS（Facebook AI Similarity Search）: ベクトルインデックスの作成とクエリを高速に処理するライブラリ。低latencyのリアルタイムVector Searchに最適。
  - Databricks Vector Search: DatabricksネイティブのVector Searchソリューション。
  - セマンティック検索 vs キーワード検索: キーワード検索は完全一致が必要で同義語・言い換えを見逃す。セマンティック検索は意味に基づいて検索するため、表現が異なっても関連ドキュメントを取得できる。
  - インデックス化の順序: Embedding生成 → インデックス保存 → Retriever構築（インデックス化の前にRetrieverをデプロイするのは誤り）。
  - ストレージ最適化: Embeddingのストレージ最適化は高throughputのリアルタイムクエリに直接影響する。

- 試験頻出ポイント：
  - 「弁護士向けRAGシステムで法的ドキュメントのRetrieval精度を確保するコンポーネント」→ **セマンティック検索のためのベクトルデータベース**。
  - 「法的RAGで契約条項を正確に取得するコンポーネント」→ **ベクトルデータベース**（ルールベース・キーワード検索・LLMのfine-tuningは不正解）。
  - 「リアルタイムVector Searchに必要なリソース」→ FAISS + 高性能GPU + 分散データベースの3つ。
  - 「RAGパイプラインの正しい構築順序」→ Embedding作成 → Vector Searchインデックス保存 → Retriever構築 → 要約モデルデプロイ。

- 誤答パターン：
  - 「LLMをfine-tuningすれば法的ドキュメントのRetrieval精度が上がる」と思いがちだが、実際は「fine-tuningは要約や生成の品質を上げるが、ドキュメントの検索（Retrieval）自体には対処しない」。理由：Retrieval精度の向上にはEmbeddingとベクトルDBの最適化が必要。
  - 「ルールベースシステムで法的条項を抽出できる」と思いがちだが、実際は「法律文書は言い換えや同義語が多く、ルールベースの硬直性では柔軟な検索ができない」。理由：セマンティック検索はコンテキストや意味を考慮できる。

### Embeddingモデルの選択と活用

- 定義・概要：
  テキストを高次元ベクトルに変換するモデル。ベクトル化されたテキスト間の距離（コサイン類似度など）を使って意味的な類似性を測定する。モデル選択の基準はタスク・ドキュメント特性・ドメインによって異なる。

- サブ概念：
  - Sentence Transformers: 文レベルのセマンティック関係のキャプチャに特化したモデル。チャットボットの意図識別など文単位の意味理解タスクに最適。
  - GloVe / Word2Vec: 単語レベルのEmbedding。文全体の意味を効果的にキャプチャしない。SemanticタスクにはSentence Transformersが優位。
  - BERT-base（分類用）: 分類タスク向けに設定されており、セマンティック検索用Embeddingの生成には最適化されていない。
  - デュアルエンコーダー（Dual Encoder）: クエリ用とドキュメント用の2つの別々のEncoderを持つ。短いクエリと長いドキュメントの非対称検索に最適。
  - ドメイン固有fine-tuning: 金融テキスト・法律テキスト・医療テキストでfine-tuningすることで、汎用Embeddingより検索精度が向上する。
  - Context Length: 長い研究論文などの長文書を処理する場合、Embeddingモデルがサポートするcontext lengthが重要。
  - セマンティック類似度精度: 高精度のEmbeddingは検索クエリが最も関連性の高い結果を取得することを保証する。

- 試験頻出ポイント：
  - 「チャットボットクエリから顧客意図を識別・抽出するEmbeddingモデル」→ **Sentence Transformers**（GloVe・Word2Vec・BERT-base分類は不正解）。
  - 「法的契約書や法的ドキュメントのRAGで検索精度を向上させるEmbedding戦略」→ **ドメイン固有データでfine-tuned Embedding**。
  - 「長い研究論文を処理する検索エンジンのEmbeddingモデル選択基準」→ **context length + セマンティック類似度精度**（推論コスト・データセットサイズは二次的）。
  - 「短いクエリと長いドキュメントの非対称検索」→ **デュアルエンコーダー**。

- 誤答パターン：
  - 「Word2VecはEmbeddingモデルだから意図識別に使える」と思いがちだが、実際は「Word2Vecは単語レベルのEmbeddingであり、文の意味（文脈）を総合的にキャプチャしない」。理由：意図識別は文全体の意味理解が必要でありSentence Transformersが適切。
  - 「RAGのEmbeddingモデルは汎用事前学習モデルで十分」と思いがちだが、実際は「ドメイン固有のテキストでfine-tuningすることで検索精度が大幅に向上する」。理由：汎用モデルは特定ドメインの専門用語や概念の意味を正確に表現できないことがある。

### LangChain によるワークフローオーケストレーション

- 定義・概要：
  LangChainはLLMを含むマルチステップワークフロー（Chain）を構築するためのフレームワーク。Retriever・Embeddingモデル・LLMなどのコンポーネントをモジュール型ツールで接続し、RAGパイプラインや対話型エージェントを効率的に構築できる。

- サブ概念：
  - Chain（チェーン）: 複数のコンポーネントを順次または条件付きで接続したワークフロー。
  - モジュール型ツール: Retriever・Embedding・LLMを標準インターフェースで接続するコンポーネント群。
  - エージェント（Agent）: ツールを使い分けて複雑なタスクを自律的にこなすLangChainの高度機能。
  - ドキュメントRetrieval + 生成の統合: LangChainの定義的な機能。RetrievalとGenerationのワークフローをオーケストレーション。
  - LangChainが対応しないこと: モデルのfine-tuning、出力比較の可視化ツール、計算コスト削減最適化は対象外。
  - プロンプトテンプレート: LangChainはプロンプトテンプレートを提供し、動的なクエリ生成をサポート。

- 試験頻出ポイント：
  - 「マルチステップLLMベースワークフローを構築するための最適なライブラリ」→ **LangChain**（PySpark・TensorFlow・Pandasは不正解）。
  - 「LangChainがRetrievalとLLMの統合に理想的な主な機能」→ **ドキュメントRetrievalと生成機能を組み合わせた構造化ワークフローを作成する能力**。
  - LangChainはfine-tuningは**しない**。オーケストレーションに特化。
  - 「LangChainでメールIDに基づくユーザー詳細抽出チェーンに必要なコンポーネント」→ **retriever + LLM + プロンプトテンプレート**。

- 誤答パターン：
  - 「LangChainはEmbeddingを自動的にfine-tuningしてくれる」と思いがちだが、実際は「LangChainはEmbeddingを活用するが、fine-tuning機能は持たない」。理由：LangChainはオーケストレーションツールであり、モデルのトレーニング機能はない。
  - 「LangChainは出力比較のための可視化ツールを提供する」と思いがちだが、実際は「LangChainの主目的はワークフロー構築であり、可視化はMLflowなど別ツールの役割」。理由：ツールの責任分担を正確に理解することが重要。

### System Prompt / Metaprompt によるスコープ制御

- 定義・概要：
  システムプロンプト（またはMetaprompt）は、アプリケーション全体のLLMの動作方針・制約・スコープをあらかじめ定義する指示文。ユーザーのクエリとは独立して、モデルが「何をすべきか・すべきでないか」を規定する。

- サブ概念：
  - システムプロンプトによるスコープ制限: 「〜に関係のない質問には答えない」などの制約を加えることで、LLMがスコープ外の回答を出力しないよう制御する。
  - Metapromptによる行動規定: 法的チャットボットに「法的アドバイスを避けて一般情報のみに限定する」「特定の機密情報を出力に含めない」などの指示を組み込む。
  - システムプロンプトとデータフィルタリングの違い: システムプロンプトはLLMの回答範囲を制限するが、ベクトルDB内の無関係なドキュメントを物理的に削除するわけではない。ただし、「この質問には答えない」という制約として機能する。
  - 単一チャンクへの統合は誤り: ドキュメントを1つのチャンクに統合すると検索粒度と精度が下がる。

- 試験頻出ポイント：
  - 「RAGアプリで無関係な情報に対するフィルタリング手法」→ **システムプロンプトで「関係のない質問には答えない」と明示する**（全ドキュメント保持・単一チャンク統合は不正解）。
  - 「metapromptによるプライベートデータ露出の軽減」→ **特定の機密情報を出力に含めないよう指示する**（数値出力ブロック・temperature調整・応答長短縮は不正解）。
  - システムプロンプトでは「不正確な法的アドバイスの防止」のためにスコープを一般情報に限定することができる。

- 誤答パターン：
  - 「データフィルタリングが実行されない場合でもシステムプロンプトで「すべての情報はSnoPen AIに関する」と書けば対処できる」と思いがちだが、実際は「システムプロンプトは適切なドキュメントフィルタリングの欠如を補えない」。理由：無関係なドキュメントが検索されLLMに渡されると、システムプロンプトだけでは防ぎきれない場合がある。
  - 「すべてのドキュメントを1チャンクに統合することでコンテキストを保持できる」と思いがちだが、実際は「単一の大きなチャンクは粒度と検索精度を低下させ、トークン制限を超えるリスクがある」。理由：RAGは細かいチャンクで精度の高い検索を行うことが前提。

### セーフティフィルターと入力バリデーション

- 定義・概要：
  LLMアプリケーションを悪意ある入力（プロンプトインジェクション・システムクラッシュを狙った入力など）から保護するためのメカニズム。入力を受け付ける前に検証・フィルタリングを行うことで安全な動作を確保する。

- サブ概念：
  - 入力バリデーション: 許容できないパターンやフォーマットを検出して拒否するルールの実装。システムクラッシュを狙った敵対的入力への防御に有効。
  - セーフティフィルター: 有害・不適切・悪意のある入力を検出し、LLMが処理しないよう遮断するフィルター。誕生日詩生成アプリなどで悪用を防ぐ。
  - 後処理フィルター: 生成された出力から機密情報・有害コンテンツを検出して削除するフィルター（後段での防御）。
  - プロファニティフィルター: 不適切な言語をスクリーニングするフィルター。コンテンツの「関連性」向上には寄与しない（言語の適切さのみを担保）。

- 試験頻出ポイント：
  - 「悪意ある入力から誕生日詩生成アプリを保護する最も効果的な技術」→ **有害な入力を検出するセーフティフィルター + LLMがサポートできないと応答する仕組み**。
  - 「敵対的入力からアプリをクラッシュから守る防御戦略」→ **入力バリデーションルールの実装**。
  - 「プロファニティフィルターは金融ニュースへの関連性向上に役立つか」→ **役立たない**。言語の安全性のみに対処するため、コンテンツの関連性とは無関係。
  - セーフティフィルターは「悪意ある入力を特定後に会話を続ける」のではなく、**LLMへの到達を阻止する**。

- 誤答パターン：
  - 「対話時間を短縮すれば悪意ある入力を減らせる」と思いがちだが、実際は「時間制限はセキュリティメカニズムとして機能しない」。理由：悪意ある入力は短い時間でも実行可能。
  - 「悪意ある入力だとLLMがユーザーに伝えれば十分で会話は続けられる」と思いがちだが、実際は「悪意ある入力と判定した後も会話を続けるとさらなるセキュリティリスクが生じる」。理由：会話を終了することで被害を最小化できる。

### テキスト要約タスク vs 他タスクの使い分け

- 定義・概要：
  テキスト要約（Text Summarization）は長文テキストを主要な情報を保持しつつ短縮するタスク。NER・感情分析・テキスト分類・RAGと比較した際のそれぞれの適用範囲の違いを正確に理解することが試験では繰り返し問われる。

- サブ概念：
  - テキスト要約: 会議議事録・ユーザーレビュー・研究論文・法的ドキュメント等を主要ポイントに凝縮。テーマ・アクションアイテム・決定事項のキャプチャに適する。
  - 感情分析（Sentiment Analysis）: テキストの感情的トーン（ポジティブ/ネガティブ/中立）を判断する。共通テーマの抽出や要約はしない。
  - 固有表現認識（NER）: 人名・地名・組織名・日付などの固有表現を抽出する。コンテンツの要約やアクションアイテムの特定はしない。
  - テキスト分類: テキストをカテゴリに分類する。会議議事録のサマリーや旅行推薦は生成しない。
  - RAG: 検索と生成を組み合わせてナレッジベースに基づいた応答を生成する。単純な要約タスクより高度で動的な回答生成が必要な場面に適する。

- 試験頻出ポイント：
  - 「会議議事録から主要ポイント・アクションアイテム・決定事項を特定」→ **テキスト要約**。
  - 「製品レビューから共通テーマとセンチメントを強調した要約」→ **テキスト要約**（センチメントは「テーマの一部」として要約に含めるのが正解。感情分析は不正解）。
  - 「旅行チャットボット（目的地回答・アクティビティ提案・宿泊推薦）」→ **RAG**（要約・NER・分類は不正解）。
  - 「ユーザーレビューを分類されたインサイトに変換する（カテゴリ：製品品質・配送・CS）」→ **分類モデル**（EmbeddingやRetrieval・要約は不正解）。

- 誤答パターン：
  - 「感情分析でレビューのテーマが要約できる」と思いがちだが、実際は「感情分析はトーンのみを判断し、共通テーマを抽出・要約する機能はない」。理由：「共通テーマとセンチメントを強調した要約」には要約タスクが適切。
  - 「旅行チャットボットには要約が適している」と思いがちだが、実際は「旅行推薦はナレッジベースからの情報取得と動的な回答生成が必要であり、RAGが最適」。理由：要約は静的なテキスト処理であり、リアルタイムのクエリ応答には向かない。

### 構造化出力（JSON形式）の生成

- 定義・概要：
  LLMに対してJSONなどの構造化フォーマットで出力するよう指示する手法。下流システムがプログラム的に処理できる形式の出力を得るために重要。プロンプト設計における出力仕様の明確化が精度を左右する。

- サブ概念：
  - JSON出力指定: プロンプトに `JSON形式で出力してください` と記述する。さらにサンプルスキーマを提示することで精度が向上。
  - フィールド指定: `item_name`, `stock_level`, `location` など必要なフィールドを明示することで、不足・余分なフィールドを防ぐ。
  - 分類ラベル付きJSON: コール分類システムなど、`{"call_id": "123", "label": "In Stock"}` 形式でラベルと関連データをセットで出力する。
  - 後処理との関係: 後処理（JSON変換など）も有用だが、プロンプトで出力形式を事前に指定することが根本的な解決策。
  - PyFunc モデルでの後処理: MLflow PyFunc モデルの `predict` 関数内で後処理（JSON変換等）を実装することで、デプロイ時のシームレスなワークフローが確保される。

- 試験頻出ポイント：
  - 「在庫クエリのJSON形式レスポンス生成プロンプト」→ `item_name, stock_level, location` などのフィールドを**明示的に指定**したプロンプトが正解。
  - 「コール分類ラベルの高精度JSON出力プロンプト」→ 明確な指示 + JSON形式例（`{"call_id": "123", "label": "In Stock"}`）の両方を含むプロンプトが正解。
  - 「フォーマットは強制しない」「汎用プロンプト」は不正解。

- 誤答パターン：
  - 「`JSON形式で返してください`だけ書けば正確なJSONが得られる」と思いがちだが、実際は「フィールド名や例を添付しないと一貫性のないJSONが出力される可能性がある」。理由：LLMは期待するスキーマを例から学習する。
  - 「フォーマットは強制せず、LLMに任せれば適切なフォーマットで返してくれる」と思いがちだが、実際は「明確なフォーマット指示がないと出力が不一致になり、下流処理でエラーが生じる」。理由：構造化出力には明示的な仕様が必要。
