### チャンキング戦略（Chunking Strategy）

- **定義・概要：**
  ドキュメントをRAGシステムでのベクトル検索・取得に適したサイズに分割する手法。トークン制限の遵守と意味的文脈の保持を両立させることが目的。チャンクのサイズや分割方法によって retrieval 品質・LLMの応答完全性が大きく変わる。

- **サブ概念：**
  - **論理的チャンキング（Logical/Structural Chunking）**: 見出し・サブ見出し・段落・チャプターなどのドキュメント構造に沿って分割する方法。コンテキストが保持され retrieval 品質が高い。**最頻出の正解パターン**。
  - **固定サイズチャンキング（Fixed-size Chunking）**: 500トークン・1000トークンなど均等サイズで分割する方法。意味のあるセクションを分断するリスクがあるため通常は誤答。
  - **オーバーラップ（Overlap）**: チャンク間でテキストを重複させてコンテキスト断絶を防ぐ手法。ただし、30%・50%・200トークンなど「過度なオーバーラップ」は冗長性を生むだけで効果が薄いとされ、**試験では誤答扱い**になることが多い。
  - **チャンクサイズを小さくする**: LLMのトークン制限超過による不完全応答への対処として有効。
  - **セグメントラベル追加**: チャンクにセクションヘッダーなどのラベルを付与してコンテキスト理解を向上させる手法。
  - **フラット化（Flatten）**: チャンク配列データフレームを「1チャンク1行」にフラット化して一意のIDを付与し Delta テーブルに保存する。Vector Search インデックス用の最適保存形式。
  - **スキーマ定義→チャンキング→Delta Lake書き込み**: フィードバックデータや断片化ドキュメントをDelta Lakeに保存する正しいワークフロー順序。

- **試験頻出ポイント：**
  - 法的テキスト・小説・政府レポートなど**構造のあるドキュメント**には「論理的チャンキング（見出し・段落・チャプター単位）」が最善。
  - チャンクサイズを**小さく**することで LLM のトークン制限内に収まり、応答の完全性が向上する。
  - ドキュメント全体を1チャンクにするのはトークン制限超過のリスクがあり誤答。
  - 「オーバーラップをすればよい」という選択肢は試験では多くの場合**誤答**。冗長性が増すだけで論理的分割より劣る。
  - Vector Search 用のチャンク保存では「1チャンク1行にフラット化＋一意ID＋Delta テーブル」が正解パターン。
  - Delta Lake への保存順序は必ず「スキーマ定義 → チャンキング → 書き込み」。スキーマを後から定義するのは誤り。
  - embedding モデルの context length（例: 512）と**チャンクサイズを一致させる**ことがコスト・latency 最小化に有効。

- **誤答パターン：**
  - 「コンテキスト断片化を減らすためにチャンクサイズを大きくする」と思いがちだが、実際は大きくするとトークン制限を超えて不完全応答が悪化する。
  - 「30%オーバーラップで必ずコンテキストが保持される」と思いがちだが、実際は過度のオーバーラップは冗長性を生むだけで retrieval 関連性を必ずしも向上させない。
  - 「均等サイズ分割は公平で一貫している」と思いがちだが、実際は意味のあるセクションを分断して関連性が低下する。
  - 「チャンクをスキップしてドキュメントを直接保存する」と思いがちだが、実際は非構造化入力は下流処理が非効率になる。

### テキスト抽出ツール（Text Extraction Tools）

- **定義・概要：**
  様々なファイル形式（PDF・HTML・画像・スキャン文書）からテキストを抽出するための Python ライブラリ群。入力形式によって適切なツールが異なり、間違ったツールを選ぶと抽出が失敗する。

- **サブ概念：**
  - **unstructured**: テキストと画像が混在するPDFからテキストを抽出するために特別設計されたライブラリ。最小限のコード行数で済む。**テキスト・画像混在PDFの正解ツール**。
  - **pdfplumber**: PDFから**構造化テキストと表**を抽出するツール。表データの抽出に優れる。スキャン画像には不可。
  - **PyPDF2**: テキストベースのPDFのみ対応。スキャン画像・表・HTML には非対応。
  - **pytesseract / PyTesseract**: OCRを使用してスキャン画像（JPG・PNG等）からテキストを抽出するツール。**画像・スキャン文書の正解ツール**。
  - **Textract**: OCRをサポートし、スキャンされたPDFからの数値データ抽出にも有効。
  - **BeautifulSoup**: HTML・XMLのパースとテキスト抽出に特化。PDFや画像には非対応。
  - **LXML**: 大規模な HTML ファイルの高速かつ効率的なパースに最適。BeautifulSoup と組み合わせて使用することが多い。
  - **OCR（Optical Character Recognition）**: 画像・スキャン文書からテキストを認識・抽出する技術。pytesseract・Textract が主なツール。
  - **Image Preprocessing（画像前処理）**: OCR 前に透かし・ロゴ等の視覚的ノイズを除去してテキスト抽出品質を向上させる手法。

- **試験頻出ポイント：**
  - **ファイル形式ごとの正解ツール対応表**が頻出：
    - スキャン画像（JPG・PNG）→ **pytesseract / Textract**（OCRベース）
    - テキスト・画像混在PDF → **unstructured**
    - PDF の構造化テキスト・表 → **pdfplumber**
    - HTML（ネストされたテーブル含む）→ **BeautifulSoup + LXML**
    - テキストベースPDF → **PyPDF2**（ただし画像には不可）
  - スキャンされたレシート・請求書からの数値データ抽出 → **pytesseract + Textract**が正解。
  - スキャン済み PNG 画像からのテキスト抽出 → **pytesseract**が正解。
  - 透かし・ロゴを含むスキャン文書 → 前処理として**Image Preprocessing**でノイズ除去してから OCR。
  - Delta Lakeへの保存パイプライン: **スキーマ定義 → OCR（pytesseract等）でテキスト抽出 → Spark で Delta Lake に書き込む** が正しい順序。

- **誤答パターン：**
  - 「pytesseract は PDF ファイルに使える」と思いがちだが、実際は pytesseract はスキャン**画像**専用（PDFの中身がスキャン画像の場合に使う）。
  - 「pdfplumber はスキャン画像に使える」と思いがちだが、実際は pdfplumber は通常 PDF（テキスト埋め込み）専用。
  - 「BeautifulSoup は PDF に使える」と思いがちだが、実際は HTML/XML 専用。
  - 「PyPDF2 はスキャン PDF に使える」と思いがちだが、実際はテキストベース PDF のみ対応。

### データクリーニング・フィルタリング（Data Cleaning & Filtering）

- **定義・概要：**
  RAGシステムや LLM のトレーニング前に、不要・無関係・有害なコンテンツをデータセットから除去するプロセス。クリーニングの質が retrieval 精度と応答品質に直結する。

- **サブ概念：**
  - **無関係セクションの削除**: 付録・目次・冗長な脚注・マーケティング説明・免責事項など。RAGタスクに関連するコンテンツのみを保持する。
  - **ノイズ除去**: 広告・メタデータ（ハイパーリンク・埋め込みメディア・著者名・タイムスタンプ等）の削除。
  - **誤情報フラグ付きドキュメントの除外**: 前処理段階でフラグの立てられたドキュメントをパイプラインから排除する。免責事項追加やモデル無視に頼るのは誤答。
  - **不適切言語フィルタリング**: 取り込み前にフィルタリングツールで検出・除去する。取り込み後のモニタリングに頼るのは遅すぎる。
  - **タイムスタンプベースフィルタリング**: 古いドキュメントをタイムスタンプで除外して最新コンテンツを優先させる。
  - **冗長例の除去**: 技術マニュアルに含まれる冗長な例を削除してユニークで意味のあるコンテンツのみを保持する。マニュアル全体を除外するのは誤り。
  - **メールの定型文削除**: 顧客クレーム抽出など特定目的のRAGでは、挨拶・署名などの定型文を前処理で削除する。
  - **選択的メタデータ保持（法的ドキュメント）**: 全メタデータを一律削除するのではなく、ケース名・引用などの有用なメタデータは保持しながら不要なもの（タイムスタンプ等）のみ削除する。

- **試験頻出ポイント：**
  - 「完全性のためにすべてを残す」という選択肢は**常に誤答**。不要コンテンツはノイズとなり retrieval を低下させる。
  - 誤情報への対処は**前処理（preprocessing）段階での除外**が正解。取り込み後のモニタリングや免責事項追加では不十分。
  - 不適切言語への対処は**取り込み前のフィルタリング**が正解。LLM に無視させることや取り込み後のモニタリングは誤答。
  - タイムスタンプベースフィルタリングは「過去の古いデータが頻繁に取得される」問題への解決策。
  - メタデータの削除は「有用なものは残す、不要なものだけ削除する」の選択的アプローチが正解。「全削除」も「全保持」も誤り。
  - 冗長な例の除去では「プレースホルダーで置き換える」や「マニュアル全体を除外する」は誤答。「ユニークで意味のあるコンテンツを保持しながら冗長例を削除する」が正解。

- **誤答パターン：**
  - 「フラグが立てられたドキュメントでも免責事項を出力に追加すれば使用できる」と思いがちだが、実際は前処理段階での除外が唯一有効な対策。
  - 「モニタリングがあれば有害コンテンツを取り込んでもよい」と思いがちだが、実際はモニタリングは予防にならない。
  - 「全メタデータを削除すれば確実にクリーンになる」と思いがちだが、実際はケース名・引用など有用なメタデータを失うリスクがある。

### ソースドキュメント品質・選定（Source Document Quality & Selection）

- **定義・概要：**
  RAGシステムに投入するドキュメントを選定する際の基準。ドキュメントの信頼性・権威性・ドメイン適合性が応答の品質と正確性を左右する。

- **サブ概念：**
  - **権威ある一次ソース**: 規制当局への提出書類・公式金融レポート・査読済み医学研究論文・政府の健康ガイドライン・政府レポート等。信頼性・正確性が最も高い。
  - **二次ソース・非推奨ソース**: ニュース記事の要約・ユーザー執筆ブログ・フォーラムの書き込み・ソーシャルメディア・患者ブログ。主観的・未検証情報を含む可能性がある。
  - **関連性優先の原則**: ドキュメントの長さ・多様性よりも**コンテンツの関連性と品質**を優先する。
  - **医療ドキュメント**: 査読済み医学研究論文と政府の健康ガイドラインが最優先。教科書の要約・SNS・患者ブログは不可。
  - **金融ドキュメント**: 規制当局への提出書類・公式金融レポートが最優先。ニュース記事・ブログ・フォーラムは不可。
  - **技術サポートドキュメント**: 技術サポート要件に基づくコンテンツの関連性と品質を優先。ドキュメントの長さや公開可用性は二次的。

- **試験頻出ポイント：**
  - ドメイン固有アプリケーション（医療・金融・法律等）では**権威ある一次ソース**を使うことが最重要。
  - 「ドキュメントのサイズを最大化するために長いドキュメントを優先」は誤答。長さは品質を保証しない。
  - 「コンテンツ関連性よりソースの多様性を重視」は誤答。関連性が多様性に勝る。
  - 「関連性に関わらず公開されているすべてのテキストを含める」は誤答。無関係テキストが精度を希薄化する。

- **誤答パターン：**
  - 「公開されているテキストは信頼できる」と思いがちだが、実際はフォーラムやブログには未検証・偏った情報が多い。
  - 「データセットを大きくすれば精度が上がる」と思いがちだが、実際は低品質・無関係データが混入すると精度が下がる。
  - 「ニュース記事は詳しい説明があるので適切」と思いがちだが、実際は二次ソースであり詳細が欠けることがある。

### メタデータ管理（Metadata Management）

- **定義・概要：**
  データセット内の補助情報（タイムスタンプ・著者名・ケース名・引用等）を適切に扱う技術。不要なメタデータはモデルのパフォーマンスを低下させるノイズとなり、有用なメタデータはコンテキストの理解に寄与する。

- **サブ概念：**
  - **無関係メタデータ（削除対象）**: タイムスタンプ・著者名・透かし・ロゴ・ハイパーリンク・埋め込みメディア等。
  - **有用なメタデータ（保持対象）**: 法的ドキュメントのケース名・引用等、ドメイン固有の重要情報。
  - **モデル登録メタデータ（MLflow/Unity Catalog）**: モデル名・バージョン・スキーマ・タグ。これらを設定することがガバナンスと発見可能性に不可欠。
  - **Vector Searchインデックス用の一意ID**: 各チャンクに一意の識別子を付与することで、インデックスでの効率的な取得が可能になる。

- **試験頻出ポイント：**
  - ファインチューニング用データセットでは不要なメタデータを前処理で**全て削除**することでモデルが実際のコンテンツに集中できる。
  - 法的ドキュメントでは「全メタデータ削除」ではなく「有用なものを保持しながら不要なものだけ削除」が正解。
  - MLflow への モデル登録時にはメタデータ（スキーマ・バージョン・タグ）の設定をスキップしてはならない。
  - 「全数値データを除外する（タイムスタンプを含む）」は誤答。一部の数値データ（日付等）は関連性がある場合がある。

- **誤答パターン：**
  - 「コンテキストのためにメタデータを保持する（ファインチューニング用）」と思いがちだが、実際は無関係なメタデータを保持するとモデルが混乱してトピック識別のパフォーマンスが低下する。
  - 「タイムスタンプを含むすべての数値データを除外する」と思いがちだが、実際は一部の数値データ（日付等）は関連性がある場合がある。

### Retrieval評価メトリクス（Retrieval Evaluation Metrics）

- **定義・概要：**
  RAGシステムの検索コンポーネントのパフォーマンスを評価するための指標群。取得されたドキュメントの関連性（precision）・網羅性（recall）・ランキング品質（NDCG）・意味的類似度（cosine similarity）を測定する。

- **サブ概念：**
  - **Precision（精度）**: 取得されたドキュメントの中で実際に関連性があるものの割合。「無関係な結果が頻出する」問題の改善指標。
  - **Recall（再現率）**: 関連するドキュメント全体のうち取得できたものの割合。「関連ドキュメントを見落とす」問題の改善指標。
  - **F1 Score**: Precision と Recall の調和平均。両者のバランスを取った総合評価指標。
  - **Cosine Similarity（コサイン類似度）**: ベクトル間の意味的アライメントを評価する指標。取得されたコンテンツの関連性特定に役立つ。
  - **NDCG（Normalized Discounted Cumulative Gain）**: 取得されたドキュメントのランキング品質を評価する指標。法的ドキュメントRAGの評価などで使用。
  - **BLEU**: 主に**翻訳タスク**の評価指標。Retrieval評価には不適切。試験では誤答扱い。
  - **Perplexity（パープレキシティ）**: 言語モデルの流暢さ・確信度を測定する指標。Retrieval評価・要約評価には不適切。
  - **ROUGE**: 生成された要約と参照要約のコンテンツ重複を測定する指標。**要約タスクの標準指標**。Retrieval評価ではなく生成評価に使用。
  - **Sentiment Analysis**: テキストのトーンを評価するもので、retrieval performanceとは無関係。

- **試験頻出ポイント：**
  - Retrieval 評価の最重要指標は **Precision と Recall**（および F1・Cosine Similarity）。
  - 「無関係な結果が頻繁に表示される」→ **Precision** の改善が必要。
  - 「関連ドキュメントを見落とす」→ **Recall** の改善が必要。
  - BLEU は retrieval 評価には**不適切**（翻訳専用）。試験では必ず誤答。
  - Perplexity は retrieval 評価には**不適切**（言語流暢さの測定）。
  - ROUGE は**要約品質の評価**に使用。Retrieval評価には使わない。
  - Cosine Similarity は最終生成回答の包括評価には不十分（retrieval 品質の一部は測定できる）。
  - RAGシステムの評価は**検索コンポーネントと生成コンポーネントを分離**してテストするデータセットを作成し、MLflow の組み込み評価指標を使用するのが最善。

- **誤答パターン：**
  - 「コサイン類似度だけで最終生成回答の品質を包括的に評価できる」と思いがちだが、実際はコサイン類似度は検索品質の一部しか評価できず生成された回答を完全には評価しない。
  - 「BLEU で retrieval 評価ができる」と思いがちだが、実際は BLEU は翻訳タスク専用で retrieval には不適切。
  - 「Perplexity で retrieval 評価ができる」と思いがちだが、実際は Perplexity は言語の流暢さを評価するだけで Retrieval の関連性・精度とは無関係。
  - 「Sentiment Analysis は retrieval performance に関連する」と思いがちだが、実際はトーンの評価であり retrieval とは無関係。

### ベクトルストア・Delta Lake へのデータ保存（Data Storage for Vector Search）

- **定義・概要：**
  チャンク化されたドキュメントを Vector Search インデックスや Delta Lake に格納するための最適な形式・順序。効率的なストレージが retrieval のパフォーマンスに直結する。

- **サブ概念：**
  - **1チャンク1行形式（Flattened DataFrame）**: テキストチャンクの配列を持つデータフレームを1チャンク1行にフラット化し、各行に一意の識別子を作成して Delta テーブルに保存する形式。Vector Search インデックスでの最適な取得が可能。
  - **スキーマ定義の優先**: データを Delta Lake に書き込む前に必ずスキーマを定義する。スキーマを後から定義するのは誤り。
  - **Delta Lake**: 効率的なバッチクエリと信頼性の高いデータ管理を提供する。JSON ファイルより大規模検索のパフォーマンスが高い。
  - **Unity Catalog Volume**: ファイル（WAV・テキスト文字起こし等）を保存する場所。Delta テーブルとは異なる用途。
  - **Vector Search インデックス（FAISS等）**: embedding をインデックス化して高速な類似検索を可能にする仕組み。FAISS は効率的なインデックス化とクエリ機能を提供。

- **試験頻出ポイント：**
  - フィードバックデータをDelta Lakeに書き込む順序: **スキーマ定義 → テキストチャンキング → Delta Lakeテーブルへの書き込み**。
  - 断片化ドキュメントの保存順序: **テキストチャンク → Delta Lake スキーマを定義 → チャンクをバッチ処理 → Delta Lake テーブルに書き込む**。
  - Vector Search 用の保存: 「1チャンク1行にフラット化 + 一意ID + Delta テーブル」が最適。
  - JSON ファイルとしての個別保存は Delta テーブルに比べてパフォーマンスが劣る。

- **誤答パターン：**
  - 「スキーマを後で定義してもよい（先にデータを書き込む）」と思いがちだが、実際はスキーマを書き込み前に定義しないと一貫性が欠如してデータが使いにくくなる。
  - 「各チャンクを個別JSONファイルとして保存する」と思いがちだが、実際は大規模検索ではDelta テーブルの方が高パフォーマンス。
  - 「全データを1つの大きなチャンクとして保存する」と思いがちだが、実際はクエリパフォーマンスとアクセス性が大幅に低下する。

### バイアス対処（Bias Handling）

- **定義・概要：**
  採用推薦や医療応用など公平性が重要なシステムにおいて、トレーニングデータやRAGデータセットに潜在するバイアスを特定・除去・軽減するプロセス。

- **サブ概念：**
  - **バイアス分析と削除・再重み付け**: データセットのバイアスを特定し、バイアスのあるエントリを削除するか再重み付けして公平性を確保する。
  - **事前学習済みモデルのバイアス**: 事前学習済みモデルも元のデータセットのバイアスを反映する可能性がある。モデルを変更しないだけではバイアスは解決しない。
  - **データセットサイズ削減**: バイアス除去の方法としては不十分。サイズを削減してもバイアスの除去は保証されない。

- **試験頻出ポイント：**
  - バイアス対処の正解は「潜在的なバイアスを分析し、バイアスのあるエントリを削除または再重み付けする」。
  - 「事前学習済みモデルのみ使用してデータセット変更を避ける」は誤答（モデルも元データのバイアスを反映）。
  - 「バイアスを無視する（ユーザークエリで出力が決まるため）」は誤答（バイアスを持続させるリスクがある）。
  - 「データセットサイズを削減する」は誤答（バイアス除去にならない）。

- **誤答パターン：**
  - 「モデルがユーザークエリに基づいて出力するため、データのバイアスは問題ない」と思いがちだが、実際はバイアスのあるデータはシステムの不公平さを持続させるリスクがある。

### 言語フィルタリング（Language Filtering）

- **定義・概要：**
  特定の言語のみを対象とするシステムで、対象外言語のドキュメントを前処理段階でフィルタリングする手法。言語の一貫性を確保することで retrieval の精度を保つ。

- **サブ概念：**
  - **言語検出ツール**: 非英語ドキュメントを自動検出してフィルタリングするツール。
  - **翻訳 vs フィルタリング**: 翻訳は追加のオーバーヘッドと潜在的な不正確さをもたらすためフィルタリングが優先される。

- **試験頻出ポイント：**
  - 英語専用RAGシステムの多言語ドキュメント前処理では、「翻訳」より「言語検出ツールで非英語ドキュメントをフィルタリング」が正解。
  - 「言語を識別せずにすべてのドキュメントを処理する」は誤答（無関係・不正確な出力のリスク）。
  - 「潜在的な多言語機能のためにすべてのドキュメントを保持する」は誤答（英語のみシステムでは品質が下がる）。

- **誤答パターン：**
  - 「すべての非英語コンテンツを英語に翻訳する」と思いがちだが、実際は翻訳は追加コストと不正確さのリスクがあり、フィルタリングの方が効率的。

### プロンプト-応答ペアの品質評価（Prompt-Response Pair Quality）

- **定義・概要：**
  LLMアプリケーション（特にチャットボット）のトレーニング・評価データとして使用するプロンプトと応答のペアの品質を評価するための基準。

- **サブ概念：**
  - **応答の完全性（Completeness）**: クエリのすべての側面に対処する応答かどうか。チャットボットの有用性とユーザー満足度を向上させる。
  - **プロンプトの関連性（Relevance）**: 顧客のクエリに対してプロンプトが関連性を持つかどうか。意味のある応答生成を確保する。
  - **プロンプトの冗長性（Redundancy）**: クエリの焦点を薄める可能性があるため、**評価基準としては重要でない**（誤答扱い）。
  - **トークン使用効率**: コスト削減には重要だが、整合性確保には関係しない（誤答扱い）。
  - **応答のランダムサンプリング**: 品質を保証しない（誤答扱い）。

- **試験頻出ポイント：**
  - プロンプト-応答ペアの評価で最重要な2軸: **「プロンプトの関連性」と「応答の完全性」**。
  - 「冗長性」「トークン効率」「ランダムサンプリング」は評価基準として重要でない（誤答）。

- **誤答パターン：**
  - 「プロンプトに多くのコンテキストを含める（冗長性）」と思いがちだが、実際は冗長性はクエリの焦点を薄める可能性がある。
  - 「トークン使用量を最適化する」と思いがちだが、実際はコスト削減には関連するが整合性の確保とは関係がない。

### トークン制限とContext Length（Token Limit & Context Length）

- **定義・概要：**
  LLM がプロセスできる最大トークン数（Context Length）を超えると、入力が切り捨てられ（Truncation）重要な情報が失われる。ドキュメントサマリーパイプラインの設計においてContext Lengthの評価が不可欠。

- **サブ概念：**
  - **Truncation**: Context Length を超えた部分が切り捨てられ、重要情報が失われる現象。
  - **Context Length の評価理由**: モデルが入力ドキュメントを truncation なく処理できるよう確保するため（要約の長さ制御やコスト削減のためではない）。
  - **チャンクサイズとContext Lengthの一致**: embedding モデルのContext Length（例: 512）とチャンクサイズを一致させることで、コスト・latency を最小化できる。
  - **Context Length 超過への対処**: チャンクサイズを小さくする・取得レコード数を削減する。新しいモデルのトレーニングや要約モデルの適用は過剰または不適切。
  - **ALiBi**: Context Length の拡張に関する技術。Context Length 問題を解決するためのモデル再トレーニングは複雑すぎて誤答扱い。

- **試験頻出ポイント：**
  - Context Length の評価の主目的は「truncation なしでドキュメント全体を処理できること」。コスト削減・要約短縮・トレーニング効率とは無関係。
  - Context Length 超過（エラー）への対処: **チャンクサイズを小さくする** と **取得レコード数を削減する** が正解。
  - 「要約モデルで長さを短縮する」は Context Length 超過問題の解決策としては誤答（重要な詳細が省略されるリスク）。
  - 512 トークンのチャンクには context length 512 の最小 embedding モデルを使用するのがコスト・latency 最小の最適設定。

- **誤答パターン：**
  - 「Context Length はコスト削減のために評価する」と思いがちだが、実際は主な役割は truncation なしの入力処理確保（コストへの影響は副次的）。
  - 「ALiBi でモデルを再トレーニングする」と思いがちだが、実際はトークン制限問題の解決には不要で複雑すぎる。
  - 「より長い Context Length で新しい embedding モデルをトレーニングする」と思いがちだが、実際はリソースを大量に消費し常に実現可能ではない。

### 冗長データの除去（Redundancy Removal）

- **定義・概要：**
  データセット内の冗長・重複したコンテンツを識別・除去して、モデルトレーニングの効率と応答品質を向上させるプロセス。

- **サブ概念：**
  - **冗長な例の削除**: 技術マニュアルに含まれる繰り返しの例を削除しつつ、ユニークで意味のあるコンテンツを保持する。
  - **プレースホルダー置換の問題**: 冗長コンテンツをプレースホルダーで置き換えてもモデルを混乱させる可能性がある（誤答）。
  - **マニュアル全体の除外の問題**: 冗長性を含むマニュアル全体を除外すると価値ある情報も失う（誤答）。

- **試験頻出ポイント：**
  - 「ユニークで意味のあるコンテンツを保持しながら冗長な例を削除する」が正解。
  - 「完全性のためにすべての例を保持する」は誤答。
  - 「冗長な例をプレースホルダーで置き換える」は誤答。

- **誤答パターン：**
  - 「完全性のためにすべての冗長な例を保持する」と思いがちだが、実際は冗長な例は価値を追加せずデータセットを膨らませるだけ。

## 補足：問題から読み取れる重要な法則まとめ

### ドキュメント種別ごとの最適処理ツール一覧

| ファイル形式 | 最適なツール | 理由 |
|---|---|---|
| スキャン画像（JPG/PNG） | pytesseract / Textract | OCR 対応 |
| テキスト・画像混在 PDF | unstructured | 最小コードで両方に対応 |
| 構造化テキスト・表を含む PDF | pdfplumber | 表抽出に最適化 |
| テキストベース PDF のみ | PyPDF2 | スキャン不可 |
| HTML（ネストテーブル含む） | BeautifulSoup + LXML | HTML 解析専用 |

### チャンキング戦略の優先順位

1. **論理的チャンキング（見出し・段落・チャプター）** ← 最優先・最頻出正解
2. チャンクサイズを小さくする（トークン制限超過問題への対処）
3. オーバーラップ ← 通常は誤答（冗長性のみ増加）
4. 固定サイズチャンキング ← 誤答（意味的セクションを分断）
5. ドキュメント全体を1チャンク ← 誤答（トークン制限超過）

### データ保存パイプラインの正しい順序

1. スキーマ定義
2. テキストチャンキング（または OCR 抽出）
3. バッチ処理
4. Delta Lake テーブルへの書き込み

### Retrieval 評価メトリクスの用途分類

| メトリクス | 用途 | 備考 |
|---|---|---|
| Precision / Recall | Retrieval 評価 | **最重要** |
| F1 Score | Retrieval の総合評価 | Precision+Recall の調和平均 |
| Cosine Similarity | Semantic アライメント評価 | Retrieval 評価の補助 |
| NDCG | ランキング品質評価 | 法的ドキュメント等 |
| ROUGE | **要約品質**の評価 | Retrieval 評価には不適切 |
| BLEU | **翻訳**タスクの評価 | Retrieval・要約には不適切 |
| Perplexity | 言語の**流暢さ**評価 | Retrieval 評価には不適切 |
