### RAGシステムの評価（検索と生成の分離評価）

- 定義・概要：
  RAGシステムは「検索（Retrieval）」と「生成（Generation）」の2つのコンポーネントで構成されており、評価も両者を分離して行うことが推奨される。検索コンポーネントにはPrecision/Recall/NDCGなどの指標を用い、生成コンポーネントにはBLEU/ROUGE/Perplexityを使う。MLflowの組み込み評価指標を活用することで体系的な評価が可能。

- サブ概念：
  - 検索評価データセット：既知の質問と正解ドキュメントペアを作成し、検索品質を定量評価する
  - 生成評価データセット：既知の質問と参照回答ペアを作成し、生成品質を定量評価する
  - 分離評価の重要性：検索の問題なのか生成の問題なのかを切り分けることで、改善の焦点が明確になる
  - MLflow組み込み評価指標：Databricksでの評価に標準的に使えるフレームワーク
  - 取得されたコンテキストの品質評価：RAGの回答関連性問題の診断の最初のステップ

- 試験頻出ポイント：
  - RAGの正式評価では「検索と生成を別々にテストするデータセットを作成し、MLflow組み込み指標を使う」が正解
  - RAGが無関係な情報を返す問題の診断では「取得されたコンテキストの品質評価」を最初に行う（LLMを変えたり検索アルゴリズムを変える前に）
  - MLflowでのRAGアプリ評価に最も関連する指標は「検索関連性指標（precision/recall）＋応答精度指標（BLEU/ROUGE）」の組み合わせ

- 誤答パターン：
  - 「コサイン類似度スコアで最終回答の品質を包括的に評価できる」と思いがちだが、コサイン類似度は検索品質を測るもので生成された回答の評価には不十分。
  - 「LLM-as-a-judgeだけで評価すれば十分」と思いがちだが、LLM-as-a-judgeは主観的で作成済みデータセットより信頼性が低い。
  - 「複数のLLMベンチマークで最適LLMを選べばよい」と思いがちだが、LLMのベンチマークは検索固有のパフォーマンスに対処しない。

### テキスト生成品質指標：ROUGE・BLEU・Perplexity

- 定義・概要：
  テキストの品質を定量的に評価するための標準指標群。ROUGE（Recall-Oriented Understudy for Gisting Evaluation）は要約品質評価に特化し、BLEU（Bilingual Evaluation Understudy）は翻訳品質評価に使われる。Perplexityは言語モデルがテキストをどれだけ自然に生成できるか（流暢さ・自然さ）を測定する。

- サブ概念：
  - ROUGE：生成サマリーと参照サマリーのRecall・Precision・F1スコアのオーバーラップを測定。要約タスク向け。
  - BLEU：生成テキストと参照テキストのn-gramオーバーラップを比較。機械翻訳品質評価の標準指標。
  - Perplexity：言語モデルの流暢さ・自然さを測定。低いほど自然な応答。
  - これらはいずれも「生成品質（Generation Quality）」の指標であり、「検索品質（Retrieval Quality）」の指標ではない

- 試験頻出ポイント：
  - 要約品質の評価 → ROUGEが正解（BLEUは翻訳向け）
  - 翻訳品質の評価 → BLEUが正解（ROUGEは要約向け）
  - 流暢さ・自然さの評価 → Perplexityが正解
  - 財務ニュース要約LLMの評価 → Perplexity（流暢さ）＋ROUGE（関連性）＋BLEU（語列精度）＋Latency（速度）の4指標が必要
  - 法的ドキュメント要約pipelineの評価 → ROUGEとRecallを優先（コンテンツの関連性・カバレッジを測定）
  - これらの指標は「検索パフォーマンス」「本番運用監視」「コスト評価」には使えない

- 誤答パターン：
  - 「ROUGEで翻訳品質を評価できる」と思いがちだが、ROUGEは主に要約タスク向けで翻訳にはBLEUが標準。
  - 「BLEUで要約品質を評価できる」と思いがちだが、BLEUは翻訳タスク向けで要約にはROUGEが標準。
  - 「BLEUとPerplexityで検索パフォーマンスを評価できる」と思いがちだが、これらは言語生成品質の指標であり検索には不適切。
  - 「本番監視にBLEUスコアやperplexityを使う」と思いがちだが、これらは言語品質・トレーニング評価指標であり運用監視には向かない。

### 検索品質指標：Precision・Recall・NDCG・Retrieval Latency

- 定義・概要：
  RAGシステムの検索ステップの品質を評価するための指標群。Precision（適合率）は取得されたドキュメントのうち関連するものの割合、Recall（再現率）は全関連ドキュメントのうち取得できたものの割合を測定する。NDCG（Normalized Discounted Cumulative Gain）はランキング品質を測定する。Retrieval Latencyは検索ステップのパフォーマンスのボトルネックを示す。

- サブ概念：
  - Precision（適合率）：取得されたドキュメント中の関連ドキュメントの割合。高いほどノイズが少ない。
  - Recall（再現率）：全関連ドキュメントのうち取得できた割合。高いほど見落としが少ない。
  - NDCG（Normalized Discounted Cumulative Gain）：ランキングの精度を評価する指標。上位に関連ドキュメントが来るほど高スコア。
  - Retrieval Latency：検索ステップの応答時間。ボトルネック特定に使う。
  - Precision高・Recall低：関連ドキュメントを取り逃している（見落とし多い）
  - Recall高・Precision低：無関係なドキュメントも大量に取得している（ノイズ多い）→ フィルタリングやembeddingの改善が必要

- 試験頻出ポイント：
  - RAG検索システムのrelevanceとボトルネック診断 → Precision・Recall・Retrieval Latencyが正解
  - RAGのretrieval effectiveness評価（MLflow） → Precision・Recall・NDCGを追跡
  - 法的ドキュメントRAGシステムの評価 → NDCGとRecallの組み合わせ（ランキング品質＋取得率）
  - チャンキング戦略の最適化 → RecallやNDCGなどの評価指標を使用してテスト
  - Recall高・Precision低が示すこと：無関係ドキュメントの混入 → より良いembeddingまたはフィルタリングで対処
  - これらは「生成品質」「コスト効率」「スケーラビリティ」の評価には使わない

- 誤答パターン：
  - 「RAG検索の評価にBLEUやROUGEを使う」と思いがちだが、これらは生成品質の指標であり検索評価には不適切。
  - 「Recall高・Precision低はバッチサイズを増やせば解決する」と思いがちだが、これは無関係ドキュメントの混入を示しており、embeddingやフィルタリングの改善が必要。
  - 「Precision低・Recall低はlatencyに問題がある」と思いがちだが、latencyはスピードに影響するが精度バランスには影響しない。
  - 「retrieval precision/recallでLLMの要約パフォーマンスを評価できる」と思いがちだが、これらはドキュメント検索品質の指標でLLMの要約性能には対応しない。

### 本番モニタリング指標：Latency・Throughput・エラーレート

- 定義・概要：
  デプロイされたLLMアプリケーションの運用パフォーマンスを監視するための3つの主要指標。Latency（応答時間）・Throughput（単位時間あたりの処理量）・Error Rate（エラー発生率）はシステムの応答性・処理能力・信頼性を示す。これらは言語品質やタスク固有の評価指標とは明確に区別される。

- サブ概念：
  - Latency（レイテンシ）：応答時間。リアルタイム性が求められるシステムで重要。
  - Throughput（スループット）：単位時間あたりに処理されたクエリ数。顧客問い合わせ処理数など。高負荷シナリオでの評価に必要。
  - Error Rate（エラーレート）：エラーの発生頻度。システムの安定性・信頼性を示す。
  - スケーラビリティ評価：様々なワークロード下でのthroughputとlatencyを測定する

- 試験頻出ポイント：
  - 本番デプロイLLMのパフォーマンス監視 → Latency・Throughput・エラーレートが正解
  - カスタマーサービス問い合わせアプリの本番パフォーマンス評価 → 単位時間あたりに処理された顧客問い合わせ件数（Throughput）が正解
  - 本番LLMアプリケーションのスケーラビリティ評価 → 様々なワークロード下でのthroughputとlatencyが正解
  - 多言語カスタマーサポートLLMの本番監視 → latency・応答精度・言語カバレッジ + token使用量・コスト
  - これら運用指標はBLEU/perplexityなどの品質評価指標やPrecision/F1スコアとは異なる

- 誤答パターン：
  - 「精度やF1スコアで本番デプロイ監視をする」と思いがちだが、これらはタスク固有の評価指標であり運用監視指標ではない。
  - 「BLEUとperplexityで本番監視する」と思いがちだが、これらは言語品質・トレーニング指標であり運用監視には不向き。
  - 「HuggingFace Leaderboardのベンチマーク値で本番パフォーマンスを評価できる」と思いがちだが、これらは汎用ベンチマークでデプロイされたアプリの性能は示さない。
  - 「エネルギー使用量でアプリケーションパフォーマンスを評価できる」と思いがちだが、コスト最適化には有用でもパフォーマンス評価には使えない。

### コスト最適化と監視

- 定義・概要：
  LLMアプリケーションの推論コストを管理・最適化するための戦略群。Token使用量の監視、モデルサイズ選択（クエリ複雑度に応じた適切なモデル）、クエリのバッチ処理、推論ロギングの活用などが主要な手法。Databricksでのコスト管理には複数の手法を組み合わせることが推奨される。

- サブ概念：
  - Token使用量監視：クエリあたりのトークン消費量を追跡し、コストの高い操作を特定する
  - モデルサイズ選択：シンプルなクエリには小型LLMを使用し、重要タスクには大型モデルを確保する
  - クエリバッチ処理：複数のクエリを1リクエストにまとめてAPI呼び出し回数を削減
  - 推論ロギング有効化：クエリコストの監視・最適化戦略の根拠データを収集
  - GPU使用率監視：ハードウェアリソースの効率的な活用を確認し不要なコストを削減
  - 動的GPUスケーリング：需要に基づいてGPUリソースを動的にスケールし、低需要時の無駄なコストを回避
  - トークン課金（従量制）：リクエスト量が少ない場合にプロビジョニングスループットよりコスト効率が高い

- 試験頻出ポイント：
  - 財務ドキュメント処理RAGのコスト最小化 → 小型LLM使用＋クエリバッチ処理＋推論ロギングの3戦略
  - DatabricksデプロイRAGのLLMコスト管理 → クエリ複雑度に応じたモデルサイズ選択＋token使用量最適化
  - 高トラフィックQAシステムのコスト最適化 → 動的GPUスケーリング＋token使用量監視＋inference logging
  - リクエスト量が低い場合のコスト効率化 → プロビジョニングスループットからトークン課金（従量制）に切り替え
  - コストとlatencyを最小化したembeddingモデル設定 → チャンクサイズと一致するcontext length（512）＋最小モデルサイズ（0.13GB）＋embedding次元384
  - context lengthを増やすとtoken使用量が増加しコストが上昇する（コスト削減にはならない）

- 誤答パターン：
  - 「外部モデル（External Models）に切り替えればスループットコストの問題が解決できる」と思いがちだが、External ModelsはリクエストスループットのコストEfficiencyに対処しない。
  - 「context lengthを増やすとtoken使用量を削減できる」と思いがちだが、context lengthを増やすと逆にtoken使用量が増加しコストが上昇する。
  - 「より大きなembeddingモデルを使えばchunkサイズ超過エラーを解決できる」と思いがちだが、エラーの原因はtoken数がcontext lengthを超えることであり、chunkサイズを小さくするか取得レコード数を減らすのが正解。
  - 「頻繁な出力のキャッシュはLLM推論コストを大幅に削減できる」と思いがちだが、キャッシュはlatencyを削減するがLLM推論コストには大きな影響を与えない。

### MLflowによる評価・追跡

- 定義・概要：
  MLflowはDatabricks上でモデルの評価・追跡・管理を行うためのオープンソースフレームワーク。RAGシステムの評価では、retrieval指標（precision/recall）の記録から、生成品質指標（BLEU/ROUGE）の追跡まで、システム全体の評価を一元管理できる。

- サブ概念：
  - MLflow組み込み評価指標：RAGシステム評価のための標準指標セット（検索・生成それぞれの指標を含む）
  - Retrieval指標の追跡：Precision/Recall等の検索品質指標をMLflowで記録・可視化
  - 評価実験管理：異なる設定・パラメータでの実験結果を比較・管理

- 試験頻出ポイント：
  - RAGシステムの正式評価 → 検索と生成を分離したデータセット＋MLflow組み込み評価指標が最適
  - MLflowでのRAG評価に最も関連する指標 → 検索関連性指標（precision/recall）＋応答精度指標（BLEU/ROUGE）
  - DatabricksのRAGアプリでretrieval性能を記録・改善するツール → MLflow

- 誤答パターン：
  - 「MLflowでトレーニング損失や検証精度を追跡してRAGを評価する」と思いがちだが、これらはモデルトレーニングに関連する指標であってデプロイ後のRAG評価には適さない。
  - 「throughputやクエリあたりのコストがMLflow評価の主要指標」と思いがちだが、これらは運用効率の指標であってアプリケーション品質の評価指標ではない。

### Inference Logging と Inference Tables

- 定義・概要：
  Inference Loggingはデプロイされたモデルエンドポイントへの推論リクエスト・レスポンスの詳細を記録する仕組み。DatabricksではInference Tables（推論テーブル）というネイティブ機能として提供され、Model Servingエンドポイントのリクエスト・レスポンスのログ記録と監視を自動的に行う。クエリパターン・応答時間・エラー発生などの運用データを提供する。

- サブ概念：
  - Inference Tables：Databricksネイティブの推論ログ記録機能。マイクロサービスによるカスタムログ実装の代替。
  - Inference Logging が提供するインサイト：クエリパターン・応答時間・エラー発生（tokenやBLEUではない）
  - コスト監視への活用：高トラフィック下でのコストとパフォーマンスの傾向分析に使用
  - パフォーマンス最適化：クエリパターンの分析によるアクション可能なデータ提供

- 試験頻出ポイント：
  - Databricksでリクエスト・レスポンスをネイティブにログ記録する機能 → Inference Tables（LakeviewやDBSQL、Vector Searchではない）
  - Inference Loggingが提供するインサイト → クエリパターン・応答時間・エラー発生（tokenやBLEUではない）
  - DatabricksのRAGアプリのシステムパフォーマンスとコスト監視 → MLflow（retrieval指標）＋Inference Logging（token/latency）＋GPU使用率監視の3点セット
  - 財務ドキュメント処理のコスト最小化 → 推論ロギングを有効にしてクエリコストを監視する

- 誤答パターン：
  - 「LakeviewはInference Tablesの代替になる」と思いがちだが、LakeviewはDelta Lakeの監視用でInference Logsを処理しない。
  - 「DBSQLでエンドポイントのログを記録できる」と思いがちだが、DBSQL（Databricks SQL）はデータクエリ用であり推論ログ記録には不適切。
  - 「Inference LoggingはBLEUスコアやperplexityを提供する」と思いがちだが、これらの言語品質指標はInference Loggingでは提供されない。

### LLM-as-a-Judge による評価

- 定義・概要：
  LLM自身を評価者（judge）として使い、他のLLMの出力品質を評価する手法。チャンキング戦略の最適化など、定量的指標が難しい評価に活用できる。ただし主観的であり、専用のテストデータセットを使った評価より信頼性が低い場合がある。

- サブ概念：
  - チャンキング評価への応用：特定のチャンクが以前の質問にどれだけ答えられているかをLLM-as-a-judgeで評価し、チャンキングパラメータを最適化する
  - 主観性の問題：LLM-as-a-judgeは主観的であり、作成済みデータセットを使った評価より信頼性が低い

- 試験頻出ポイント：
  - チャンキング戦略の最適化アプローチ → RecallやNDCGなどの評価指標でのテスト＋LLM-as-a-judgeメトリクスの組み合わせが有効
  - LLM-as-a-judgeの位置づけ：補助的手段であり、専用テストデータセット＋MLflow指標が主軸

- 誤答パターン：
  - 「RAGシステムの評価はLLM-as-a-judgeだけで十分」と思いがちだが、主観的で信頼性が低く、作成されたデータセットとMLflow指標を組み合わせた評価が推奨される。

### Context Length管理とToken Limit問題

- 定義・概要：
  LLMにはトークン数の上限（Context Length）が存在し、入力がこれを超えるとエラーが発生する。RAGシステムではベクトルDBから取得するドキュメントのチャンクサイズや取得レコード数がContext Lengthに大きく影響する。モデルを変更せずにToken Limit問題を解決する方法として、チャンクサイズの削減と取得レコード数の削減が有効。

- サブ概念：
  - Context Length：LLMが一度に処理できるトークンの最大数
  - BAD_REQUEST / InvalidArgument エラー：token数がcontext lengthを超えた場合に発生
  - チャンクサイズ縮小：各ドキュメントから取得されるトークン数を減らし、context limit内に収める
  - 取得レコード数の削減：vector databaseから取得するレコード数を制限して合計token数を削減
  - ALiBi：Attention with Linear Biases。長いコンテキストに対応するためのモデル再トレーニング手法（token limit問題の解決には不要で複雑すぎる）

- 試験頻出ポイント：
  - Context Length超過エラーへの対処（モデル変更なし） → チャンクサイズを小さくする＋取得レコード数を削減（2つ選択）
  - embeddingモデルのサイズ変更はContext Lengthエラーの解決策にならない
  - output tokensの削減はinput tokenの超過問題に対処しない
  - ALiBiでのモデル再トレーニングはtoken limit問題解決に不要で複雑すぎる
  - コスト・latency優先のembeddingモデル設定 → チャンクサイズに合わせたcontext length（512）＋最小モデルサイズ＋最小embedding次元

- 誤答パターン：
  - 「小さなembeddingモデルに変えればContext Lengthエラーを解決できる」と思いがちだが、エラーはinput tokenの超過が原因であり、embeddingモデルサイズは関係ない。
  - 「output tokensの上限を削減すれば解決できる」と思いがちだが、これはinput token数の問題であり、output token数の制限では解決しない。

### チャンキング戦略の評価・最適化

- 定義・概要：
  RAGシステムのドキュメントチャンキング戦略（チャンクの分割方法・サイズ）を評価・最適化するための体系的アプローチ。直感的なパラメータ選択から、RecallやNDCGなどの評価指標とLLM-as-a-judgeを組み合わせた科学的アプローチへの移行が重要。

- サブ概念：
  - 評価指標による戦略テスト：RecallやNDCGを用いて複数のチャンキング戦略（段落分割・チャプター分割など）を比較
  - LLM-as-a-judge：各チャンクが過去の質問にどれだけ答えられているかをLLMで評価し、チャンキングパラメータを最適化
  - embeddingモデル変更との区別：embeddingモデルの変更はセマンティックアライメント改善だがチャンキング戦略の最適化ではない
  - 分類器によるretrievalフィルタリング：チャンキング戦略自体の最適化にはならない

- 試験頻出ポイント：
  - チャンキング戦略の体系的最適化 → RecallやNDCGでの評価指標テスト＋LLM-as-a-judgeの2アプローチが正解
  - 「分類器追加でretrievalをフィルタリング」は誤り：チャンキング戦略の最適化にはならない
  - 「LLMに統計量でチャンクサイズを決めさせる」は誤り：構造化されたアプローチが欠ける

- 誤答パターン：
  - 「embeddingモデルを変更してチャンキング戦略を最適化する」と思いがちだが、embeddingモデルの変更はセマンティックアライメントの改善であり、チャンキング戦略の最適化とは別問題。
  - 「LLMに質問と回答を渡して最適トークン数を特定させ、要約統計でチャンクサイズを決める」と思いがちだが、この方法は構造化されたアプローチが欠けており信頼性が低い。

### スケーラビリティ評価

- 定義・概要：
  本番LLMアプリケーションが増加するクエリ量に対応できるか（スケーラビリティ）を評価するための指標と戦略。主にThroughputとLatencyを様々なワークロード条件で測定することで評価する。モデルサイズや言語品質指標はスケーラビリティの直接指標にはならない。

- サブ概念：
  - Throughput（スループット）：様々なワークロード下での処理能力。高負荷シナリオでの測定が特に重要。
  - Latency（レイテンシ）：応答時間。スケール時に許容範囲内に収まるかを確認。
  - 動的GPUスケーリング：需要に応じてGPUリソースを動的に増減させる

- 試験頻出ポイント：
  - 本番LLMのスケーラビリティ評価 → 様々なワークロード下でのthroughputとlatencyが最重要
  - LLMのコスト効率比較 → クエリあたりのコスト・latency・高負荷シナリオでのthroughputの3指標
  - モデルサイズ単独ではスケーラビリティを評価できない
  - BLEUやperplexityはスケーラビリティ評価には使えない

- 誤答パターン：
  - 「モデルサイズとパラメータ数でスケーラビリティを評価できる」と思いがちだが、モデルサイズ単独ではスケーラビリティは評価できない。スループットとlatencyを実測することが必要。
  - 「BLEUとperplexityでコスト効率を評価できる」と思いがちだが、これらは品質評価指標でありコスト評価には不適切。

### 応答の多様性（エンゲージメント指標）

- 定義・概要：
  チャットボットのユーザーエンゲージメントとリテンションを高めるための指標。応答の多様性（Response Diversity）は、同じような応答の繰り返しを避けてユーザーの興味を維持するために重要。ランダム性（Randomness）とは異なり、文脈に合った多様な応答を生成する能力を指す。

- サブ概念：
  - 応答の多様性（Response Diversity）：ユーザーの興味を維持し、エンゲージメントとリテンションを向上させる
  - ランダム性との違い：ランダム性はユーザーを混乱させるリスクがあるが、多様性は文脈に合った異なる応答を提供する
  - 応答の繰り返しの問題：同じ応答の繰り返しはユーザーを飽き飽きさせリテンションを低下させる

- 試験頻出ポイント：
  - ゲーミングchatbotのエンゲージメント・リテンション向上 → 応答の多様性が正解（ランダム性や応答の繰り返しは不正解）
  - ランダム性はユーザーを混乱させエンゲージメントを低下させる可能性がある

- 誤答パターン：
  - 「多様性とランダム性は同じ意味」と思いがちだが、ランダム性はユーザーを混乱させるリスクがあり、多様性は文脈に合った異なる応答を指す。

### プロビジョニングスループット vs トークン課金（従量制）

- 定義・概要：
  Databricksでのモデルデプロイにおける2つの課金モデル。プロビジョニングスループット（Provisioned Throughput）は一定のスループットを確保するための固定コストモデル。トークン課金（従量制）はトークン使用量に応じてコストがスケールするペイパーユーズモデル。リクエスト量が少ない場合はトークン課金の方がコスト効率が高い。

- サブ概念：
  - プロビジョニングスループット：一定のリクエスト処理能力を保証するが、使用量が少ない場合はコスト非効率
  - トークン課金（従量制）：使用量に応じたコスト。低リクエスト量では効率的、高トラフィックでは高コストになりうる
  - コスト効率の判断基準：リクエスト量がプロビジョニングスループットに対して不十分な場合、従量制が最適

- 試験頻出ポイント：
  - リクエスト量がプロビジョニングスループットに対して不十分な場合のコスト効率化 → トークン課金（従量制）に切り替えが正解
  - 手動スロットリングはコスト効率を最適化しない
  - パラメータ数を減らしたモデルへの変更はスループット関連のコスト問題に直接対処しない

- 誤答パターン：
  - 「External Modelsに切り替えれば低リクエスト量のコスト問題が解決する」と思いがちだが、External ModelsはリクエストスループットのコストEfficiencyに対処しない。
  - 「受信リクエストを手動スロットルすればコスト効率が改善する」と思いがちだが、スロットリングはリクエスト管理には役立つがコスト効率の最適化にはならない。

### モデル選定基準（LLM選択時の評価軸）

- 定義・概要：
  ユースケースに適したLLMを選択するための評価基準。カスタマーサポートチャットボット向けにはレイテンシと推論コストのバランス、長文ドキュメント要約RAGにはContext Length・要約品質・Inference Speed、FAQシステムにはモデルサイズ・Latency指標・応答Perplexityが重要な選択軸となる。

- サブ概念：
  - カスタマーサポートチャットボット向け：クエリあたりのレイテンシ＋推論コストのバランス
  - 長文ドキュメント要約RAG向け：Context Length（長文処理能力）＋要約品質＋Inference Speed
  - FAQシステム向け：モデルサイズ・アーキテクチャ（コスト管理）＋Latency指標（応答性）＋応答Perplexity（流暢さ）
  - BLEUスコア：モデル選定においてtoken limitとの組み合わせでは要約品質や推論速度に対応しないため不適切
  - Inference Speed（推論速度）：タイムリーな回答提供に直結し、RAG選定で重要

- 試験頻出ポイント：
  - カスタマーサポートLLM選定指針 → クエリあたりのレイテンシと推論コスト
  - 長文要約RAGのLLM選定 → Context Length・要約品質・Inference Speedの3要素
  - FAQシステムのLLM選定（精度・コスト・スケーラビリティのバランス） → モデルサイズ・Latency指標・応答Perplexityの3要素
  - LLMのコスト効率比較 → クエリあたりのコスト・Latency・高負荷でのThroughput

- 誤答パターン：
  - 「retrieval precision/recallでLLM要約性能を評価できる」と思いがちだが、retrieval指標はドキュメント検索品質を評価するものであり、LLMの要約性能とは別。
  - 「token limitとBLEUスコアで要約向けLLMを選定できる」と思いがちだが、BLEUは要約品質に対応しておらず、推論速度にも対応しない。
  - 「モデルサイズとトレーニングデータの多様性がRAG LLM選定の主要基準」と思いがちだが、これらは特定アプリケーション要件（context length・速度・コスト）と直接対応しない。

### 翻訳ユースケースにおけるLLM安全性評価

- 定義・概要：
  翻訳ユースケースでLLMの出力を定性的に評価する際の安全性指標。応答の精度（タスク要件を満たしているか）と関連性（コンテキストに沿っているか）が、翻訳における安全性と有効性の主要指標となる。レイテンシやテキスト長などのパフォーマンス指標は「安全性」の指標ではない。

- サブ概念：
  - 応答の精度と関連性：安全性の重要な指標。出力がエラーや有害なコンテンツなしにタスク要件を満たすことを確認。
  - レイテンシ・テキスト長：パフォーマンス指標であり安全性指標ではない
  - コード生成能力：翻訳タスクの安全性評価とは無関係
  - 言語的類似性：言語的類似性の維持ではなく翻訳の関連性と適切さが安全性の本質

- 試験頻出ポイント：
  - 翻訳ユースケースでのLLM出力安全性評価 → 応答の精度と関連性が正解（レイテンシ・テキスト長ではない）

- 誤答パターン：
  - 「応答のレイテンシと生成されたテキストの長さで安全性を評価する」と思いがちだが、これらはパフォーマンス指標であり安全性の指標ではない。
