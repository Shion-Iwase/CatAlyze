プロンプトエンジニアリング

LLMから望ましい出力を引き出すために入力テキスト（プロンプト）を設計・最適化する技術。明示的な指示・出力フォーマット指定・few-shot examples・コンテキストの追加が主な手法。汎用的な指示では一貫性が下がるため、具体的な指示と例の組み合わせが重要。

  • 明示的指示：「簡潔で事実に基づいた回答を提供してください」など具体的な動作を指定する
  • 出力フォーマット指定：箇条書き・フィールド名・JSON形式など出力形式をプロンプトで明示する
  • few-shot examples：期待する入出力ペアの具体例をプロンプト内に含める（例：{"date": "April 16, 2024", "order_id": "ORD123"}）
  • コンテキスト拡張：ユーザー入力に関連する追加情報をプロンプトに組み込む
  • metaprompt（システムプロンプト）：モデルの振る舞い全体を定義するシステムレベルの指示
試験頻出ポイント
  • 一貫した出力が必要 → 明示的指示 ＋ few-shot examplesの組み合わせ
  • JSON等の特定フォーマット出力 → フォーマット仕様 ＋ 出力例をプロンプトに含める
  • 「一般的なプロンプト（汎用指示）」は不正解 → 不完全・不一貫な出力につながる
  • temperatureはトーン・精度を制御しない（ランダム性のみに影響）

RAGアーキテクチャとパイプライン設計

RAG（Retrieval-Augmented Generation）は、LLMの生成とベクトルストアからの文書検索を組み合わせたアーキテクチャ。モデルの知識限界を補い、最新情報・専門知識に基づいた回答生成を可能にする。ハルシネーション（幻覚）防止に最も有効な手法。

  • 構成要素：Embeddingモデル → ベクトルストア → Retriever → LLM（生成）
  • セマンティック検索：ベクトルDBによる意味ベースの検索（キーワード一致ではない）
  • 引用の追加：取得コンテンツに関連する引用をパイプラインに含めることで透明性向上
  • フォールバックメカニズム：未解決のクエリを人間エージェントにエスカレーション

正しいデプロイ順序：

  1. ドキュメントの前処理・Embeddingモデル準備
  2. Embeddingの生成
  3. ベクトルストアへの保存・インデックス作成
  4. Retrieverの構築
  5. エンドポイントのデプロイ
試験頻出ポイント
  • パイプライン順序は必ず「Embedding生成 → ベクトルストア → Retriever → デプロイ」の順
  • 「先にエンドポイントをデプロイして後でEmbeddingを設定」は不正解
  • 法的・医療などニュアンスが必要なクエリ → キーワード検索ではなくセマンティック検索（ベクトルDB）
  • 回答に引用が不足している場合 → temperatureやファインチューニングではなく、引用ステップをパイプラインに追加
  • ルールベースシステムは動的クエリに対応できず不正解

Embeddingモデル選択と最適化

Embeddingモデルはテキストを高次元ベクトルに変換し、意味的類似度に基づいた検索・推薦を可能にする。ユースケースに応じて選択すべき特性（Context Length・次元数・ドメイン特化度）が異なる。

  • Sentence Transformers：文全体のセマンティクスをキャプチャ。チャットボットの意図識別に最適
  • ドメイン固有モデル：医療・法的・財務データで学習されたモデルは汎用モデルより高精度
  • Word2Vec・GloVe：単語レベルのEmbeddingで文レベルのセマンティクスをキャプチャできない（文理解タスクには不適）
  • ANN（近似最近傍）検索：高速ルックアップと高精度のバランス（FAQシステムに有効）
  • Context Length：長い研究論文には長いContext LengthをサポートするEmbeddingモデルが必要
ユースケース優先すべき基準
チャットボット意図識別Sentence Transformers（文レベルセマンティクス）
長い研究論文の検索Context Length ＋ セマンティック類似度精度
FAQの高速・高精度検索セマンティック類似度ベンチマーク ＋ ANN検索
コスト・Latency優先（512tokenチャンク）Context Length 512 ＋ モデルサイズ 0.13GB ＋ 次元384
試験頻出ポイント
  • Word2Vec・GloVeは文レベルのセマンティクスタスクに不適（単語レベルのみ）
  • BERT-base分類用モデルはセマンティック検索Embedding生成には最適化されていない
  • 長い文書にはContext LengthとセマンティCk精度が推論コストより優先

マルチターン対話・ステートトラッキング

複数のやり取りを通じてコンテキストを維持しながら会話を続けるシステム。各ターンで過去の会話内容を記憶・参照するステートトラッキング（メモリ）の実装が品質の鍵。

  • ステートトラッキング：以前の対話ターンのコンテキストを保持するメカニズム
  • メモリ：ターンをまたいでユーザーコンテキストを保存する仕組み
  • コンテキストリセットの問題：各ターン後にリセットすると断片的・不一貫な会話になる
試験頻出ポイント
  • チャットボットが以前のターンを忘れる問題 → ステートトラッキング（メモリ）の実装が解決策
  • 「tokenの上限を増やす」は不正解 → 入力サイズに影響するがコンテキスト保持とは別の問題
  • 「temperatureを下げる」は不正解 → ランダム性に影響するがコンテキスト記憶とは無関係

LangChainとチェーン設計

LLMを使ったマルチステップのワークフロー（チェーン）を構築するフレームワーク。入力処理・検索・生成を統合した複雑なパイプラインを、複数コンポーネントの連結で実現する。

  • LLMChain：PromptTemplate ＋ LLMを連結した基本的なチェーン
  • Retriever：ベクトルストアからドキュメントを検索するコンポーネント
  • PromptTemplate：ユーザー入力を構造化するためのテンプレート
  • 標準構成：PromptTemplate ＋ Retriever ＋ LLMの3点セット
  • 相互運用性：マルチステップ推論ではステージ間でデータをシームレスに共有することが重要
試験頻出ポイント
  • チェーンに含めるべき標準コンポーネント：PromptTemplate ＋ Retriever ＋ LLM
  • 「ハードコードされたテンプレート」は不正解 → 多様なクエリへの柔軟性がない
  • 「Retrieverなしのembedding」は不正解 → embeddingだけでは動的な検索ができない
  • 「要約モデルを追加」は不正解 → 重要な詳細を削除し応答品質を低下させる

MLflow PyFuncモデル

Pythonで書いた任意のモデルをMLflowのシリアライズ形式でラップし、前処理・後処理を含むパイプライン全体をカプセル化するインターフェース。デプロイ時に入出力の互換性と一貫性を確保する。

  • 前処理（Preprocessing）：モデル入力前にデータをクリーニング・標準化・変換
  • 後処理（Postprocessing）：モデル出力を人間が読めるフォーマットに変換（例：スコア → 「承認」「拒否」）
  • predict関数：PyFuncモデルのコアとなるメソッド。前処理・推論・後処理を内包
  • Unity Catalogへの登録：モデルガバナンス・バージョン管理・アクセス制御のため

ワークフロー：入力前処理 → MLflow PyFunc予測 → 出力後処理（ユーザーフレンドリーなフォーマット）

試験頻出ポイント
  • PyFuncが有用な理由：推論の前後にデータを変換でき互換性と最適化された出力を確保（APIコスト最小化・再トレーニング削減ではない）
  • 「後処理なしに前処理のみ」は不正解 → 後処理はモデル出力を解釈可能にするために不可欠
  • 「生の入力を直接モデルに渡す」は不正解 → 前処理でデータ品質を確保し予測信頼性を向上させる

ガードレール・セーフティフィルター

LLMアプリケーションの出力が安全・正確であり、アプリケーション要件に沿っていることを確保するための制御機構。ハルシネーション・有害応答・プライバシー侵害などのリスクを軽減する。

  • metapromptによるガードレール：機密情報を出力しないよう指示するルールをシステムプロンプトで定義
  • 入力サニタイズ：特殊文字の削除とデータ形式検証によるインジェクション攻撃防止
  • コンテンツフィルタリング：安全でない・有害な応答を除外する事前定義フィルター
  • Retrievalベース拡張：検索した情報で応答を根拠づけ不正確な生成を防止
試験頻出ポイント
  • ガードレールの目的：latency削減・多様性向上ではなく、安全・正確・アプリ要件整合の確保
  • インジェクション攻撃防止 → 入力サニタイズ（特殊文字削除 ＋ データ形式検証）
  • 「temperatureを下げれば安全になる」は不正解 → temperatureは安全性に影響しない
  • 「監視・ログだけで十分」は不正解 → 悪意ある入力の実行を防がない

コンテキスト長とチャンキング戦略

LLMやEmbeddingモデルにはトークン制限（Context Length）があるため、長いドキュメントは適切なサイズに分割（チャンキング）して処理する。チャンキング戦略の選択は検索精度・要約品質に直接影響する。

  • セクションベースチャンキング：ドキュメントの論理的なセクション（「イントロ」「調査結果」「条項」）に基づいて分割。文書構造を保持し品質が高い
  • 均等サイズチャンキング：一定トークン数で機械的に分割。論理的なセクションを壊すリスクがある
  • オーバーラップチャンキング：チャンク間に重複を持たせてコンテキストを補完するが、論理的セクション境界との整合が取れない場合がある
  • モデルとの一致：チャンクサイズ（例：512token）とEmbeddingモデルのContext Length（512）を合わせる
ドキュメント種別推奨戦略
10,000語超の政府レポート・学術論文セクション（イントロ・調査結果等）に基づくチャンキング
法的契約書論理的セクション（条項・付属文書）に基づくチャンキング
512token制限のEmbeddingモデル利用時512token以内のチャンクに分割して各チャンクを独立処理
試験頻出ポイント
  • 「ドキュメント全体を1チャンク」は不正解 → トークン上限を超える
  • 「均等サイズに分割」は一般に不正解 → 意味のあるセクションを壊すリスク
  • 「より長いContext Lengthのモデルを新規トレーニング」は不正解 → コスト大・実現性が低い
  • 「要約モデルで長さを短縮」は不正解 → 重要な詳細が欠損する

エージェントシステムとツール設計

複数のツール・データソース・操作を組み合わせて複雑なタスクを自律的に処理するLLMベースのシステム。注文状況・配送・返金など複数操作が必要なカスタマーサポートに適している。

  • ツール定義：利用可能な機能（create_record, update_record, delete_record など）を明示的に定義し使用例も提供
  • プロンプトテンプレートによる機能公開：エージェントの機能と使用例を指定したテンプレートで正しいツール呼び出しを誘導
  • マルチターン対応：複数回のやり取りをまたいでコンテキストを維持
  • フォールバックメカニズム：処理できないクエリは人間にエスカレーション
試験頻出ポイント
  • エージェント機能の最善の公開方法 → 機能と使用例を明示したプロンプトテンプレート
  • 「embeddingを適用」は不正解 → 検索は支援するが機能公開は処理しない
  • 「ルールベースシステム」は不正解 → 動的なマルチエージェントタスクへの柔軟性が不足

LLMモデル選択基準

ユースケースに応じて適切なLLMを選択することが重要。コンテキストウィンドウサイズ・ドメイン特化度・多言語対応・コスト・latencyが主要な選択基準。

モデル / 特性適したユースケース
MPT-30B（大コンテキストウィンドウ）長いドキュメントの処理・大量のコンテキストが必要なタスク
Llama2-70B（高品質・機密処理可）機密データを扱うタスク・高品質な出力が必要なケース
CodeLlama-34Bコード生成・コードのデバッグ
小型ファインチューニング済み ＋ キャッシュコスト効率と低latencyのバランス（商品推薦など）
ユースケース優先基準
リアルタイム言語翻訳低latency ＋ 高翻訳精度
eコマース多言語チャットボット多言語パフォーマンスベンチマーク
財務アドバイス生成ドメイン精度 ＋ 財務データセット精度（流暢さより優先）
学術論文要約学術テキストでファインチューニングされた要約LLM
試験頻出ポイント
  • 「モデルサイズが大きいほど良い」は不正解 → latency増加・コスト効率低下
  • 財務アドバイス → 流暢さ（低perplexity）やlatencyよりドメイン精度とコンプライアンスが優先
  • コスト効率 → 最大モデルではなく小型ファインチューニング済みモデル ＋ クエリキャッシュ
  • 「パラメータ数が多いモデル」は不正解 → パラメータ数だけではドメイン関連性・精度が保証されない

評価指標（ROUGE・BLEU・Precision/Recall/NDCG）

LLMアプリケーションの品質評価には用途に応じた指標の選択が重要。「要約品質」「翻訳品質」「検索品質」「運用パフォーマンス」の4カテゴリに分類できる。

指標用途特徴
ROUGEテキスト要約評価生成された要約と参照要約のコンテンツ重複を測定
BLEU機械翻訳評価出力テキストと参照テキストの類似度を測定（要約には不適）
Precision検索精度取得ドキュメントのうち関連するものの割合
Recall検索網羅性全関連ドキュメントのうち取得できたものの割合
NDCG検索ランキング品質関連性とランキング順序を評価
Perplexity言語モデルの流暢さ確信度を測定（要約品質は測定しない）
Latency/Throughput/エラーレート本番運用監視応答性・処理能力・信頼性の監視指標
試験頻出ポイント
  • 要約品質の評価 → ROUGE（BLEUは翻訳向けで要約には不適）
  • 法的ドキュメント要約の評価 → ROUGE ＋ Recall（カバレッジも重要）
  • RAGシステムの評価 → Precision/Recall ＋ NDCG
  • 本番LLM監視 → Latency・Throughput・エラーレート（BLEU・F1は運用監視指標ではない）
  • 「Perplexityで要約品質を評価」は不正解 → Perplexityは流暢さを評価するが情報量は測定しない

コスト・Latency最適化

本番環境でのLLMアプリケーション運用において、コスト効率と低latencyを両立させることが重要。モデルサイズ・インフラ・キャッシュ・スケーリング戦略が主な最適化手段。

  • 小型ファインチューニング済みモデル：品質を維持しながらコストとlatencyを削減
  • クエリキャッシュ：頻繁なクエリをキャッシュし繰り返しクエリのパフォーマンス向上
  • 水平スケーリング ＋ Autoscaling：変動するワークロードに対応し過少/過剰リソースを防ぐ
  • GPU ＋ 分散クラウドインフラ：高throughputのリアルタイム要約・EmbeddingにはGPUアクセラレーション ＋ 分散クラウドインフラ
  • Inference Tables（Databricks）：本番エンドポイントのリクエスト・レスポンスをネイティブにログ記録
試験頻出ポイント
  • コスト・Latency優先（512tokenチャンク）→ Context Length 512 ＋ モデルサイズ 0.13GB ＋ 次元384 の最小構成
  • スケーラビリティ最適化 → Autoscalingポリシーによる水平スケーリング（固定リソースやレート制限だけでは不十分）
  • 「ローカルストレージ」は不正解 → 本番システムのスケーラビリティとlatencyを損なう
  • 「シングルノードサーバー」は不正解 → 高需要アプリには不十分

ファインチューニングの適用場面

事前学習済みモデルを特定ドメインのデータでさらに学習させる手法。汎用モデルではカバーしきれないドメイン固有の知識・スタイル・トーンをモデルに習得させるために使用する。

  • ドメイン固有の用語・知識が必要（法的・医療・財務テキスト）
  • 特定のトーン・スタイルを維持したい（丁寧なカスタマーサービス）
  • 学術論文の要約など特定タスクへの最適化
  • 企業のブランディングや通信ガイドラインへの適合
問題適切な解決策ファインチューニングでは不十分な理由
Retrievalの精度が低いドメイン固有Embeddingモデル・ベクトルDB改善生成を改善するがretrieval自体には対処しない
回答に引用が不足パイプラインに引用ステップを追加引用問題を直接解決しない
スケーラビリティの確保Autoscalingによる水平スケーリングthroughput向上には直接対処しない
試験頻出ポイント
  • 丁寧なトーン ＋ ドメイン知識 → 丁寧なトーンのデータセット ＋ ドメインデータでのファインチューニング
  • 学術要約 → 学術テキストでファインチューニングされた要約LLM（汎用モデルはドメイン固有インサイトが不足）
  • 「retrieval精度低下にファインチューニング」は不正解 → retrieval問題はEmbeddingモデルやベクトルDBで対処
  • 「temperatureを上げて多様な要約」は不正解 → ランダム性が増加し要約精度が低下

Q3. チャットボットが以前のターンの内容を忘れてしまう問題の原因と解決策は何ですか？

▶ 答えを見る

A. 原因はステートトラッキング（メモリ）が実装されておらず、各ターン後にコンテキストがリセットされること。解決策はステートトラッキングの実装（過去の対話ターンのコンテキストを保持する仕組みの追加）。tokenの上限増加やtemperatureの変更では解決しない。

Q4. テキスト要約の品質評価に適した指標と、翻訳品質に適した指標をそれぞれ答えてください。

▶ 答えを見る

A. 要約品質 → ROUGE（生成要約と参照要約のコンテンツ重複を測定）。翻訳品質 → BLEU（出力テキストと参照テキストの類似度を測定）。BLEUは要約評価に使うと不正解になるので注意。

Q5. MLflow PyFuncモデルの前処理・後処理の役割をそれぞれ説明してください。

▶ 答えを見る

A. 前処理：モデル入力前にデータをクリーニング・標準化・変換し、予測信頼性を向上させる。後処理：モデルの生の出力（スコアなど）を人間が読めるフォーマット（「承認」「拒否」など）に変換する。両方揃って初めてパイプラインが完結する。

Q6. 10,000語を超える法的契約書をRAGシステムのEmbedding処理に使う場合、どのチャンキング戦略が最適ですか？また、均等サイズ分割が不適な理由は？

▶ 答えを見る

A. 最適策：論理的セクション（条項・付属文書）に基づくセクションベースチャンキング。均等サイズ分割が不適な理由：意味のある論理的セクション境界を無視して機械的に分割するため、一つの条項が複数チャンクに分断され検索・要約品質が低下する。

Q7. ファインチューニングが適切でない場面を2つ挙げ、それぞれの適切な解決策を説明してください。

▶ 答えを見る

A. ① Retrievalの精度が低い場合 → ドメイン固有EmbeddingモデルやベクトルDBの改善が適切（ファインチューニングは生成を改善するがretrieval自体には対処しない）。② スケーラビリティの確保 → Autoscalingによる水平スケーリングが適切（ファインチューニングはthroughput向上に対処しない）。

Q8. LangChainでRAGチェーンを構築する際の標準コンポーネント3点を挙げてください。

▶ 答えを見る

A. ① PromptTemplate（ユーザー入力を構造化）② Retriever（ベクトルストアからドキュメントを検索）③ LLM（応答を生成）。この3点が標準構成。要約モデルの追加は重要な詳細を削除するため不正解。
